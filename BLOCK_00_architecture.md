# Блок 0. Обзор платформы и архитектуры BPMSoft

**Цель:** понять, что такое BPMSoft, из чего состоит система, какие технологии используются и как всё связано между собой.

**Статус:** ✅ завершён

---

## 1. Что такое BPMSoft

BPMSoft — это **low-code платформа** для создания бизнес-приложений. «Low-code» означает, что многие задачи решаются через визуальные конструкторы (без написания кода), но платформа полностью открыта для программирования, когда нужно что-то нестандартное.

Типичные приложения: CRM-системы, управление обращениями (кейсами), автоматизация бизнес-процессов, сервисные порталы.

---

## 2. Три уровня системы (логическая модель)

Это **взгляд разработчика** на систему — где что находится и что можно менять.

### 2.1. Конфигурация (можно менять)

Всё, что настраивается и кастомизируется. Здесь живут три вида логики:

| Вид логики | Технология | Что делает |
|---|---|---|
| Клиентская логика | JavaScript (ExtJS, AMD-модули) | Интерфейс, который видит пользователь в браузере |
| Серверная логика | C# (.NET 8) | Обработка данных, бизнес-правила на сервере |
| Бизнес-процессы | BPMN | Визуальные схемы автоматизации |

Конфигурация организована в **пакеты** — модули, которые устанавливаются поверх ядра. Каждый пакет может содержать:
- Клиентские модули (JavaScript)
- Серверный код (C#)
- SQL-сценарии
- Бизнес-процессы (BPMN)
- Внешние сборки и данные

**Принцип открытости/закрытости:** система закрыта для прямых модификаций ядра, но открыта для расширения через пакеты. Ты не меняешь существующий код — ты добавляешь свой пакет, который расширяет или замещает функциональность.

### 2.2. База данных (можно менять)

Уровень хранения данных:
- Пользовательские данные (контакты, заявки, сделки)
- Системная информация (настройки, права доступа)
- Метаданные

Некоторые задачи удобнее решать прямо на уровне БД — через представления (views) и хранимые процедуры. Эта логика потом используется в конфигурационных элементах.

BPMSoft имеет **СУБД-независимую архитектуру** — встроенная ORM-система позволяет работать с данными одинаково, независимо от того, какая СУБД используется (PostgreSQL, MS SQL и др.).

### 2.3. Ядро (Core) — нельзя менять

Фундамент системы, написанный на C# / .NET 8. Содержит базовые библиотеки и функциональность, необходимую для работы приложения.

Разработчик может:
- ✅ Использовать всю функциональность ядра (API, классы, сервисы)
- ❌ Изменять ядро

**Аналогия:** ядро — как операционная система. Ты пользуешься её API, но не лезешь в исходники.

## 3. Трёхуровневая архитектура (инфраструктурная модель)

Это **взгляд инфраструктурщика/DevOps** — из каких серверов и компонентов состоит система.

### 3.1. Уровень представления (Presentation)

То, что видит пользователь — веб-приложение в браузере. Весь клиентский код (JavaScript, ExtJS, HTML/CSS) выполняется здесь. Браузер общается с сервером через HTTP-запросы и WebSocket-соединения.

Опционально: мобильное приложение.

### 3.2. Уровень приложения (Application)

Сервер, на котором работает BPMSoft. Реализован на .NET 8 (или .NET Framework для старых версий).

| Платформа | ОС | Веб-сервер |
|---|---|---|
| .NET 8 | Linux / Windows | Kestrel / IIS |
| .NET Framework | Windows | IIS |

На .NET 8 приложение является **неделимым** — выполняет функции и загрузчика (аутентификация, проверка лицензий), и конфигурационной части (бизнес-логика).

Сервер приложений отвечает за:
- Аутентификацию и авторизацию пользователей
- Проверку лицензий
- Выполнение бизнес-логики
- Запуск бизнес-процессов

### 3.3. Уровень данных (Data)

Два компонента:

**Сервер базы данных:**
- Поддерживаемые СУБД: PostgreSQL, MS SQL Server, Postgres Pro, Tantor, Pangolin, Arenadata Prosperity
- Хранит пользовательские данные, конфигурацию и метаданные

**Сервер кэширования:**
- Поддерживаемые: Redis, Dragonfly, KeyDB
- Хранит сессии пользователей, кэшированные данные
- Обеспечивает обмен данными между узлами веб-фермы (при горизонтальном масштабировании)

## 4. Связь двух моделей

Логические уровни и инфраструктурные — два взгляда на одну систему:

| Логический уровень | Где физически работает |
|---|---|
| Конфигурация (клиентская часть — JS) | Уровень представления (браузер) |
| Конфигурация (серверная часть — C#) | Уровень приложения (сервер) |
| Ядро (Core) | Уровень приложения (сервер) |
| База данных | Уровень данных (сервер БД + кэш) |

---

## 5. Варианты развёртывания

- **On-site** — на локальных серверах организации
- **Cloud** — в облаке

Поддерживается **горизонтальное масштабирование** — несколько серверов приложений за балансировщиком нагрузки. Redis в этом случае критически важен для синхронизации сессий и кэша между узлами.

Контейнеризация и оркестрация: **Docker** и **Kubernetes** (особенно для микросервисов).

---

## 6. Микросервисы

Помимо основного приложения, BPMSoft включает вспомогательные микросервисы:

| Микросервис | Назначение |
|---|---|
| Глобальный поиск | Полнотекстовый поиск (Elasticsearch) |
| ML-сервис | Машинное обучение, предсказания |
| Синхронизация почты | Работа с email |
| Поиск дублей | Дедупликация записей |

Каждый развёртывается отдельно (Docker/Kubernetes) и масштабируется независимо.

## 7. Low-code и no-code инструменты

Перед тем как писать код, нужно понимать: многие задачи в BPMSoft решаются визуальными инструментами. Код пишется только тогда, когда они не покрывают требования.

### 7.1. Дизайнер процессов

Визуальный редактор BPMN-процессов на основе drag-and-drop. Позволяет строить цепочки: от простых (отправить письмо) до сложных (многоуровневая интеграция). Поддерживает многопользовательскую разработку.

### 7.2. Мастер разделов

Low-code инструмент для создания и редактирования разделов, деталей, страниц, вкладок и мини-карточек. Позволяет добавлять бизнес-логику на страницах без кода.

С версии 1.9 доступен **продвинутый дизайнер интерфейсов** на базе Angular — с более гибкой сеткой для формирования разметки страниц.

### 7.3. Дизайнер кейсов

Для настройки жизненного цикла объектов через стадии и шаги. Построен на движке процессов (ProcessEngineService), но с упрощённой моделью. Пользователь видит, на какой стадии находится объект и какие шаги нужно выполнить.

### 7.4. Инструменты интеграции

Подключение к внешним системам через REST, SOAP, OData, вебхуки — без кода. Алгоритм: генерация запроса → отправка → получение ответа → извлечение данных.

### 7.5. Инструменты машинного обучения

Встроенные ML-модели:
- Прогнозирование справочных и числовых полей
- Поиск похожих текстов
- Рекомендательное прогнозирование
- Предиктивный анализ данных и скоринг
- Прогнозирование оттока
- Сегментация

## 8. Компетенции разработчика BPMSoft

### 8.1. Минимальные навыки

- Опыт работы с **C#** от 6 месяцев
- Опыт работы с **JavaScript** от 6 месяцев
- Опыт работы с **SQL** (PL/pgSQL, T-SQL) от 6 месяцев

### 8.2. Рекомендуемые навыки

- Опыт от 1 года с C#, JavaScript, SQL
- Понимание технологий: WCF, OData, Angular 18, RxJS, Sencha Ext.JS, RequireJS

### 8.3. Технологический стек и его роль

| Технология | Где используется в BPMSoft |
|---|---|
| JavaScript | Клиентская логика: интерфейс, формы, взаимодействие |
| C# | Серверная логика: бизнес-правила, обработка данных, веб-сервисы |
| SQL | Прямая работа с БД: отчёты, сложные выборки, миграции |
| ExtJS | Система классов — фундамент клиентского кода |
| AMD / RequireJS | Модульная система — загрузка и связывание кода |
| Angular | Продвинутые интерфейсы (с версии 1.9) |
| OData / REST | Интеграции с внешними системами |
| BPMN | Описание и автоматизация бизнес-процессов |
| LESS | Препроцессор CSS — стилизация интерфейса |

### 8.4. Ключевые правила разработки

1. Вся пользовательская реализация — **в персональном пакете** (не в системных, не в `Custom`)
2. В персональном пакете должны быть добавлены все необходимые **зависимости**
3. Доступна встроенная IDE (C#, JS, TypeScript) и интеграция с внешними IDE (Visual Studio, VS Code)

## 9. Пакетная архитектура (подробно)

### 9.1. Что такое пакет

Пакет — это контейнер для группировки компонентов системы. Всё, что ты разрабатываешь, живёт в пакете.

Пакет может включать:
- Бизнес-объекты (схемы данных)
- Бизнес-процессы (BPMN)
- Схемы с исходным кодом (C#)
- Клиентские модули (JavaScript)
- SQL-сценарии
- Внешние сборки (.dll)
- Привязанные данные

### 9.2. Структура пакета на файловой системе

| Папка | Содержимое |
|---|---|
| `Schemas` | Схемы пакета (клиентские модули, объекты, исходный код) |
| `Assemblies` | Внешние сборки |
| `Data` | Привязанные данные |
| `SqlScripts` | SQL-сценарии |
| `Resources` | Локализованные ресурсы |
| `Files` | Файловый контент |

Файл `descriptor.json` — метаданные: имя, версия, издатель (`Maintainer`), массив зависимостей (`DependsOn`).

```json
{
  "Descriptor": {
    "UId": "a00051f4-cde3-4f3f-b08e-c5ad1a5c735a",
    "Name": "Custom",
    "BPMSoftVersion": "0.0.0",
    "Maintainer": "Customer",
    "Description": "Custom Package",
    "DependsOn": [
      {
        "UId": "003be62f-2ffc-4b4f-a2a7-60f101277377",
        "PackageVersion": "1.0.0",
        "Name": "ML"
      }
    ]
  }
}
```

### 9.3. Иерархия и зависимости

Пакеты образуют **дерево зависимостей**, аналогичное наследованию классов в ООП:
- Если пакет C зависит от A и D — в C доступна функциональность обоих
- Если пакет E зависит от B и D — в E доступна функциональность A, B и D (транзитивно)

Типичная цепочка:
```
[Системные пакеты]                                                    [Пользовательские]
Base → ... → NUI → ... → CaseService → ... → SLM → ... → Custom  →  YourPackage
```

Пакет `Custom` — последний в цепочке предустановленных (системных) пакетов. Твой пользовательский пакет создаётся после `Custom` и зависит от него или от другого пакета в конце системной иерархии. Пакет `Custom` не рекомендуется использовать напрямую для разработки — лучше создать собственный.

**Диаграмму зависимостей** можно посмотреть в разделе «Конфигурация» → Действия → «Диаграмма зависимостей пакетов».

### 9.4. Типы пакетов

| Тип | Описание | Когда использовать |
|---|---|---|
| Обычный пакет | Код компилируется вместе со всем приложением | Стандартная разработка |
| Пакет-сборка | Компилируется в отдельную .dll | Ускорение компиляции и поставки |
| Пакет-проект | C#-проект, компиляция за 1–2 сек | Быстрая итерация при разработке |

### 9.5. Конфигурационные элементы

Внутри пакета живут **конфигурационные элементы** — это всё, что можно создать, редактировать и замещать:
- Объекты (схемы данных)
- Клиентские модули (модели представления)
- Исходный код (C#)
- Бизнес-процессы
- SQL-сценарии
- Данные (привязанные записи)

Управление элементами — через раздел «Конфигурация» в дизайнере системы.

## 10. Система кэширования (обзор)

Мы уже знаем, что инфраструктура BPMSoft включает три сервера: сервер приложений, сервер базы данных и **сервер кэширования** (Redis, Dragonfly или KeyDB). Зачем нужен кэш?

**Проблема:** каждый раз, когда пользователь открывает страницу, системе нужны данные — настройки, права доступа, справочники. Ходить за ними в базу данных при каждом запросе — медленно и создаёт нагрузку на БД.

**Решение:** часто используемые данные хранятся в кэше (Redis). Redis работает в оперативной памяти, поэтому отдаёт данные в десятки раз быстрее, чем база данных. Если у тебя несколько серверов приложений (горизонтальное масштабирование), Redis также синхронизирует сессии пользователей между ними.

В BPMSoft кэш разделён на **уровни видимости** — данные могут быть доступны только в рамках одного запроса, в рамках сессии пользователя или для всего приложения целиком. Это позволяет изолировать информацию между пользователями и управлять временем её жизни.

Подробно — как работают уровни хранилищ, как обращаться к кэшу из C#-кода и какие есть правила оптимизации — мы разберём в **Блоке 12** (работа с данными на сервере), когда уже будем знать C#.

## Итоги Блока 0

**Что нужно запомнить:**

1. **BPMSoft = low-code платформа.** Сначала пробуй без кода, потом пиши код.

2. **Три логических уровня:** конфигурация (твоя зона работы) → база данных (хранение) → ядро (не трогай).

3. **Три инфраструктурных уровня:** браузер → сервер приложений (.NET 8) → сервер данных (БД + Redis).

4. **Пакет — единица модульности.** Всё, что ты делаешь — в своём пакете, с правильными зависимостями.

5. **Иерархия пакетов = наследование.** Твой пакет видит всё, от чего зависит.

6. **Технологический стек:** JS + ExtJS + AMD (клиент), C# + .NET 8 (сервер), SQL (данные), BPMN (процессы), OData/REST (интеграции).

7. **Кэш (Redis)** — ускоряет работу, хранит сессии, синхронизирует узлы. Подробности — в Блоке 12.
