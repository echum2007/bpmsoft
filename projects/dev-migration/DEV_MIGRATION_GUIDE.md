# Инструкция: Перенос кастомизаций на dev-инстанс

**Цель:** Перенести пакеты CTI и Custom с прода на dev и создать минимальный набор тестовых данных.

**Обновлено:** 2026-04-20  
**Dev-сервер:** 192.168.102.46 (SSH: `ssh -i ~/.ssh/id_bpmsoft_dev gore@192.168.102.46`)

---

## Что переносим

Все кастомизации хранятся в двух пакетах:

**CTI** — основной пакет:
- Замещения объектов: `Case`, `ServicePact`, `ServiceItem`, `ConfItem`
- Поля: `Case.UsrCcEmails`, `ServicePact.UsrCcEmails`
- Страницы: `CasePage`, `ServicePactPage`, `ServiceItemPage`, `ConfItemPage` и др.
- C#: `UsrActivityCcEventListener`, `UsrCcAddressResolver`, `UsrEmployeeNotificationManager`
- BPMN-процессы: `UsrSendEmailToSROwnerCustom1`, `UsrSendNotificationToCaseOwnerCustom1` и др.
- Новые сущности: `UsrLaborRecords`, `UsrConfItemCatalog`, `UsrContractType`, `UsrVendorList`

**Custom** — legacy-пакет с дублями (ServiceItem, ConfItem, ServicePact, UsrConfIteminService), который зависит от CTI и применяется поверх него.

---

## Шаг 1. Экспорт пакетов с прода

1. Войти в прод как администратор
2. Открыть **Дизайнер системы** (кнопка шестерёнки) → блок **«Конфигурирование разработчиком»** → **«Управление конфигурацией»**
3. В списке слева найти пакет **CTI** → нажать **«...»** → **«Экспортировать»**
4. Браузер скачает файл `*.zip` (внутри — `*.gz` с пакетом)
5. Повторить для пакета **Custom**

> Экспортировать можно сразу несколько пакетов: зажать Ctrl, выбрать CTI + Custom → **«Экспортировать выделенные»** — получится один ZIP со обоими пакетами.

Сохранить файлы в:
```
projects/dev-migration/
  CTI_export_YYYYMMDD.zip
  Custom_export_YYYYMMDD.zip
```

---

## Шаг 2. Импорт пакетов на dev

> ⚠️ **Перед импортом** — убедиться, что сделан `pg_dump` базы dev (резервная копия). Система делает её автоматически в процессе установки, но лучше иметь свою.

1. Войти в dev как администратор
2. Открыть **Дизайнер системы** → блок **«Приложения»** → **«Установка и удаление приложений»**
3. Нажать **«Добавить приложение»** → **«Установить из файла»**
4. Выбрать (или перетащить) файл `CTI_export_YYYYMMDD.zip`
5. Дождаться завершения — система автоматически создаст резервную копию конфигурации (несколько минут), затем установит пакет
6. Повторить для `Custom_export_YYYYMMDD.zip`

> **Порядок важен:** сначала CTI, потом Custom — Custom зависит от схем CTI.

После успешной установки система покажет сообщение об успехе. В случае ошибки — там же можно скачать лог и откатиться к резервной копии («Восстановить пакеты из резервной копии»).

## Шаг 3. Компиляция и перезапуск

Пакет CTI содержит C# (EntityEventListener, SourceCode-схемы). После импорта:

### 3.1 Перекомпилировать конфигурацию

В dev: **Дизайнер системы** → **«Управление конфигурацией»** → кнопка **«Компилировать»** → **«Перекомпилировать все»**

Дождаться уведомления о завершении компиляции (раздел Конфигурация блокируется на время компиляции).

### 3.2 Перезапустить приложение (Kestrel)

На dev-сервере используется Kestrel (не IIS). После компиляции C# изменения вступят в силу только после перезапуска:

```bash
ssh -i ~/.ssh/id_bpmsoft_dev gore@192.168.102.46

# Найти имя сервиса
sudo systemctl list-units --type=service | grep -i bpm

# Перезапустить
sudo systemctl restart <имя_сервиса>

# Проверить статус
sudo systemctl status <имя_сервиса>
```

### 3.3 Перелогиниться в браузере

После перезапуска — выйти из учётной записи и войти снова, чтобы изменения UI вступили в силу.

## Шаг 4. Создание тестовых данных через UI

**Не использовать прямые SQL INSERT** для бизнес-данных — BPMSoft ORM заполняет системные поля (CreatedById, ModifiedById, ProcessListeners и др.), которые не видны в схеме. Вставка в обход ORM нарушает целостность. Создавать всё через интерфейс.

### 4.1 Контакт

**Контакты** → **Новый**:
- Имя: `Тестовый Клиент`
- Email: `test.client@example.com`
- Тип: Клиент

### 4.2 Контрагент

**Контрагенты** → **Новый**:
- Название: `Тестовая Компания`
- Тип: Клиент
- Привязать контакт из шага 4.1

### 4.3 Сервисный элемент (ServiceItem)

Проверить сначала: **Сервисы** → **Услуги и сервисы** — в чистом дистрибутиве могут быть дефолтные записи. Если нет:

**Новый**:
- Название: `Техническая поддержка`
- Код: `TP-001`

### 4.4 Сервисный договор (ServicePact)

**Сервисы** → **Сервисные договоры** → **Новый**:
- Название: `Тестовый договор`
- Контрагент: `Тестовая Компания`
- Дата начала: сегодня, дата окончания: +1 год
- На форме договора в детали **«Услуги и сервисы»** → добавить `Техническая поддержка`

> Связь ServicePact ↔ ServiceItem через деталь на форме (создаёт запись в `ServiceInServicePact`) — нужна для корректного расчёта SLA.

### 4.5 Обращения (Case)

**Сервисная служба** → **Обращения** → **Новый** (создать 2–3 штуки):
- Тема: `Тестовое обращение 1`
- Контакт: `Тестовый Клиент`
- Контрагент: `Тестовая Компания`
- Сервисный договор: `Тестовый договор`
- Ответственный: свой аккаунт
- Описание: `Описание тестовой проблемы`

Создать обращения с разными статусами (Новое, В работе, Решено) — пригодится для тестирования уведомлений при смене статуса.

## Шаг 5. Проверка почтовой интеграции

Gmail OAuth 2.0 настроен на dev с прошлой сессии. Проверить:

1. **Администрирование** → **Email** (или «Почтовые ящики») → статус должен быть активен
2. Проверить что ExchangeListenerService запущен (Docker-микросервис):

```bash
ssh -i ~/.ssh/id_bpmsoft_dev gore@192.168.102.46
sudo docker ps | grep -i exchange
```

## Шаг 6. Проверка кастомизаций

| Что проверяем | Где | Признак успеха |
|---|---|---|
| Поле CC на форме обращения | Открыть любое обращение | Поле «Копия (CC)» присутствует |
| Поле CC на форме договора | Открыть сервисный договор | Поле CC присутствует |
| Индикатор режима обслуживания | Системные настройки | Настройка есть, индикатор отображается |
| BPMN-процессы | Дизайнер системы → Библиотека процессов | `UsrSendEmailToSROwnerCustom1`, `UsrSendNotificationToCaseOwnerCustom1` активны |
| C# EventListener | Создать обращение → изменить статус | Письмо уходит с CC |

## Возможные проблемы

| Симптом | Причина | Решение |
|---|---|---|
| Поля CC не видны | Kestrel не перезапущен | Перезапуск сервиса на dev |
| Пакет не ставится — ошибка зависимостей | На dev нет базовых пакетов нужной версии | Скачать лог ошибки → разобраться какого пакета не хватает |
| BPMN-процессы неактивны | После импорта процессы деактивируются | Активировать вручную в дизайнере процессов |
| Письма не уходят | Токен OAuth истёк или ExchangeListenerService упал | Переподключить почту через UI; перезапустить Docker-контейнер |
| CC есть, но письма без CC | `UsrActivityCcEventListener` не зарегистрировался | Убедиться что Kestrel перезапущен после импорта |
