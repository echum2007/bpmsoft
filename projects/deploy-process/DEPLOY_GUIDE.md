# Инструкция по переносу изменений на прод BPMSoft

**Обновлено:** 2026-04-23  
**Источник:** Документация BPMSoft 1.9 (NotebookLM, раздел «Управление поставками»)

---

## Концепция

```text
Dev-сервер (режим РФС включён)
    │
    │  1. Разработка: редактируем файлы в VS Code
    │     git pull → «Обновить пакеты из РФС» в UI → публикуем схемы → тестируем
    │
    │  2. Экспорт готового CTI → .zip
    ▼
Прод-сервер
    │  Импорт .zip через «Установка и удаление приложений»
    ▼
Готово
```

**Git** — система хранения исходников и истории изменений, **не механизм деплоя**.  
**Деплой** — всегда через экспорт `.zip` → импорт через UI BPMSoft.

---

## Как работает режим РФС (важно понимать)

На **dev-сервере** включён режим разработки в файловой системе (`fileDesignMode enabled="true"`).  
Это означает:

- Редактируем файлы в VS Code → `git pull` подтягивает изменения на сервер
- Идём в UI BPMSoft → **Конфигурация → Действия → «Обновить пакеты из файловой системы»** — изменения из файлов попадают в БД
- Для C#-схем: после обновления из РФС — **«Компилировать»**
- Для JS-схем: достаточно очистить кэш браузера

> ⚠️ **На проде РФС отключён.** Прямое редактирование файлов на проде недопустимо.  
> ⚠️ **Перед импортом на прод** рекомендуется отключить РФС на dev (`fileDesignMode enabled="false"`), чтобы избежать рассинхронизации.

## Чеклист переноса на прод

```text
□ Шаг 1. Убедиться что изменения протестированы на dev
□ Шаг 2. Экспортировать ТЕКУЩИЙ прод → сохранить как страховку отката
□ Шаг 3. Поставить git-тег на текущий master (точка отката в git)
□ Шаг 4. Смержить feature → dev → master, пушнуть
□ Шаг 5. Поставить git-тег на новый master
□ Шаг 6. Экспортировать CTI с dev → сохранить архив
□ Шаг 7. Импортировать .zip на прод
□ Шаг 8. Перезапустить Kestrel (если был C# / EventListener)
□ Шаг 9. Перелогиниться в браузере на проде
□ Шаг 10. Проверить функциональность на проде
   □ OK → готово
   □ Проблема → Шаг 11
□ Шаг 11 (откат). Восстановить предыдущую версию
```

---

### Шаг 1. Убедиться что изменения протестированы на dev

Перед деплоем на прод убедиться:

- Все схемы опубликованы / скомпилированы на dev
- Тест-план задачи пройден
- Нет незакоммиченных изменений в git

---

### Шаг 2. Экспортировать текущий прод (СТРАХОВКА)

В BPMSoft **прод** UI:

```text
Дизайнер системы → Конфигурирование разработчиком → Управление конфигурацией
→ Пакет CTI → «...» → «Экспортировать»
→ Сохранить как: projects/prod-releases/CTI_YYYY-MM-DD_до-[задача].zip
```

Пример: `CTI_2026-04-23_до-labor-records.zip`

---

### Шаг 3. Тег на текущий master

```bash
git checkout master
git tag prod-YYYY-MM-DD-before -m "прод до: [название задачи]"
git push origin --tags
```

---

### Шаг 4. Мерж feature → dev → master

```bash
git checkout dev
git merge feature/название-задачи

git checkout master
git merge dev
git push origin master
```

---

### Шаг 5. Тег на новый master

```bash
git tag prod-YYYY-MM-DD-[задача] -m "прод: [название задачи]"
git push origin --tags
```

---

### Шаг 6. Экспортировать CTI с dev

> ⚠️ **Не использовать старый .zip повторно** — объекты, удалённые в БД, могут сохраняться в архивных копиях. Всегда экспортировать заново с dev.

В BPMSoft **dev** UI:

```text
Дизайнер системы → Управление конфигурацией
→ Пакет CTI → «...» → «Экспортировать»
→ Сохранить как: projects/prod-releases/CTI_YYYY-MM-DD_[задача].zip
```

---

### Шаг 7. Импортировать на прод

В BPMSoft **прод** UI:

```text
Дизайнер системы → Приложения → Установка и удаление приложений
→ «Добавить приложение» → «Установить из файла»
→ Выбрать .zip из Шага 6
```

Система автоматически создаст резервную копию конфигурации перед установкой (займёт несколько минут). После завершения — сообщение об успехе или ошибке с возможностью скачать лог.

---

### Шаг 8. Перезапустить Kestrel (если нужно)

| Тип изменений | Перезапуск |
| --- | --- |
| C# (Исходный код) | **Да** |
| EntityEventListener | **Да** |
| IMacrosInvokable | **Да** |
| JS / BPMN (без скриптов) | Нет |

```bash
ssh gore@192.168.102.46  # подключиться к прод-серверу
sudo systemctl restart bpmsoft
sudo systemctl status bpmsoft
```

---

### Шаг 9. Перелогиниться в браузере на проде

Выйти из учётной записи и войти снова — необходимо для применения изменений UI.

---

### Шаг 10. Проверка

Проверить функциональность по тест-плану задачи.

---

### Шаг 11. Откат

**Быстрый откат через UI** (рекомендуется — система уже сделала резервную копию на Шаге 7):

```text
Дизайнер системы → Приложения → Установка и удаление приложений
→ «Восстановить пакеты из резервной копии»
```

**Ручной откат** (если автоматическая резервная копия недоступна):

```text
Дизайнер системы → Приложения → Установка и удаление приложений
→ «Добавить приложение» → «Установить из файла»
→ Выбрать архив из Шага 2 (CTI_..._до-[задача].zip)
```

После отката — перезапустить Kestrel если нужно.

---

## Хранение архивов

Держать последние **5 архивов прода**. Хранить локально в `projects/prod-releases/` (`.zip` в `.gitignore`).

```text
projects/prod-releases/
├── CTI_2026-04-11_до-wave2.zip       ← перед текущим (ДЕРЖАТЬ)
├── CTI_2026-04-11_wave2.zip          ← текущий прод
└── ...
```

## Справка: git-теги

```bash
# Посмотреть историю релизов
git tag -l "prod-*" --sort=-version:refname

# Посмотреть что было в теге
git show prod-2026-04-11-wave2

# Восстановить состояние кода на момент тега (только просмотр)
git checkout prod-2026-04-11-wave2
git checkout master  # вернуться
```

## Рабочий процесс разработки (для справки)

```bash
# Начало задачи
git checkout dev && git pull origin dev
git checkout -b feature/название-задачи

# Разработка: редактируем файлы в VS Code
# На dev-сервере:
#   git pull origin feature/название-задачи
#   UI BPMSoft → Конфигурация → Действия → «Обновить пакеты из файловой системы»
#   Публикуем схемы → тестируем

# Коммит
git add src/CTI/CTI/
git commit -m "feature: описание изменения"
git push origin feature/название-задачи
```
