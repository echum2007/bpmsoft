Описание работающего механизма (CTI / BPMSoft 1.9). Источники: анализ кода + документация BPMSoft v1.9 + выгрузка vwprocesslib с прода (15.04.2026)

Дата: 15.04.2026 (v2)  |  Архив прода: CTI от 2026-04-11

## 0. Кому отправляются уведомления — логика получателей

В обращении есть два разных контакта на стороне клиента:

- **«Инициатор»** — контакт, который зарегистрировал обращение (кто написал/позвонил)
- **«Пользователь услуги»** — контакт, для которого зарегистрировано обращение (кто фактически пострадал)

Это могут быть разные люди. Например: секретарь (инициатор) подала обращение за директора (пользователь услуги).

Контактное лицо (кому реально уйдёт письмо) определяется системными настройками:

| Системная настройка | Код | По умолчанию | Что означает |
|---|---|---|---|
| Оповещение пользователя услуги как контактного лица | DefaultNotifyContactInCase | ВКЛЮЧЕНА | Письма клиенту уходят на «Пользователь услуги». По умолчанию это главный получатель. |
| Оповещение инициатора как контактного лица | DefaultNotifyInitiatorInCase | ВЫКЛЮЧЕНА | Письма клиенту уходят на «Инициатора». По умолчанию выключено — инициатор не получает. |

**Примечание:** Если обе настройки включены — письма уходят обоим. Если инициатор = пользователь услуги — кнопки выбора контактного лица на странице обращения не отображаются. На странице обращения инженер может вручную переключить контактное лицо (кнопки рядом с полями). Одна из настроек должна быть включена — иначе уведомления не будут отправляться никому.

**Примечание:** При регистрации обращения из входящего email: уведомление о регистрации получают ВСЕ адресаты исходного письма (кому/копия). Поле «От кого» = адрес почтового ящика службы поддержки, на который пришло письмо. Если обращение создано иначе (звонок, вручную) — «От кого» берётся из системной настройки «E-mail службы поддержки».

## 1. Уведомления в сторону Заказчика (клиента)

Все уведомления клиенту при смене статуса — через C#-класс SendEmailToCaseStatusChanged (feature-toggle SendEmailToCaseOnStatusChangeClass=1), который читает справочник CaseNotificationRule и подбирает шаблон по паре (статус + категория). Для каждого события есть ДВА шаблона: один для Пользователя услуги, другой для Инициатора. Исключение — регистрация: отдельный BPMN-процесс UsrProcess_send_reg_mail.

**Примечание:** CC: к письмам клиенту автоматически добавляются CC-адреса из контракта и аккаунта — кастомизация через UsrActivityCcEventListener (CTI).

Поле «Процитировать оригинальный email» в справочнике CaseNotificationRule позволяет включить цитату исходного письма (ParentActivity), но это корневое письмо, не последний ответ.

### 1.1 События и шаблоны для клиента

| Событие | Получатель | Шаблон (Пользователь услуги) | Шаблон (Инициатор) | Механизм | Тип |
|---|---|---|---|---|---|
| Регистрация обращения | Контактное лицо (по настройке) | Создание нового обращения из чата | — (отдельного нет) | UsrProcess_send_reg_mail | BPMN-процесс (CTI) |
| -> «В работе» | Контактное лицо (по настройке) | Сообщение о взятии обращения в работу | Сообщение о взятии обращения в работу (для инициатора) | SendEmailToCaseStatusChanged + CaseNotificationRule | C#-класс (CaseService) + справочник |

### 1.2 Текст ответа инженера в уведомлении заказчику

**Текущее состояние:** текст ответа инженера **НЕ включается** в email заказчику. Шаблон «Сообщение о получении нового ответа по обращению» содержит только номер/тему/статус.

**Возможное улучшение:** invokable-макрос `[#@Invoke.UsrLastEngineerReply#]` в шаблоне. Запланировано как задача 2.6.

### 1.3 Имя отправителя

**Текущее состояние:** все письма уходят от `servicedesk@cti.ru` без подписи инженера.

**Возможное улучшение:** добавить макрос `[#Owner.Name#]` в подвал шаблонов. Запланировано как задача 2.7.

### 2.1 Текущие механизмы (три разных, без единой конфигурации)

| Событие | Получатель | Шаблон письма | Содержание | Механизм | Тип |
|---|---|---|---|---|---|
| Назначен ответственным | Назначенный сотрудник | Назначение ответственного в обращении | «Вас назначили ответственным по обращению №...» | UsrSendEmailToSROwnerCustom1 | BPMN-процесс (CTI) |
| Создано обращение с ответственным | Ответственный сотрудник | Назначение ответственного в обращении | То же | UsrSendEmailToSROwnerCustom1 (второй StartSignal) | BPMN-процесс (CTI) |
| Назначено на группу (Owner=NULL, Group!=NULL) | Все сотрудники группы | На группу назначено обращение | «На группу назначено обращение №...» | RunSendEmailToCaseGroupV2 | BPMN-процесс (CaseService) |
| Клиент написал ответ -> «Получен ответ» | Ответственный + CC: роль «1-я линия» | Добавление нового email по обращению | «Получен новый email по обращению №...» **Текст письма клиента НЕ включён.** | UsrProcess_0c71a12CTI5 | BPMN-процесс (CTI) |
| Сообщение с портала | Ответственный | (по шаблону) | — | CasePortalMessageHistoryNotificationProcessML | BPMN (Portal) |

### 2.2 Критические GAP-ы в текущем механизме

| GAP | Описание | Влияние |
|---|---|---|
| **Нет текста письма клиента** | Инженер получает «Получен новый email...» без текста. Надо открывать систему | 3-5 мин на каждое уведомление (задача 2.3) |
| **Фильтр по статусам** | UsrProcess_0c71a12CTI5 срабатывает ТОЛЬКО при смене статуса на «Получен ответ». Если обращение в статусе «Новое» или «В работе» — email инженеру **не уходит** (только push в колокольчик) | Ответ клиента может быть пропущен (задача 2.3/2.4) |
| **Нет единого механизма** | Три разных BPMN на три события. Нет справочника правил (как CaseNotificationRule для клиентов) | Сложно добавлять новые события, нет гибкой настройки |
| **Нет уведомлений о смене статуса** | Инженер не получает email при эскалации, переоткрытии, отмене и т.д. | Задача 2.4 |
| **Нет SLA-предупреждений** | Только визуальные индикаторы на странице | Задача 2.1 |
| **Нет напоминаний о зависании** | Обращение может «зависнуть» в «Получен ответ» | Задача 2.2 |

### 2.3 UsrSendEmailToSROwnerCustom1 — диаграмма (по скриншоту 15.04.2026)

```
StartSignal1: «Обращение назначено на ответственного» (смена Owner)
StartSignal2: «Создано обращение с ответственным» (создание Case с Owner)
  |
  v
[Задание-сценарий] → [ExclusiveGateway]
                          |
        ┌─────────────────┴──────────────────┐
        | «Ответственный = Изменил»          | (другое условие → ?)
        v                                     |
  [Читать данные обращения]                   |
        |                                     |
  [Читать данные ответственного]              |
        |                                     |
        ├── (нет email) → [Terminate]         |
        |                                     |
        | «E-mail ответственного указан»      |
        v                                     |
  [Читать данные схемы Обращение]             |
        |                                     |
  [Обработать шаблон письма с макросами]      |
        |                                     |
  [ExclusiveGateway: «Происхождение по email?»]
        |                                     |
  ┌─────┴─────────────────┐                   |
  | Не по email           | По email          |
  v                       v                   |
  [Добавить данные        [Первая активность] |
   email уведомления]      |                  |
        |                 [Изменить тему       |
        |                  письма (threading)] |
        |                       |              |
        └───────┬───────────────┘              |
                v                              |
          [Отправить письмо] ←─────────────────┘
```

**Примечание:** Процесс обрабатывает ТОЛЬКО назначение ответственного (два сигнала: смена Owner + создание с Owner). Группового назначения здесь нет — для группы работает отдельный процесс `RunSendEmailToCaseGroupV2`.

Ветка «Происхождение по email» — если обращение создано из email, процесс ищет первую Activity (корневое письмо) и меняет тему для email-threading (In-Reply-To).

## 4. Полная цепочка при получении email от клиента

**Шаг 1. BPMSoft получает входящее письмо**
- Создаётся Activity (Type=Email, MessageType=Incoming). Письмо привязывается к обращению по номеру в теме.

**Шаг 2. Сохранение CC-адресов**
- UsrActivityCcEventListener (CTI) фиксирует CC-адреса из входящего письма в Case.UsrCcEmails.

**Шаг 3. Запуск оркестратора**
- Сигнал на новую Activity по обращению -> BPMN RunSendNotificationCaseOwnerProcess (CaseService).

**Шаг 4. Выбор пути (feature-toggle)**
- toggle RunReopenCaseAndNotifyAssigneeClass=1 (на проде) -> C#-класс ReopenCaseAndNotifyAssignee.
- При toggle=0 — шёл бы BPMN UsrSendNotificationToCaseOwnerCustom1 (на проде не используется).

**Шаг 5. Переоткрытие (смена статуса)**
- Предикат ReopeningCondition: если статус IsResolved или IsPaused -> меняет на «Получен ответ».
- Если «Новое», «В работе», уже «Получен ответ» или финальный — статус НЕ меняется.

**Шаг 6. Push-уведомление сотруднику (всегда)**
- NotifyOwner() -> запись Reminding -> колокольчик BPMSoft. Срабатывает ВСЕГДА при наличии ответственного.

**Шаг 7. Email сотруднику (только если статус изменился)**
- Смена статуса на «Получен ответ» -> StartSignal -> UsrProcess_0c71a12CTI5 (CTI).
- Читает данные обращения и email ответственного -> отправляет по шаблону.
- **GAP:** Текст клиентского письма в email НЕ включён.
- **GAP:** Если статус не изменился (обращение в «Новое»/«В работе») — email НЕ уходит.

## 5. Логика статуса «Получен ответ» — откуда берётся перечень статусов

### 5.1 Где задаётся в системе

Путь: Администрирование -> Справочники -> Состояния обращений (таблица CaseStatus)

Каждое состояние имеет флаги:
- «На паузе» (IsPaused) — обращение ожидает чего-либо
- «Является решением» (IsResolved) — обращение считается решённым
- «Финальный» (IsFinal) — обращение завершено (Закрыто/Отменено)
- «Переоткрыто» (IsReopen) — статус «Получен ответ»

Чтобы добавить новый статус в перечень -> поставить ему флаг «На паузе» или «Является решением» в справочнике. Изменения в коде не требуются.

### 5.2 Предикат в C#-коде (ReopenCaseAndNotifyAssignee.cs, CaseService)

Переход в «Получен ответ» происходит, если выполнены ВСЕ три условия:
1. Статус НЕ является уже «Получен ответ» (!IsReopenStatus)
2. Статус НЕ финальный (!IsFinalStatus) — не «Закрыто» и не «Отменено»
3. Статус является «Решённым» ИЛИ «На паузе» (IsResolved OR IsPaused)

Значения флагов читаются из справочника CaseStatus через CaseStatusStore — список не зашит в коде, берётся из БД.

### 5.3 Перечень статусов (с прода, апрель 2026)

| Статус обращения | На паузе (IsPaused) | Решено (IsResolved) | Переход в «Получен ответ»? |
|---|---|---|---|
| Новое | — | — | **Нет** -> email НЕ уйдёт |
| В работе | — | — | **Нет** -> email НЕ уйдёт |
| Ожидает ответа | Да | — | Да |
| Отложен | Да | — | Да |
| Передано сервис-менеджеру | Да | — | Да |
| Направлено в группу специализации | Да | — | Да |
| Находится в работе у вендора | Да | — | Да |
| Передано в отдел логистики | Да | — | Да |
| Ожидаем поставщика | Да | — | Да |
| Ожидается применение релиза | Да | — | Да |
| Ожидает повторение проблемы | Да | — | Да |
| Идёт доработка | Да | — | Да |
| Идёт тестирование | Да | — | Да |
| Передано в отдел разработки | Да | — | Да |

```
═══ ПРИ СМЕНЕ СТАТУСА ОБРАЩЕНИЯ ═══

  Case.Status изменился
    |
    +-- toggle SendEmailToCaseOnStatusChangeClass=1 (наш случай)
    |    -> C#-класс SendEmailToCaseStatusChanged
    |         -> CaseNotificationRule -> шаблон -> email КЛИЕНТУ
    |
    +-- toggle=0
         -> BPMN SendEmailToCaseStatusChangedProcess
              -> SubProcess: SendEmailToCaseContactPersonsProcess (хаб)
                   +-- SendEmailToCaseContactProcessMultiLanguage (контакт ML)
                   +-- SendEmailToCaseContactProcess (контакт legacy)
                   +-- SendEmailToInitiatorMultiLanguage (инициатор ML)
                   +-- SendEmailToInitiator (инициатор legacy)

═══ ПРИ НАЗНАЧЕНИИ ОТВЕТСТВЕННОГО ═══

  Case.Owner изменился ИЛИ Case создан с Owner
    -> UsrSendEmailToSROwnerCustom1 (CTI, замещает SendEmailToSROwner)
         -> email ОТВЕТСТВЕННОМУ

═══ ПРИ НАЗНАЧЕНИИ НА ГРУППУ ═══

  Case создан/изменён, Owner=NULL, Group!=NULL
    -> RunSendEmailToCaseGroupV2 (CaseService)
         -> email ВСЕЙ ГРУППЕ

═══ ПРИ ОТВЕТЕ КЛИЕНТА (EMAIL) ═══

  Входящий email -> Activity -> сигнал
    -> RunSendNotificationCaseOwnerProcess (оркестратор)
         |
         +-- toggle RunReopenCaseAndNotifyAssigneeClass=1 (наш случай)
              -> C#-класс ReopenCaseAndNotifyAssignee.Run()
                   |
                   +-- ReopeningCondition: IsResolved || IsPaused
                   |    -> StatusId = «Получен ответ» (если условие выполнено)
                   |    -> статус НЕ меняется (если «Новое», «В работе» и т.д.)
                   |
                   +-- NotifyOwner() -> push (Reminding) — ВСЕГДА
                   |
                   +-- [Побочный эффект: смена статуса -> StartSignal]
                        -> UsrProcess_0c71a12CTI5 (CTI)
                             -> email ОТВЕТСТВЕННОМУ + CC: роль «1-я линия»
                             -> шаблон: «Добавление нового email по обращению»

    -> UsrProcess_send_reg_mail (CTI) -> email КЛИЕНТУ
```

## 10. Архитектурное решение: UsrEmployeeNotificationManager

### 10.1 Проблема

Для клиентов существует единый структурированный механизм: C#-класс `SendEmailToCaseStatusChanged` + справочник `CaseNotificationRule` (статус + категория -> шаблон). Для сотрудников такого механизма нет — есть три разрозненных BPMN-процесса, привязанных к конкретным событиям, без общей конфигурации.

### 10.2 Решение

Создать единый C#-класс **`UsrEmployeeNotificationManager`** + справочник правил **`UsrEmployeeNotificationRule`** — по аналогии с механизмом для клиентов.

### 10.3 Справочник UsrEmployeeNotificationRule

| Поле | Тип | Назначение |
|------|-----|-----------|
| Событие (UsrEvent) | Справочник | Тип события: ОтветКлиента, НазначениеОтветственного, НазначениеГруппы, СменаСтатуса, SLA-порог, Зависание |
| Статус обращения (StatusId) | Справочник (CaseStatus) | Для каких статусов срабатывает (NULL = любой) |
| Шаблон email (EmailTemplateId) | Справочник (EmailTemplate) | Какой шаблон использовать |
| Получатель (UsrRecipientType) | Справочник | Ответственный / Группа / Руководитель группы / Наблюдатели |
| Правило использования | Справочник | Отправляется сразу / С задержкой / Не используется |
| Задержка, мин | Integer | Пауза перед отправкой |
| Активно | Boolean | Вкл/Выкл |

### 10.4 Что заменяет

| Текущий механизм | Что станет |
|---|---|
| UsrProcess_0c71a12CTI5 (email при «Получен ответ») | Правило: событие=ОтветКлиента, статус=любой |
| UsrSendEmailToSROwnerCustom1 (email при назначении) | Правило: событие=НазначениеОтветственного |
| RunSendEmailToCaseGroupV2 (email группе) | Правило: событие=НазначениеГруппы |
| (нет) | Правило: событие=СменаСтатуса (задача 2.4) |
| (нет) | Правило: событие=SLA-порог (задача 2.1) |
| (нет) | Правило: событие=Зависание (задача 2.2) |

### 10.5 Точки вызова

- **EventListener на Case (OnUpdated)** — смена статуса, смена ответственного
- **EventListener на Activity (OnInserted)** — входящий email (заменяет косвенную связь через смену статуса)
- **Фоновый процесс (BPMN-таймер / CronJob)** — SLA-пороги, зависание

### 10.6 Порядок реализации

1. **Задача 2.3** — первая реализация через UsrEmployeeNotificationManager (событие: ответ клиента). Включает текст клиентского письма.
2. **Задачи 2.2, 2.4, 2.1** — подключаются как новые события в том же справочнике.
3. Миграция существующих процессов (UsrSendEmailToSROwnerCustom1, UsrProcess_0c71a12CTI5) — после стабилизации нового механизма.
