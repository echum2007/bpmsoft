## Текущая архитектура уведомлений

### Как работает цепочка

```
Событие (смена статуса Case, новый комментарий)
   ↓
BPMN-процесс (UsrSendNotificationToCaseOwnerCustom1 / SendEmailToSROwner и т.д.)
   ↓
CaseNotificationRule → выбор шаблона EmailTemplate по статусу
   ↓
EmailWithMacrosManager.SendEmail(caseId, templateId)
   ↓
BaseEmailWithMacrosManager.GetTemplateBody() → читает шаблон из БД (EmailTemplate / EmailTemplateLang)
   ↓
MacrosProcessor.GetTextTemplate() → резолвит макросы [#Case.Number#], [#@Invoke.SymptomsGenerator#] и т.д.
   ↓
Создаётся Activity (Type=Email) → UsrCaseEmailNotificationEventListener добавляет CC-адреса
   ↓
AsyncEmailSender → IEmailClient → SMTP/Exchange
```

### Ключевые компоненты

| Компонент | Пакет | Назначение |
|-----------|-------|------------|
| `BaseEmailWithMacrosManager` | CaseService | Базовый: чтение шаблона из БД, резолв макросов, создание Activity |
| `EmailWithMacrosManager` | CaseService | Расширение для Case: получение CaseData (контакт, инициатор, CC, ParentActivity) |
| **`ExtendedEmailWithMacrosManager`** | CaseService | **Добавляет тело родительского письма как цитату!** |
| `EmailFooterSupplier` | CaseService | Форматирует цитату: From, Sent, To, CC, Subject + Body |
| `InvokableMacrosHelper` | CaseService | Расширенные макросы `[#@Invoke.*#]` через рефлексию |
| `CaseNotificationRule` | CaseService | Справочник: статус → шаблон email |
| `UsrCaseEmailNotificationEventListener` | CTI | Наш EventListener: инжектит CC-адреса из контракта и аккаунта |

### Система макросов

Шаблоны хранятся в таблице `EmailTemplate`. Поддерживаются:

- **Колоночные макросы:** `[#Number#]`, `[#Subject#]`, `[#Contact.Name#]`, `[#Status.Name#]`, `[#Owner.Name#]`
- **Навигация по связям:** `[#ServicePact.Name#]`, `[#Contact.Account.Name#]`
- **Invokable-макросы:** `[#@Invoke.SymptomsGenerator#]`, `[#@Invoke.EstimateLinksGenerator#]` — вызов C#-классов через рефлексию
- **Даты с таймзоной:** `CaseTimeZoneMacrosConverter`
- **Мультиязычность:** `EmailTemplateLang` + `MLangContentFactory`

## Оценка текущей системы

### Сильные стороны

1. **Шаблоны в БД** — редактируются через UI без компиляции и перезапуска
2. **Макросы с навигацией** — подстановка данных из любых связанных сущностей через точечную нотацию
3. **Invokable-макросы** — позволяют встраивать произвольную логику C# в шаблон
4. **Async-отправка** — не блокирует основные операции
5. **CC-инжекция** (наша доработка) — автоматические копии по контракту и аккаунту
6. **Мультиязычность** — поддержка `EmailTemplateLang` для разных языков
7. **Расширяемость** — `ExtendedEmailWithMacrosManager` и наследование показывают, что архитектура готова к расширениям
8. **Email threading** — In-Reply-To / MessageId обеспечивают группировку писем в почтовом клиенте

### Слабые стороны

1. **«Слепые» уведомления** — инженер получает «пришёл ответ», но не видит текст. Надо открывать систему, искать обращение
2. **Нет SLA-эскалаций** — система не предупреждает о приближении к порогам (75/85/95%). Просрочка обнаруживается постфактум
3. **Нет наблюдателей** — нельзя «подписаться» на обращение без назначения ответственным
4. **Нет контроля зависания** — обращение может «зависнуть» в статусе «Получен ответ» на дни без напоминания
5. **Шаблоны не в коде** — хранятся только в БД, не в пакете CTI. Нет версионирования, нет переноса через Git
6. **Один канал** — только email. Нет push в интерфейсе, Telegram только заготовка
7. **BPMN непрозрачны** — логика уведомлений в визуальных процессах, тяжело отлаживать и отслеживать изменения
8. **Нет дайджестов** — каждое событие = отдельное письмо. Инженер с 20+ обращениями получает шквал
9. **Footer не локализован** — `EmailFooterSupplier` использует английские подписи (From, Sent on, To, Cc, Subject) — не критично, но заметно
10. **ParentActivity — не всегда последнее письмо** — `Case.ParentActivity` ссылается на корневое письмо, а не на последний ответ клиента

## Критическая находка: ExtendedEmailWithMacrosManager

В платформе **уже есть** класс, который добавляет тело родительского письма в уведомление:

```csharp
// ExtendedEmailWithMacrosManager.cs (CaseService)
protected override void FillActivityWithCaseData(Activity activity, CaseData data) {
    base.FillActivityWithCaseData(activity, data);
    var replyPart = GetReplyPartFromParentActivity(data.ParentActivityId);
    activity.Body = activity.Body + replyPart;  // ← ДОБАВЛЯЕТ ТЕЛО ПИСЬМА КАК ЦИТАТУ
}
```

`EmailFooterSupplier` форматирует это красиво — с HR-линией, заголовками и полным телом.

**Проблема:** текущий BPMN-процесс использует `EmailWithMacrosManager`, а не `ExtendedEmailWithMacrosManager`. Поэтому цитата не добавляется.

**Вторая проблема:** `data.ParentActivityId` — это `Case.ParentActivity`, т.е. **корневое** письмо (которым создано обращение), а не **последний ответ клиента**. Для задачи 2.3 нужно именно последнее входящее письмо.

## Рекомендации по задачам Волны 2

### 2.3 — Текст письма клиента в уведомлении

**Рекомендуемый подход: кастомный invokable-макрос**

Создать класс `UsrLastCustomerEmailGenerator`, реализующий `IMacrosInvokable`:
- Получает `caseId` из аргументов
- ESQ-запрос: последняя Activity с Type=Email, CaseId=caseId, входящее (MessageType != Outgoing)
- Возвращает отформатированное тело письма (аналогично EmailFooterSupplier)
- Регистрируется как макрос `[#@Invoke.UsrLastCustomerEmailGenerator#]`
- Добавляется в шаблон EmailTemplate в нужное место

**Преимущества:**
- Гибкость: макрос можно поставить в любое место шаблона
- Можно ограничить длину (первые N символов)
- Не затрагивает базовые классы платформы
- Работает со всеми шаблонами, где макрос добавлен

**Альтернативный подход:** переключить BPMN-процесс на `ExtendedEmailWithMacrosManager` — но он берёт `ParentActivity` (корневое письмо), а не последний ответ, поэтому потребуется наследование и переопределение.

### 2.1 — Оповещение о критических кейсах + SLA-пороги (75/85/95%)

**Подход:** EventListener на Case (OnUpdated) + проверка % израсходованного SLA-времени.

При изменении `Case.SolutionDate`, `Case.ResponseDate` или по таймеру:
- Рассчитать текущий % израсходованного времени через `CaseTermCalculationManager`
- При пересечении порога (75%, 85%, 95%) — отправить уведомление ответственному + руководителю группы
- Хранить факт отправки (чтобы не дублировать) в кастомном поле или отдельной таблице

**Важно:** нужен фоновый процесс (BPMN-таймер или CronJob), т.к. EventListener сработает только при изменении записи, а SLA истекает со временем без изменений.

### 2.2 — Информирование при «Получен ответ» + зависание

Два компонента:
1. **При переходе в «Получен ответ»** — уведомление инженеру (уже работает через `UsrSendNotificationToCaseOwnerCustom1`). Дополнить текстом письма (п. 2.3)
2. **Зависание** — фоновый процесс (BPMN с таймером), который находит обращения в статусе «Получен ответ» дольше N часов и шлёт напоминания ответственному. При повторном зависании — эскалация руководителю.

### 2.4 — Уведомление о смене статуса + истекающее время

- Расширить справочник `CaseNotificationRule` — добавить шаблоны для всех значимых переходов статуса (не только «Получен ответ»)
- «Истекающее время» — связать с п. 2.1 (SLA-пороги)
- Рассмотреть уведомление и клиенту (не только инженеру) — лучшая практика сервис-деска

### 2.5 — Роль наблюдателя

Самый сложный пункт. Потребуется:
- Новый объект `UsrCaseWatcher` (связь M:N: Case ↔ Contact/SysAdminUnit)
- Деталь на странице обращения «Наблюдатели»
- Кнопка «Подписаться» (или автоматическая подписка при первом комментарии)
- Модификация всех точек отправки уведомлений для рассылки наблюдателям
- Возможно, настройки подписки: на какие события подписан (все, только статусы, только комментарии)

## Лучшие практики сервис-деска (ITIL / ServiceNow / Jira SM)

1. **Контекстные уведомления** — письмо должно содержать достаточно информации для принятия решения БЕЗ входа в систему: номер, тема, приоритет, SLA-статус, текст последнего сообщения, ссылка
2. **Эскалация по таймеру** — система сама эскалирует, не ждёт пока кто-то заметит
3. **Дифференциация по срочности** — критические = немедленно + Telegram, обычные = email
4. **Минимум шума** — подавление дублей, дайджесты, «тихие часы»
5. **Прозрачность для клиента** — клиент тоже получает информативные письма (статус, что ожидается)
6. **Подписка и наблюдение** — любой участник может «следить» за обращением
7. **Аудит уведомлений** — лог отправленных уведомлений для разбора инцидентов
