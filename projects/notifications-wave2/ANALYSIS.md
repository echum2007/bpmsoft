## 1. Штатная модель уведомлений BPMSoft (по документации)

### 1.1 По каким событиям платформа отправляет email-уведомления

Документация описывает следующие события, при которых система отправляет email:

#### Уведомления клиенту (инициатору / пользователю услуги)

| # | Событие | Шаблон | Кому |
|---|---------|--------|------|
| 1 | **Регистрация обращения** (из email, чата, вручную) | «Создание нового обращения из чата» | Контактное лицо |
| 2 | **Взятие в работу** (переход в «В работе») | «Сообщение о взятии обращения в работу» / «...для инициатора» | Контактное лицо |
| 3 | **Получен новый ответ** по обращению | «Сообщение о получении нового ответа по обращению» / «...для инициатора» | Контактное лицо |
| 4 | **Решение обращения** (переход в «Решено») | «Сообщение о разрешении обращения» / «...для инициатора» | Контактное лицо |
| 5 | **Решение обращения (только решение)** | «Сообщение о разрешении обращения — только с решением» / «...для инициатора» | Контактное лицо |
| 6 | **Закрытие обращения** | «Сообщение о закрытии обращения» / «...для инициатора» | Контактное лицо |
| 7 | **Подтверждение закрытия** (долгое ожидание ответа) | «Подтверждение закрытия обращения» / «...для инициатора» | Контактное лицо |
| 8 | **Отмена обращения** | «Сообщение об отмене обращения» / «...для инициатора» | Контактное лицо |
| 9 | **Запрос оценки** (после решения) | «Запрос оценки по обращению» / «...для инициатора» | Контактное лицо |

#### Уведомления внутренние (сотрудникам)

| # | Событие | Шаблон | Кому |
|---|---------|--------|------|
| 10 | **Назначение ответственного** | «Назначение ответственного в обращении» | Ответственный |
| 11 | **Назначение на группу** | «На группу назначено обращение» | Группа ответственных |
| 12 | **Новый email по обращению** (входящее письмо) | «Добавление нового email по обращению» | Ответственный |

#### Внутрисистемные уведомления (не email)

| # | Событие | Канал |
|---|---------|-------|
| 13 | Переоткрытие (входящее письмо/портал по решённому) | Единое окно (группе) |
| 14 | Задачи, визирование, лента, знаменательные события | Центр уведомлений (коммуникационная панель) |
| 15 | Push-уведомления (браузер) | Всплывающие окна |

### 1.2 Механизм правил уведомлений

Справочник **«Правила уведомлений контакта по обращению»** (`CaseNotificationRule`) определяет:

| Поле | Назначение |
|------|-----------|
| Категория обращения | Инцидент / Запрос на обслуживание / ... |
| Состояние обращения | При переходе в какое состояние срабатывает |
| Шаблон для инициатора | Какой EmailTemplate использовать для инициатора |
| Шаблон для пользователя услуги | Какой EmailTemplate для пользователя услуги |
| Правило использования | «Отправляется сразу» / «Отправляется с задержкой» / «Не используется» |
| Задержка отправки, минут | Пауза перед отправкой (для правила с задержкой) |
| Процитировать оригинальный email | Да/Нет — включать ли цитату исходного письма |

**Контактное лицо** определяется системными настройками:
- `DefaultNotifyContactInCase` — пользователь услуги (по умолчанию ВКЛ)
- `DefaultNotifyInitiatorInCase` — инициатор (по умолчанию ВЫКЛ)

### 1.3 SLA-расчёты (документация)

Документация описывает **только расчёт и отображение** сроков реакции/разрешения:
- Правила расчёта сроков (5 стратегий: по сервису, по договору, по приоритету и т.д.)
- Индикаторы оставшегося времени на странице обращения (оранжевый цвет при просрочке)
- Возможность просмотра параметров расчёта

**Важно:** документация НЕ описывает никаких email-уведомлений о приближении SLA-порогов или просрочках. Есть только визуальные индикаторы на странице обращения.

### 1.4 Что НЕ описано в документации (отсутствует в штатной модели)

| Функция | Статус в BPMSoft |
|---------|-----------------|
| **SLA-пороговые уведомления** (75/85/95%) | Нет — только визуальные индикаторы |
| **Напоминания о зависших обращениях** | Нет штатного механизма |
| **Роль наблюдателя** (подписка без ответственности) | Нет |
| **Текст письма клиента в уведомлении инженеру** | Частично — есть опция «Процитировать оригинальный email» + ExtendedEmailWithMacrosManager, но по умолчанию не используется для внутренних уведомлений |
| **Дайджесты** (сводка вместо потока отдельных писем) | Нет |
| **Уведомления в Telegram** | Нет (только заготовка) |
| **Эскалация руководителю при зависании** | Нет автоматической |

### 2.4 Анализ поля «Категория обращения» (CaseCategory)

Справочник с двумя значениями (ITIL-классификация):
- **Инцидент** (`1B0BC159-...`) — сбой, неожиданная проблема
- **Запрос на обслуживание** (`1C0BC159-...`) — плановая операция

**Где участвует в технологии работы (по коду):**

| Точка влияния | Как работает |
|---------------|-------------|
| **Правила уведомлений** | `SendEmailToCaseStatusChanged` фильтрует `CaseNotificationRule` по паре (Status + Category) → разные шаблоны для инцидента и запроса |
| **Выбор почтового ящика** | `EmailSenderObtainer` при включённой настройке `DefineCaseCategoryFromMailBox` фильтрует ящики по категории |
| **Авторегистрация из email** | `IncidentRegistrationFromEmaillHelper` определяет категорию из почтового ящика или системной настройки `DefaultCaseCategory` |
| **Тема портального обращения** | `Case.SetPortalCaseSubject()` — разное слово: «Инцидент по...» vs «Запрос на обслуживание для...» |
| **BPMN-процесс инцидентов** | `IncidentDiagnosticsAndResolvingV2` запускается только для категории «Инцидент» → создаёт задачу диагностики |
| **Действие «Зарегистрировать проблему»** | Доступно только для категории «Инцидент» |
| **Связь с сервисом** | На странице сервиса указана CaseCategory → автозаполнение при выборе сервиса в обращении |

**Где НЕ участвует:**
- **SLA-расчёты** — категория никак не влияет на сроки реакции/разрешения. SLA считается по приоритету, сервису, договору, уровню поддержки
- **Маршрутизация** — нет штатной логики разделения по категориям
- **AI-прогнозирование** — модели прогнозируют сервис и группу, но не категорию

**Принятое решение для Волны 2:**

> Все пороги, тайминги и эскалации привязываем **только к SLA** (плановое разрешение, плановая реакция, приоритет). Категорию **не используем** как критерий строгости контроля. Причина: если привязать строгость к категории, то ошибочный выбор «Запрос на обслуживание» для обращения с высоким SLA снизит степень контроля. Реальная срочность определяется SLA, а не ITIL-классификацией.
>
> При этом наши правила уведомлений в CaseNotificationRule должны работать **одинаково для обеих категорий**, чтобы ошибка в категории не влияла на информирование.

## 3. Архитектура email-уведомлений (по коду)

### 3.1 Цепочка отправки

```
Событие (смена статуса, входящее письмо)
   |
   v
BPMN-процесс (запускается сигналом по Case)
   |
   v
CaseNotificationRule --> выбор шаблона EmailTemplate по состоянию + категории
   |
   v
EmailWithMacrosManager.SendEmail(caseId, templateId)
   |
   v
BaseEmailWithMacrosManager:
   - GetTemplateBody() --> читает шаблон из EmailTemplate / EmailTemplateLang
   - MacrosProcessor.GetTextTemplate() --> резолвит макросы
   |
   v
Создаётся Activity (Type=Email)
   --> UsrCaseEmailNotificationEventListener добавляет CC-адреса
   |
   v
AsyncEmailSender --> IEmailClient --> SMTP/Exchange
```

### 3.2 Ключевые компоненты

| Компонент | Пакет | Назначение |
|-----------|-------|------------|
| `BaseEmailWithMacrosManager` | CaseService | Базовый: чтение шаблона, резолв макросов, создание Activity |
| `EmailWithMacrosManager` | CaseService | Расширение для Case: CaseData (контакт, инициатор, CC, ParentActivity) |
| `ExtendedEmailWithMacrosManager` | CaseService | Добавляет тело родительского письма как цитату |
| `EmailFooterSupplier` | CaseService | Форматирует цитату: From, Sent, To, CC, Subject + Body |
| `InvokableMacrosHelper` | CaseService | Расширенные макросы `[#@Invoke.*#]` через рефлексию |
| `CaseNotificationRule` | CaseService | Справочник: категория + состояние → шаблон email |
| `UsrCaseEmailNotificationEventListener` | CTI | Наш EventListener: инжектит CC-адреса |

### 3.3 Система макросов

Шаблоны хранятся в таблице `EmailTemplate`. Поддерживаются:

- **Колоночные макросы:** `[#Number#]`, `[#Subject#]`, `[#Contact.Name#]`, `[#Status.Name#]`, `[#Owner.Name#]`
- **Навигация по связям:** `[#ServicePact.Name#]`, `[#Contact.Account.Name#]`
- **Invokable-макросы:** `[#@Invoke.SymptomsGenerator#]`, `[#@Invoke.EstimateLinksGenerator#]` — вызов C#-классов через рефлексию
- **Даты с таймзоной:** `CaseTimeZoneMacrosConverter`
- **Мультиязычность:** `EmailTemplateLang` + `MLangContentFactory`

### 3.4 Критическая находка: ExtendedEmailWithMacrosManager

В платформе **уже есть** класс, который добавляет тело родительского письма:

```csharp
// ExtendedEmailWithMacrosManager.cs (CaseService)
protected override void FillActivityWithCaseData(Activity activity, CaseData data) {
    base.FillActivityWithCaseData(activity, data);
    var replyPart = GetReplyPartFromParentActivity(data.ParentActivityId);
    activity.Body = activity.Body + replyPart;
}
```

**Проблема 1:** текущий BPMN использует `EmailWithMacrosManager`, а не `ExtendedEmailWithMacrosManager`.  
**Проблема 2:** `data.ParentActivityId` = `Case.ParentActivity` — это **корневое** письмо, а не последний ответ клиента.

Также в справочнике CaseNotificationRule есть поле **«Процитировать оригинальный email»** — но оно работает через тот же механизм `ParentActivity` и тоже даёт корневое письмо.

> **Уточнение (2026-04-12):** Feature-toggle `EmailMessageMultiLanguageV2` = **1 (включён)** на проде. Конвейер `EmailWithMacrosManager` **активен** для процессов, которые через него идут (например, `UsrSendEmailToSROwnerCustom1`). Однако процесс push-уведомления `UsrSendNotificationToCaseOwnerCustom1` не отправляет email через этот конвейер — он создаёт только Reminding.

## 4. Сравнение: документированная модель vs. лучшие практики ITSM

### 4.1 Что BPMSoft делает хорошо (по документации)

1. **Гибкие правила уведомлений** — справочник позволяет настроить шаблон для каждой пары (категория + состояние)
2. **Разделение инициатор / пользователь услуги** — разные шаблоны для разных ролей
3. **Задержка отправки** — можно отложить уведомление на N минут (защита от «шквала» при быстрых переключениях)
4. **Контактное лицо** — выбор кому слать (инициатору, пользователю услуги, обоим)
5. **Цитирование оригинального email** — опция есть, хотя с ограничениями
6. **Автопереоткрытие** — при входящем письме по решённому обращению оно автоматически переоткрывается
7. **Очереди в Едином окне** — визуальные уведомления о новых обращениях в группе
8. **Email threading** — In-Reply-To / MessageId для группировки писем в почтовом клиенте

### 4.2 Что документация НЕ покрывает (GAP vs. ITIL/ITSM best practices)

| Лучшая практика ITSM | Статус в BPMSoft | Наша потребность |
|-----------------------|-----------------|-----------------|
| **Контекстные уведомления** — письмо содержит достаточно информации для принятия решения БЕЗ входа в систему | Частично: макросы дают номер/тему/статус, но **не текст последнего сообщения клиента** | Задача 2.3 |
| **SLA-эскалация по таймеру** — система сама предупреждает при 75/85/95% израсходованного времени | **Нет** — только визуальные индикаторы на странице | Задача 2.1 |
| **Напоминание о зависании** — если обращение долго в одном статусе без действий | **Нет** штатного механизма | Задача 2.2 |
| **Дифференциация по срочности** — критические = немедленно + альтернативный канал | **Нет** — все уведомления одинаковым способом (email) | Будущее |
| **Подписка/наблюдение** — любой участник может «следить» за обращением | **Нет** | Задача 2.5 |
| **Аудит уведомлений** — лог что/кому/когда отправлено | Частично — Activity создаётся, но нет удобного отчёта | Будущее |
| **Дайджесты** — сводка вместо потока отдельных писем | **Нет** | Будущее |
| **Уведомление инженеру при смене статуса** (не только назначение) | **Нет** — штатно инженер получает только «назначение» и «новый email» | Задача 2.4 |

## 5. Архитектурное решение: UsrEmployeeNotificationManager (принято 15.04.2026)

### 5.1 Обоснование

По результатам обсуждения с командой (15.04.2026) принято решение **не латать существующие процессы**, а создать единый механизм уведомления сотрудников — по аналогии со штатным механизмом для клиентов.

**Проблема:** для клиентов есть единый C#-класс `SendEmailToCaseStatusChanged` + справочник `CaseNotificationRule`. Для сотрудников — три разрозненных BPMN-процесса:
- `UsrSendEmailToSROwnerCustom1` — email при назначении
- `UsrProcess_0c71a12CTI5` — email при «Получен ответ» (только при смене статуса!)
- `ReopenCaseAndNotifyAssignee` — push в колокольчик

Критические GAP-ы текущего подхода:
1. **Фильтр по статусам** — если обращение в «Новое»/«В работе», email сотруднику при ответе клиента НЕ уходит
2. **Нет единой конфигурации** — каждое событие = отдельный BPMN, нет справочника правил
3. **Нет расширяемости** — для каждого нового события нужен новый процесс

### 5.2 Решение: UsrEmployeeNotificationManager + UsrEmployeeNotificationRule

**Класс `UsrEmployeeNotificationManager`** (C#, Исходный код, пакет CTI):
- Единая точка входа для всех уведомлений сотрудникам
- Читает правила из справочника `UsrEmployeeNotificationRule`
- Подбирает шаблон по событию + условию
- Поддерживает макросы (включая invokable для текста клиентского письма)
- Формирует и отправляет email через `EmailWithMacrosManager` или `ActivityEmailSender`

**Справочник `UsrEmployeeNotificationRule`** (объект, пакет CTI):

| Поле | Тип | Назначение |
|------|-----|-----------|
| Событие (UsrEventType) | Справочник | ОтветКлиента, НазначениеОтветственного, НазначениеГруппы, СменаСтатуса, SLA-порог, Зависание |
| Статус обращения (CaseStatusId) | Справочник (CaseStatus) | Для каких статусов (NULL = любой) |
| Шаблон email (EmailTemplateId) | Справочник (EmailTemplate) | Какой шаблон |
| Получатель (UsrRecipientType) | Справочник | Ответственный / Группа / Руководитель / Наблюдатели |
| Правило использования | Справочник | Сразу / С задержкой / Не используется |
| Задержка, мин | Integer | Пауза |
| Активно | Boolean | Вкл/Выкл |

**Точки вызова:**
- EventListener на Case (OnUpdated) — смена статуса, смена ответственного
- EventListener на Activity (OnInserted) — входящий email от клиента
- Фоновый процесс — SLA-пороги (2.1), зависание (2.2)

### 5.3 Что заменяет новый механизм

| Текущий механизм | Событие в UsrEmployeeNotificationRule |
|---|---|
| UsrProcess_0c71a12CTI5 | Событие=ОтветКлиента, статус=любой |
| UsrSendEmailToSROwnerCustom1 | Событие=НазначениеОтветственного |
| RunSendEmailToCaseGroupV2 | Событие=НазначениеГруппы (или параллельно) |
| (нет — задача 2.4) | Событие=СменаСтатуса |
| (нет — задача 2.1) | Событие=SLA-порог |
| (нет — задача 2.2) | Событие=Зависание |

> **Примечание:** миграция существующих процессов (UsrSendEmailToSROwnerCustom1, UsrProcess_0c71a12CTI5) — после стабилизации нового механизма. На первом этапе новый механизм работает параллельно.

## 6. Рекомендации по задачам Волны 2

### 6.1 Задача 2.3 — Текст письма клиента в уведомлении + email из любого статуса (приоритет 1)

**Проблемы:**
1. Сотрудник получает «Получен новый email...», но не видит текст
2. Если обращение в «Новое»/«В работе» — email вообще не уходит

**Подход:** Первая реализация через `UsrEmployeeNotificationManager`:
- Событие = ОтветКлиента
- Триггер: EventListener на Activity (OnInserted), входящий email по обращению
- Шаблон содержит текст клиентского письма (invokable-макрос или прямая подстановка)
- Срабатывает для ЛЮБОГО статуса обращения (без фильтра)

### 6.2 Задача 2.2 — Зависание в «Получен ответ» (приоритет 2)

**Подход:** Фоновый процесс (BPMN-таймер или CronJob):
- Находит обращения в статусе «Получен ответ» дольше N часов
- Вызывает `UsrEmployeeNotificationManager` с событием=Зависание
- При повторном зависании (2N часов) — эскалация руководителю группы

### 6.3 Задача 2.4 — Уведомление сотруднику о смене статуса (приоритет 3)

**Подход:** Добавить событие=СменаСтатуса в `UsrEmployeeNotificationRule`:
- EventListener на Case.OnUpdated отслеживает смену StatusId
- Вызывает `UsrEmployeeNotificationManager`
- Правила определяют: для каких переходов отправлять, какой шаблон

### 6.4 Задача 2.1 — SLA-пороги 75/85/95% (приоритет 4)

**Подход:** Фоновый процесс:
- Периодически (каждые 15-30 минут) проверяет активные обращения
- Рассчитывает % израсходованного времени
- Вызывает `UsrEmployeeNotificationManager` с событием=SLA-порог
- Таблица `UsrSlaNotificationLog` для дедупликации

### 6.5 Задача 2.5 — Роль наблюдателя (приоритет 5)

**Подход:**
- Новый объект `UsrCaseWatcher` (связь M:N: Case <-> Contact)
- Деталь на странице обращения «Наблюдатели»
- `UsrEmployeeNotificationManager` при отправке проверяет наблюдателей и рассылает им тоже

### 6.6 Задача 2.6 — Текст ответа инженера в уведомлении клиенту (новая)

**Проблема:** клиент получает «Получен новый ответ по обращению», но без текста ответа.

**Подход:** Invokable-макрос `[#@Invoke.UsrLastEngineerReply#]` в шаблоне «Сообщение о получении нового ответа по обращению». Встраивается в существующий механизм `SendEmailToCaseStatusChanged` + `CaseNotificationRule`.

### 6.7 Задача 2.7 — Имя отправителя в уведомлении клиенту (новая)

**Проблема:** все письма от servicedesk@cti.ru без подписи инженера.

**Подход:** Макрос `[#Owner.Name#]` в подвал шаблонов (через справочник EmailTemplate, без кода).
