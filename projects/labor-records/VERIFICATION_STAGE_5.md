### 🟡 Находка 5.1 — Сценарии для администратора отсутствуют

**Проблема:** в T-06 матрица прав даёт админу полный доступ. Не проверено, что:
- Админ может вручную исправить запись на любой стадии
- Админ может удалить запись (чего не могут другие роли)
- Админ может вручную перевести стадию (для расшивки заблокированных кейсов)

**Правка:** добавить в T-18 сценарий 11:
> 11. Администратор — ручное вмешательство. Создать запись инженером, привести в стадию "На утверждении". Под админом: открыть карточку → изменить стадию вручную → удалить запись → проверить отсутствие в БД. Цель: убедиться, что у админа есть полный контроль для расшивки нештатных ситуаций.

### 🔴 Находка 5.2 — Negative сценарий "не-инженер пишет в ленту" отсутствует

**Проблема:** Q5 явно говорит — popup открывается ТОЛЬКО для инженеров. Если другая роль (СМ, руководитель, админ) откроет CasePage и нажмёт "Опубликовать" — popup НЕ должен появиться, сообщение должно опубликоваться как обычно.

В T-18 этого сценария нет. Если регрессия — другие роли получат лишний popup и не смогут писать в ленту.

**Правка:** добавить в T-18 сценарий 12:
> 12. Не-инженер пишет в ленту. Под СМ (или любой не-инженерной ролью) открыть CasePage → написать сообщение → нажать "Опубликовать". Ожидается: popup НЕ открывается, сообщение публикуется напрямую. Цель: подтвердить, что регрессия для других ролей отсутствует (Q5).

### 🟡 Находка 5.3 — Сервис-менеджер отклоняет визу

**Проблема:** в сценариях 2 и 7 СМ только утверждает или работает с подсветкой. Сценарий "СМ отклонил визу → стадия 'На доработке' → инженер дорабатывает" — не покрыт. Но именно этот путь проверяет половину DCM-логики.

Сценарий 8 (возврат на доработку) — это отклонение **руководителя**, не СМ.

**Правка:** добавить в T-18 сценарий 13:
> 13. СМ отклоняет на этапе верификации. Создать запись с переработкой, дойти до стадии "На верификации". Под СМ: отклонить визу с комментарием "Не вижу подтверждения переработки". Ожидается: стадия → "На доработке", виза руководителя НЕ создана, инженер получил email со ссылкой и комментарием. Цель: проверить переход СМ → "На доработке" (отдельно от руководителя).

### 🔴 Находка 5.4 — Сценарий 9 покрывает только 1 из 3 уровней защиты Q1

**Проблема:** ARCHITECTURE раздел 7 сценарий 9 описывает оба пути:
> В popup-lookup «Обращение» соответствующие обращения не отображаются (фильтр) — основной путь блокируется на UI.
> Если запись всё-таки создаётся (например, через деталь или импорт) → OnSaving бросает Exception

В T-18 сценарий 9 описан одной строкой:
> 9. Защита от договоров без СМ → ошибка с понятным текстом.

Это покрывает только уровень 2 (сервер). Не указано:
- Проверка уровня 1: попытка сохранить ServicePact с пустым СМ → UI required ошибка
- Проверка уровня 3: lookup в popup/детали скрывает Case без СМ

И уровень 3 — блокер 2.1 (не реализован).

**Правка:** разбить сценарий 9 в T-18 на три:
> 9a. Уровень 1: попытка сохранить ServicePact с пустым `UsrResponsibleServiceManager` → UI блокирует ("обязательное поле").
> 9b. Уровень 2: запись через деталь на договор с пустым СМ → EventListener throws → пользователь видит модальное окно с понятным текстом.
> 9c. Уровень 3: в popup на CasePage с пустым СМ договора → popup НЕ открывается, показано сообщение "обратитесь к руководителю сервиса". Аналогично — lookup "Обращение" в детали скрывает такие Case.

### 🟡 Находка 5.5 — Регресс старых функций не проверяется

**Проблема:** в T-18 нет проверки, что после внедрения работает то, что работало раньше:
- Существующие бизнес-правила автозаполнения (UsrServicePact, UsrLeadNumber из UsrCase)
- Деталь UsrSchema021c2f40Detail на CasePage и ServicePactPage остаётся видимой
- Старые записи UsrLaborRecords (созданные до внедрения) не сломаны
- Существующие отчёты по UsrLaborRecords (если есть) продолжают работать

**Правка:** добавить в T-18 сценарий 14:
> 14. Регресс. Открыть CasePage существующего обращения с уже имеющимися записями трудозатрат (созданными до внедрения). Проверить: (a) деталь видна и показывает старые записи, (b) старые записи можно открывать без ошибок (несмотря на отсутствие UsrWorkDate, UsrLaborRecordsStage), (c) автозаполнение UsrServicePact из UsrCase работает при создании новой записи через деталь.

### 🟡 Находка 5.6 — Производительность не оценена количественно

**Проблема:** в T-18 чеклист:
> Performance: открытие popup ≤ 1 сек, сохранение записи ≤ 2 сек.

Это всё. Не покрыто:
- Загрузка реестра с 1000+ записями (типичная нагрузка для аналитика)
- Применение фильтра "моё внимание" с большим кол-вом записей
- BPMN отправка email (может тормозить, если SMTP-сервер медленный)
- Время отклика DCM при переходе стадий

**Правка:** дополнить чеклист T-18:
> - Раздел "Управление трудозатратами" с 1000+ записей открывается ≤ 3 сек
> - Фильтр "Моё внимание" применяется ≤ 1 сек
> - Email после смены стадии приходит ≤ 30 сек
> - Переход DCM между стадиями (после клика "Утвердить") ≤ 2 сек
