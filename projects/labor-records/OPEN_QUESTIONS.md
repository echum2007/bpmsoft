# Подсистема трудозатрат — Вопросы к заказчику перед стартом разработки

**Дата:** 2026-04-24
**Основано на:** `ARCHITECTURE_v1.md`, раздел 8
**Статус:** Ожидаем ответы заказчика перед финализацией ТЗ v1.6

---

## Зачем эти вопросы

Мы выбрали каноническую архитектуру BPMSoft (Визирование + DCM) и прошли ТЗ v1.5 с карандашом. Нашлись 10 мест, где решение заказчика напрямую влияет на архитектуру — без ответов стартовать разработку нельзя, иначе либо угадываем и потом переделываем, либо блокируемся на середине этапа.

Вопросы отсортированы по важности для архитектуры. Каждый даёт краткий контекст «почему спрашиваем» — если заказчик не в контексте, этого достаточно, чтобы понять зачем.

## Блок 1 — Процесс согласования

### Q1. Договор без ответственного сервис-менеджера — куда уходит виза?

**Контекст:** В договор добавляется новое поле «Ответственный сервис-менеджер». Для старых договоров оно будет пустым (или заполнится не сразу).

**Варианты:**
- **(a)** Виза уходит **роли** «Руководители сервис-менеджеров» (не конкретному человеку) — любой из роли может взять.
- **(b)** Виза **не создаётся**, запись переходит сразу на Этап 2 (к руководителю).
- **(c)** Запись блокируется с ошибкой «У договора не назначен ответственный».

**Рекомендую (a)** — не блокирует работу, аудит сохраняется, миграция старых договоров идёт в своём темпе.

---

### Q3. Возврат руководителем на доработку — после исправления цикл с какого этапа?

**Контекст:** Руководитель (Этап 3) возвращает на доработку. Инженер правит запись и переотправляет.

**Варианты:**
- **(a)** Цикл **с Этапа 1** — сервис-менеджер верифицирует заново, потом снова руководитель.
- **(b)** Цикл **с Этапа 2** — сразу к руководителю, без повторной верификации СМ.

**Рекомендую (a)** — каноническое поведение визирования, логичнее (исправления могли повлиять на то, что проверяет сервис-менеджер).

---

### Q4. Записи БЕЗ переработки — запускать ли на них DCM-кейс?

**Контекст:** Обычная запись трудозатрат (не переработка) согласования не требует. Вопрос — проходит ли она через кейс (стадия «Завершено» сразу) или кейс на ней вообще не запускается.

**Варианты:**
- **(a)** Кейс запускается, стадия сразу = «Завершено». Единообразие: у любой записи есть стадия, реестр фильтруется одинаково.
- **(b)** Кейс запускается только для переработок. Проще, но в реестре стадия будет пустой у обычных записей.

**Рекомендую (a)** — единообразие в реестре для аналитика.

---

### Q7. Удаление старого поля `UsrOverTimeApproval` (Boolean)

**Контекст:** На проде сейчас есть поле `UsrOverTimeApproval` (Boolean). Оно заменяется механизмом виз.

**Варианты:**
- **(a)** Оставить в MVP, держать синхронизированным (`true` = есть положительная виза). В следующей волне удалить.
- **(b)** Удалить сразу. Если у записей значения есть — мигрировать их в визы.

**Рекомендую (a)** — безопаснее, без данных-потерь, даёт время отчётам/интеграциям перестроиться.

## Блок 2 — Popup и карточка списания

### Q5. Popup при отправке сообщения — для кого и для каких сообщений?

**Контекст:** Ключевая фича — карточка появляется при отправке сообщения в ленту обращения.

**Подвопросы:**
- **Q5.1.** Для каких ролей показывается popup?
  - Только для инженеров
  - Для всех, кто пишет в ленту Case
  - По условию (например, текущий пользователь в роли `CTI.Инженеры`)
- **Q5.2.** Попадают ли под popup **автосообщения**?
  - BPMN пишет в ленту (закрытие обращения, смена статуса) — эти сообщения **не** должны триггерить popup (там не с кого списывать)
  - Интеграции (email, телеграм) — аналогично
- **Q5.3.** Ответы клиенту vs внутренние комментарии — одинаково показываем popup?

**Рекомендую:**
- Q5.1: только для инженеров (роль-condition)
- Q5.2: автосообщения — всегда без popup
- Q5.3: одинаково (любое ручное сообщение инженера требует списания)

---

### Q8. `UsrWorkDate` — дата или дата+время?

**Контекст:** Поле «Дата работ» в popup.

**Варианты:**
- **(a)** Только **дата** (`Date`). Проще для пользователя. Но для проверки «формально рабочее время» придётся использовать фиксированный час (например, середину рабочего дня) — теряем точность.
- **(b)** **Дата и время** (`DateTime`). Точнее для календаря, но пользователю надо вводить и время.
- **(c)** Гибрид: хранить `DateTime`, но показывать пользователю только дату; время = момент создания записи (если сегодня) или 12:00 (если задним числом).

**Рекомендую (c)** — UX не страдает, точность сохраняется. В выгрузке время можно скрывать.

---

### Q10. «Не ранее N дней назад» — календарные или рабочие?

**Контекст:** Системная настройка `UsrLaborRecordsMaxBackdateDays = 3` по умолчанию.

**Варианты:**
- **(a)** Календарные дни (включая выходные).
- **(b)** Рабочие дни (по системному календарю).

**Рекомендую (a)** — проще в реализации, предсказуемо пользователю. Если N=3, после длинных выходных можно списать за понедельник в среду. Вариант (b) сложнее и менее предсказуем.

## Блок 3 — Роли и права

### Q2. Аналитик — отдельная роль?

**Контекст:** В таблице участников процесса роль «Аналитик», отдельная от «Руководителя». Но по описанию — «выгружает данные, контроль своевременного заполнения».

**Варианты:**
- **(a)** Отдельная функциональная роль `CTI.Аналитики трудозатрат`. Даёт чтение всех записей + экспорт, но не даёт утверждать.
- **(b)** Объединяем с «Руководителем» — один человек делает и то, и другое.

**Рекомендую (a)** — если заказчик реально выделил эту роль, значит нужен человек вне цепочки согласования, с правами только на чтение и выгрузку. Это полезно для аудита.

## Блок 4 — Детали ТЗ

### Q6. Массовое согласование — в MVP или v2?

**Контекст:** «Утвердить выбранные» — ключевой сценарий, если руководитель согласует пачкой (раз в неделю). Из коробки в BPMSoft нет — реализуется кастомно (~полдня работы).

**Варианты:**
- **(a)** В MVP. Добавляем кнопку «Утвердить выбранные» / «Отклонить выбранные» в разделе.
- **(b)** В v2. Пока только по одной через центр уведомлений + страницу записи.

**Рекомендую (a)** — если руководитель реально будет согласовывать пачкой. Если он будет делать это по мере поступления — можно (b).

---

### Q9. Комментарии верификации/отклонения — где хранить?

**Контекст:** Комментарий при отклонении. Отдельное поле `UsrRejectionComment` на `UsrLaborRecords` или внутри визы?

**Варианты:**
- **(a)** Только внутри визы (`UsrLaborRecordsVisa.Comment`). Каноническое поведение BPMSoft. Для отчётности — JOIN.
- **(b)** Дублировать на `UsrLaborRecords` (для простой выгрузки). Риск рассинхронизации.

**Рекомендую (a)** — каноника. Если аналитику нужен комментарий в Excel — добавим как виртуальную колонку в реестр через JOIN (стандартная практика BPMSoft).
