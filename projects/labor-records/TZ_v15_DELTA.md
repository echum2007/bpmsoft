## Что изменилось в v1.5 → реализация

### 1. Раздел 9 ТЗ v1.5 (Текстовая схема процесса) — **устарел целиком**

| Что в ТЗ v1.5 | Что на самом деле |
|---|---|
| Поле «Тип работ: Аварийные работы» в карточке | **Удалено.** Вместо типа работ — флаг `UsrOverTime` (переработка) |
| Автоопределение переработки по календарю (система ставит сама) | **Изменено.** Инженер сам ставит галку «Переработка». Система только подсвечивает «формально рабочее время» жёлтым (`UsrIsFormallyWorkTime`) для контроля руководителя |
| Статусы: «Не требует согласования / Ожидает согласования / Согласовано / Отклонено» | **Заменены.** Стадии DCM: «Черновик / На верификации / На утверждении / На доработке / Утверждено / Отклонено». Согласование двухуровневое (СМ → Руководитель), а не одноуровневое |

### 2. Раздел 2 ТЗ v1.5 (Участники) — **расширен**

В ТЗ перечислено 4 роли (Инженер, Руководитель, Аналитик, Администратор). На самом деле — **6 ролей** (Q2):
- `CTI.Инженеры` (была)
- `CTI.Ответственные сервис-менеджеры` (новая, верификация переработок по своим договорам)
- `CTI.Руководители сервис-менеджеров` (новая, верификация по любым договорам)
- `CTI.Руководители согласования трудозатрат` (новая, утверждение Этап 2)
- `CTI.Аналитики трудозатрат` (новая отдельная роль с правом экспорта без права утверждать)
- Системный администратор

### 3. Раздел 3 ТЗ v1.5 (Описание процесса) — **уточнено**

| Пункт ТЗ | Уточнение |
|---|---|
| «Каждая запись трудозатрат привязана к сообщению в ленте обращения» | Добавлена колонка `UsrSourceMessage → Activity` (в v1.5 этого FK не было заложено). Заполняется автоматически после публикации сообщения |
| «Дата работ можно указать не ранее чем за N дней» | Добавлена явная **верхняя граница**: дата ≤ Now (нельзя в будущее). N = 3 календарных дня (Q10) |
| «Часы 0..23, минуты 1..59» | Уточнено: итоговая длительность ≥ 1 минута; Hours ∈ [0..23], Minutes ∈ [0..59] (т.е. при Hours=0 минуты ≥ 1) |
| «Popup появляется при сообщении в ленту» | Уточнено (Q5, 2026-04-25): popup обязателен на **ЛЮБОЕ ручное сообщение инженера** в ленте (не только переработка). Признак переработки — опциональная галка в popup |

### 4. Раздел 4 ТЗ v1.5 (Договор) — **дополнено**

Добавлено поле в `ServicePact` (новое в реализации, не было в v1.5): **`UsrResponsibleServiceManager → Contact`** — обязательное (Q1). Защита на 3 уровнях: UI required + EventListener throw + UI-фильтр lookup в popup.

### 5. Раздел 5 ТЗ v1.5 (Цикл согласования) — **подтверждено и уточнено**

Возврат руководителем «На доработку» — **полный цикл с Этапа 1** (Q3). После правок инженера запрашивается **новая виза СМ** (а не сразу руководитель). Реализуется кнопкой «Отправить на согласование заново» на странице записи (T-13.5), видна только автору в стадии «На доработке».

### 6. Раздел 6 ТЗ v1.5 (Права) — **детализировано**

- «СМ видит записи только по своим договорам» — реализуется **через фильтр "Моё внимание"**, не через record-level (Q11, 2026-04-26). Все СМ имеют Read all, но в обычном workflow видят только свои.
- Аналитик имеет Read all без исключений, включая переработки руководителей (Q12, 2026-04-26).

### 7. Раздел 7 ТЗ v1.5 (Выгрузка) — **упрощено для MVP**

«Время конвертируется в минуты автоматически при выгрузке» — реализуется **виртуальной колонкой `UsrTotalMinutes`** в реестре. Кастомной формы выгрузки в MVP нет — только стандартный экспорт реестра в Excel.

### 8. Раздел 8 ТЗ v1.5 (Уведомления) — **расширено**

В ТЗ только 2 события («согласована/отклонена инженеру»). На самом деле — **6 событий**:

| Событие | Получатель | Канал |
|---|---|---|
| Назначена виза (На верификации) | СМ договора | Центр уведомлений + email |
| Назначена виза (На утверждении) | Руководитель | Центр + email |
| Виза положительная | Инженер | Email (шаблон `UsrLaborRecordsVisaPositive`) |
| Виза отрицательная | Инженер | Email с причиной (`UsrLaborRecordsVisaNegative`) |
| Возврат на доработку | Инженер | Email с комментарием (`UsrLaborRecordsSendBack`) |
| Напоминание о просроченной визе | Визирующий | Email (стандартное системное, если настроено) |

Комментарии отклонения хранятся **только в `UsrLaborRecordsVisa.Comment`** (Q9), без дублирования в `UsrLaborRecords`.

## Что добавилось сверх v1.5 (не было в исходном ТЗ)

- **Защита от договоров без СМ** на 3 уровнях (Q1) — критично для непрерывности процесса
- **Подсветка «формально рабочего времени»** (`UsrIsFormallyWorkTime` + жёлтый цвет в реестре) — визуальная подсказка руководителю
- **Серверная валидация** длительности и даты в `UsrLaborRecordsEventListener` — защита от обхода UI (импорт, REST API, SQL)
- **Кнопка «Отправить на согласование заново»** (T-13.5) — техническая необходимость для замыкания цикла Q3
- **Системные настройки:** `UsrLaborRecordsMaxBackdateDays`, `UsrLaborRecordsCalendarId`, `UsrLaborRecordsRequireVisaComment` — настраиваемое поведение
- **DCM-кейс `UsrLaborRecordsCase`** — каноническая платформенная реализация согласования вместо самописного workflow
