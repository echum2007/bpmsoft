трудозатрат

Описание бизнес-процесса

Версия 1.5 — Апрель 2026



1. Текущая ситуация и цель
Что есть сейчас
В BPMSoft уже реализован базовый механизм списания трудозатрат: инженеры, работающие с обращениями в BPMSoft, фиксируют потраченное время прямо в карточке обращения. Однако текущий механизм не совершенен — отсутствует процесс согласования переработок, нет контроля обязательности списания, ограничена возможность анализа и выгрузки данных.
Параллельно используется внешняя система EV-Time, которая обслуживает выгрузку трудозатрат из старой системы сервис-деск ExtraView (EV). По мере перевода заявок из EV в BPMSoft потребность в EV-Time будет снижаться, и со временем она будет исключена из работы.
Цель доработки
Доработать существующий механизм регистрации трудозатрат в BPMSoft:
Добавить процесс согласования переработок (сверхурочных трудозатрат).
Обеспечить контроль обязательности списания при работе с обращениями.
Создать рабочее место с разграничением доступа по ролям: инженер, сервис-менеджер, руководитель, анатилитк.
Подготовить систему к полному отказу от EV-Time, когда все заявки будут обрабатываться только в BPMSoft.

3. Как инженер списывает время
Принцип: списание встроено в рабочий процесс
Списание трудозатрат происходит не отдельным действием, а является частью обычного рабочего процесса. Каждый раз, когда инженер отправляет ответ заказчику, добавляет внутреннее сообщение или комментарий по обращению, система показывает всплывающую карточку списания трудозатрат. Пока карточка не заполнена — сообщение не сохраняется.
Таким образом, каждое сообщение в ленте обращения сопровождается записью о потраченном времени. Это обеспечивает точный и полный учёт: видно, сколько времени заняла каждая конкретная активность по обращению.
Как это выглядит:
1. Инженер работает с обращением: изучает проблему, выполняет диагностику, готовит ответ.
2. Инженер пишет сообщение (ответ заказчику или внутренний комментарий) и нажимает «Отправить» / «Сохранить».
3. Система показывает всплывающую карточку списания трудозатрат с полями:
    — Дата выполнения работ (по умолчанию — сегодня; не ранее N дней назад)
    — Часы и минуты (минимум 1 минута)
    — Признак переработки: [ ] Переработка (ставит инженер самостоятельно)
    — Комментарий к работе (необязательный)
4. Система автоматически заполняет: договор, номер лида, имя сотрудника.
5. Если инженер отметил «Переработка», система проверяет: попадает ли указанное время в рабочие часы по системному календарю. Если да — запись подсвечивается для дополнительной проверки при согласовании.
6. Инженер подтверждает карточку — система одновременно сохраняет и сообщение в ленте, и запись трудозатрат.

Важно: минимальное время списания — 1 минута. Поставить 0 нельзя. Для коротких ответов («Принято, беру в работу») карточка заполняется за несколько секунд.

Связь сообщения и списания: каждая запись трудозатрат привязана к конкретному сообщению в ленте обращения. Это упрощает аудит: всегда можно увидеть, на какой именно ответ было потрачено время.

Ограничения:
Нельзя сохранить сообщение в ленте обращения без заполнения карточки списания.
Дату работ можно указать не ранее чем за N дней до текущей даты (N задаётся системной настройкой, по умолчанию — 3 дня).
Часы: от 0 до 23, минуты: от 1 до 59. Минимальная длительность — 1 минута.

Дополнительно: инженер может добавлять обычные трудозатраты вручную через вкладку «Списание трудозатрат» в обращении — для работ без переписки. Признак «Переработка» при ручном добавлении недоступен: переработка фиксируется только при отправке сообщения в ленту обращения.
4. Что считается переработкой
Переработка — это трудозатраты, выполненные в нерабочее время. Инженер ставит признак «Переработка» самостоятельно при создании записи.
Дополнительный контроль: если признак «Переработка» проставлен, но время записи формально попадает в рабочие часы по системному календарю (задаётся системной переменной), — запись подсвечивается в рабочем месте для тщательной проверки при согласовании.
Записи с признаком «Переработка» проходят многоуровневый цикл согласования: верификация сервис-менеджером, затем утверждение руководителем.
Признак допустимости переработок в договоре
В карточку сервисного договора добавляется новый признак — «Разрешены сверхурочные работы» (да/нет). По умолчанию — нет.
Этот признак не запрещает инженерам фиксировать переработки — он носит информационный характер и используется руководителем при согласовании. Если по договору сверхурочные не предусмотрены, а переработка зафиксирована, руководитель видит это в рабочем месте и принимает решение с учётом условий контракта.
Ответственный сервис-менеджер в договоре
В карточку сервисного договора добавляется поле «Ответственный сервис-менеджер» (справочник — пользователь системы). Именно этот сотрудник получает задачу верификации переработок по данному договору.
Руководитель сервис-менеджеров может верифицировать переработки по всем договорам — независимо от того, кто назначен ответственным. Это регулируется системой прав BPMSoft.
Поле «Ответственный сервис-менеджер» будет полезно и для других задач (маршрутизация обращений, отчётность по договорам и т.д.).
5. Процесс согласования переработок
Все записи с признаком «Переработка» проходят многоуровневое согласование. При принятии решения учитываются:
Условия договора — разрешены ли сверхурочные по данному контракту.
Объём и обоснованность — комментарий инженера о выполненной работе.
Шаги:
Этап 1 — Инженер создаёт запись переработки при отправке сообщения в ленту обращения. Статус → «Ожидает верификации».
Этап 2 — Сервис-менеджер, назначенный ответственным в карточке договора, проверяет: корректность данных, соответствие условиям договора. Если ответственный не назначен — задача уходит его руководителю.
    — Подтверждает → статус «Ожидает утверждения».
    — Возвращает на доработку → статус «На доработке»; инженер получает уведомление с комментарием.
Этап 3 — Руководитель принимает окончательное решение:
    — «Утвердить» → статус «Утверждено», инженер получает уведомление.
    — «Вернуть на доработку» → статус «На доработке», цикл начинается заново с Этапа 1.
    — «Отклонить» → статус «Отклонено» (окончательно), инженер получает уведомление с причиной.
Этап 4 — Аналитик выгружает утверждённые записи для передачи во внешние системы.

Статусы согласования:
Ожидает верификации — создана инженером, ждёт проверки сервис-менеджера.
Ожидает утверждения — верифицирована сервис-менеджером, ждёт решения руководителя.
На доработке — возвращена инженеру с комментарием; после исправления цикл повторяется.
Утверждено — подтверждена руководителем; доступна для выгрузки.
Отклонено — отклонена руководителем окончательно.

6. Рабочее место руководителя
Отдельный раздел «Управление трудозатратами» с разграничением доступа по ролям. Роли реализуются через встроенную систему прав BPMSoft.
Возможности:
Роли и доступ:
    Инженер — создаёт записи; видит только свои.
    Сервис-менеджер — верифицирует переработки по договорам, в которых он назначен ответственным; видит записи только по своим договорам.
    Руководитель сервис-менеджеров — верифицирует переработки по всем договорам.
    Руководитель — утверждает или возвращает; видит все записи.
    Аналитик — выгружает данные; видит все записи.
Просмотр всех записей трудозатрат по всем сотрудникам и договорам.
Гибкая фильтрация: по периоду, сотруднику, договору, обращению, статусу согласования.
Согласование и отклонение переработок (в том числе массовое — несколько записей сразу).
Выгрузка данных в Excel для передачи в финансовые системы.
7. Выгрузка данных
Аналитик или руководитель выгружает трудозатраты в Excel. Перед выгрузкой настраиваются фильтры (период, договор, сотрудник и др.). Время конвертируется в минуты автоматически. Выгружаются утверждённые переработки и обычные трудозатраты.
