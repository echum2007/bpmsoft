# Подсистема трудозатрат — Принятые решения (ответы на Q1–Q10)

**Дата:** 2026-04-24
**Источник:** ответы Евгения по `OPEN_QUESTIONS.md`
**Статус:** Все 10 вопросов закрыты. Архитектура финализирована для формирования ТЗ v1.6 и плана работ.

---

## Общий архитектурный принцип (зафиксирован 2026-04-24)

> **Сначала встроенные механизмы платформы, и только если никак — пилим своё.**
> Чем меньше меняем систему — тем стабильнее.

Этот принцип — базовое правило проекта. Каждое архитектурное решение обосновывается: «почему именно этот механизм». Встроенный — с цитатой из документации, кастомный — с объяснением, почему встроенного нет или недостаточно.

Порядок рассмотрения при проектировании:
1. No-code (мастер разделов, бизнес-правила, справочники, DCM, BPMN)
2. Платформенные программные API (`Визирование`, `ICalendarRepository`, `EntityEventListener`)
3. Кастомная клиентская разработка (замещения схем, AMD, ModalBox)
4. Кастомная серверная разработка (свои сервисы, EventListener)

## Решения

### Q1 — Договор без ответственного СМ: защита на входе

**Решение:** Не позволяем списывать время на договоры без ответственного СМ. Защита — на трёх уровнях:

1. **UI (форма ServicePact):** поле `UsrResponsibleServiceManager` обязательное (business rule на форме + required в EntitySchema).
2. **Сервер (EventListener `UsrLaborRecordsEventListener.OnSaving`):** если `UsrServicePact.UsrResponsibleServiceManager IS NULL` → бросаем исключение с понятным сообщением «Невозможно списать время: у договора не назначен ответственный сервис-менеджер».
3. **UI-фильтр (popup списания):** поле «Обращение» фильтруется — не отображаются обращения, у договоров которых не заполнен СМ. (через фильтр lookup: `UsrServicePact.UsrResponsibleServiceManager IS NOT NULL`).

**Миграция:** до релиза — аудит существующих договоров, заполнить СМ ручным прогоном (количество договоров небольшое, проверяется отдельно).

**Why:** нет списания → нет цикла согласования → нет зависшей визы → нет данных-сирот в отчётах.

### Q2 — Аналитик отдельной ролью

**Решение:** (a) Отдельная функциональная роль `CTI.Аналитики трудозатрат`.

**Права:** чтение всех записей `UsrLaborRecords`, чтение `UsrLaborRecordsVisa`, право экспорта в Excel. **Не** может утверждать визы.

**Why:** аналитик — это отдельный игрок процесса (контроль своевременного заполнения, выгрузка), аудит должен показывать, кто выгрузил данные.

---

### Q3 — Возврат руководителя на доработку → цикл с Этапа 1

**Решение:** После возврата руководителем (Этап 3) инженер правит → кейс возвращается в стадию «На верификации», создаётся **новая виза для сервис-менеджера**. Полный цикл.

**Why:** исправления инженера могли повлиять на то, что проверяет СМ (например, изменил дату/длительность/комментарий). Сервис-менеджер должен подтвердить корректность заново.

**DCM-реализация:** стадия «На доработке» → при редактировании инженером + явном действии «Отправить на согласование заново» → переход на «На верификации» с созданием новой визы СМ.

---

### Q4 — DCM только для переработок

**Решение:** (b) DCM-кейс запускается **только** для записей с `UsrOverTime == true`.

Записи без переработки:
- Не имеют стадии DCM (поле «Стадия» пустое или скрыто в реестре).
- Не создают виз.
- Попадают в выгрузку напрямую (как «готовые к выгрузке»).

**Why:** экономит визы и стадии для 95% записей (обычные трудозатраты без переработки). В реестре колонка «Стадия» показывается только для переработок.

**Фильтр в реестре:** «Все / только переработки / только без переработки» — стадия отображается только в первых двух режимах.

### Q5 — Popup только для сообщений инженеров

**Решение:**
- **Кого:** только пользователи в роли `CTI.Инженеры` (или аналогичной — уточнить в процессе).
- **Какие сообщения:** только **ручные** действия пользователя в UI ленты. Автосообщения BPMN и интеграций — без popup.
- **Какие именно ручные сообщения:** и ответы клиенту, и внутренние комментарии — одинаково (любое ручное сообщение инженера требует списания).

**Реализация:** в замещении `SocialMessagePublisherPage` — проверка `CurrentUserInRole("CTI.Инженеры")` перед открытием модалки.

**Ответ через внешнюю почту (Q5.5) — вынесен в отдельную подзадачу.** См. Q5.5 ниже.

### Q5.5 — Контроль списания при ответе через внешнюю почту

**Контекст:** В рамках **волны 2 нотификаций** (параллельный проект) прорабатывается сценарий: инженер отвечает клиенту не из интерфейса BPMSoft, а из почтового клиента. Это сообщение попадает в BPMSoft через интеграцию и публикуется в ленту. Popup при этом не всплывает (это автосообщение), но списание всё равно нужно.

**Решение:** фиксируем как **отдельную подзадачу**, логически относящуюся к волне 3 (или как задача 2.5 волны 2).

**Требования (зафиксировать и прорабатывать отдельно):**
- Детектор: входящее письмо от клиента → затем исходящее от инженера (не через UI) → триггер задачи на списание.
- Задача инженеру на списание (BPMSoft Task или Reminding).
- Механизм принудительного выполнения (эскалация если не списано, контроль SLA).
- Система мотивации / контроля — отдельная проработка с заказчиком.

**В MVP labor-records НЕ входит.**

### Q6 — Массовое согласование в v2 (НЕ в MVP)

**Решение:** в MVP руководитель согласует **по одной записи**:
- Через центр уведомлений (колокольчик) — «Утвердить» / «Отклонить» в выпадашке рядом с задачей визирования.
- Через страницу записи — кнопки «Утвердить» / «Отклонить» на панели действий.
- Через вкладку «Визы» на странице записи.

В v2 добавим кастомное действие раздела «Утвердить выбранные» / «Отклонить выбранные» (`getSectionActions` + `BatchQuery` на визах).

**Why:** упрощает MVP, снимает риск с сложной механики «массовое обновление статуса виз может не триггерить DCM» (R4 в рисках). Если после релиза окажется, что руководители хотят пачкой — делаем в v2 по реальным требованиям.

**Удаление из архитектуры:** раздел 3.6 документа `ARCHITECTURE_v1.md` — пункт «Действия раздела → Утвердить/Отклонить выбранные» удаляется из MVP.

### Q7 — Старое поле `UsrOverTimeApproval` (Boolean)

**Решение:** поле **удаляется** с миграцией в визы.

**Обоснование:** поле делалось под ручную реализацию согласования. С переходом на визы поле не нужно. Но **информация о статусе визирования должна храниться и быть выгружаемой** для отчётов.

**План:**
1. В MVP: **не удалять** поле сразу (чтобы не сломать возможные отчёты/интеграции). Но **не использовать** его — вся логика согласования идёт через `UsrLaborRecordsVisa`.
2. В выгрузке реестра и любых отчётах — подтягивать статус визы через JOIN с `UsrLaborRecordsVisa` (например, виртуальная колонка `UsrApprovalStatusFromVisa` = последняя виза со статусом).
3. **v2:** аудит использования старого поля (кто-то ссылается? интеграции?). Если чисто — удалить.
4. Заложить в БД **всё нужное для отчётности через визы** (дата утверждения, автор, комментарий) — всё это есть в `UsrLaborRecordsVisa` из коробки.

**Why:** баланс между «не сломать текущее» и «не тянуть deprecated-поле вечно».

### Q8 — `UsrWorkDate` — тип DateTime

**Решение:** (b) **DateTime** (дата + время).

**Обоснование:** для определения переработки время критично. Пример: пятница 20:00 — переработка; пятница 14:00 — обычное рабочее время. Если хранить только дату — теряем точность, `IsWorkTime()` даёт неправильный результат.

**UI:**
- В popup списания — поля отдельно: «Дата» + «Время» (либо один DateTime-picker).
- По умолчанию: сегодня + текущее время.
- Если списываем задним числом — инженер явно выбирает дату и время.

**Связь с `UsrSourceMessage`:** если запись создана через popup при отправке сообщения, `UsrWorkDate` по умолчанию = `Now` при подтверждении (можно корректировать).

---

### Q9 — Комментарии верификации/отклонения — только в визах

**Решение:** (a) комментарии живут **только в `UsrLaborRecordsVisa.Comment`**. На `UsrLaborRecords` своего поля нет.

**Выгрузка отчётов:** через JOIN с визами — стандартный паттерн BPMSoft. Виртуальные колонки в реестре.

**Why:** каноническое поведение платформы, аудит в одном месте, нет риска рассинхронизации.

---

### Q10 — N дней назад = календарные

**Решение:** (a) **календарные дни**.

**Обоснование:** проще в реализации, предсказуемо пользователю. Если `N=3`, после длинных майских можно будет списать за последний рабочий день до праздников, когда закончатся праздники.

**Системная настройка:** `UsrLaborRecordsMaxBackdateDays` (Integer, по умолчанию 3).
