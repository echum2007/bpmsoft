## 0. Резюме

- **Базовый принцип проекта:** *встроенные механизмы платформы прежде кастома.* Порядок рассмотрения при проектировании: no-code → платформенные API → кастомный клиент → кастомный сервер. Чем меньше меняем систему, тем стабильнее.
- Основной подход: **каноническое визирование BPMSoft + дизайнер кейсов (DCM)** для цикла согласования переработок. Решение снимает ~40% кастомной разработки (UI кнопок, центр уведомлений, права, делегирование, история согласований), даёт пользователям знакомый платформенный UX и упрощает поддержку.
- **DCM-кейс запускается только для записей с переработкой** (`UsrOverTime == true`, см. Q4). Записи без переработки идут в выгрузку напрямую, без стадии и виз. Условие запуска кейса задаётся штатным механизмом мастера разделов («По какой колонке настраивать условие запуска кейса?») — BPMN-обёртка не требуется.
- **Popup списания при сообщении в ленту обязателен на ЛЮБОЕ ручное сообщение инженера** (см. Q5, уточнение от 2026-04-25) — мотивирует не забывать списывать. Признак «Переработка» — опциональная галка в popup; именно она определяет, запустится ли DCM-цикл.
- Реализация popup — замещение `SocialMessagePublisherPage` и `ModalBox.show()` + отдельный модуль карточки. Паттерн подтверждён документацией.
- **Определение «формально рабочего времени»** (для подсветки переработок) — платформенное API `ICalendarRepository` + `ICalendar.IsWorkTime(DateTime)` (namespace `BPMSoft.Configuration.Calendars`). Каскад календарей: `ServicePact → ServiceItem → BaseCalendar`.
- **Массовое утверждение/отклонение виз — НЕ в MVP** (см. Q6). В MVP — утверждение по одной записи штатными средствами (центр уведомлений, панель действий страницы, вкладка «Визы»). Кнопки доступны из коробки без клиентского кода.
- В ТЗ v1.5 обнаружено одно серьёзное внутреннее расхождение (раздел 9 «Схема процесса» рассинхронизирован с разделами 3–5) — будет устранено в ТЗ v1.6.

### 3.2. Конвейер «списание при отправке сообщения в ленту»

**Условие срабатывания popup (Q5, уточнено 2026-04-25):**

- Pre-condition: `CurrentUserInRole("CTI.Инженеры")` AND ручное действие пользователя в UI ленты.
- Автосообщения BPMN/интеграций — **мимо** popup (popup не показывается).
- **Popup обязателен на ЛЮБОЕ ручное сообщение инженера** — независимо от того, переработка это или обычное время. Цель — мотивировать инженеров не забывать списывать время.
- Признак «Переработка» в popup — **опциональная галка**, которую инженер ставит сам, если работа сверхурочная. Постановка галки определяет, запустится ли DCM-цикл (см. 3.5). Если галка не стоит — запись создаётся, идёт в выгрузку напрямую, без виз.
- **Q5.5** (контроль списания при ответе через внешнюю почту) — отдельная подзадача волны 2.5/3, в MVP labor-records НЕ входит. См. `DECISIONS_v1.md`.

**Схема модалки (`UsrLaborRecordsModalPage`):**
- Модуль AMD, зависимости: `["ModalBox", "ConfigurationConstants", "css!UsrLaborRecordsModalPageCss"]`.
- Поля: Дата работ (DateTime), Часы, Минуты, Переработка (чекбокс), Комментарий (textarea).
- Кнопки: «Подтвердить», «Отмена».
- Валидация на стороне модалки:
  - Дата в диапазоне `[now - N дней; now]` (N читается из SysSettings).
  - Длительность ≥ 1 минута.
- При нажатии «Подтвердить» — возвращает данные в `CasePage` через `sandbox.publish(...)`.

**Важные нюансы:**
- Модалка **не** пишет в БД сама — запись создаёт CasePage (чтобы был единый транзакционный контекст).
- При отмене модалки сообщение тоже **не публикуется** (прерываем `onPublishButtonClick`).
- `UsrSourceMessage` заполняется **после** публикации сообщения, потому что ID `Activity` известен только после сохранения. Это двухшаговая операция, но атомарность не критична (записи связаны через `UsrCase`, максимум — потеряем детализацию «к какому именно сообщению относится списание»).

### 3.4. Определение «формально рабочего времени» (для подсветки)

**EventListener `UsrLaborRecordsEventListener`** на объекте `UsrLaborRecords`:
- Событие: `OnSaving`
- Логика (две независимые проверки в порядке):

**Шаг 1 — защита от договоров без СМ (Q1, уровень 2):**

```csharp
var servicePactId = entity.GetTypedColumnValue<Guid>("UsrServicePact");
if (servicePactId == Guid.Empty) {
    throw new Exception("Невозможно списать время: договор обращения не определён.");
}
// Подтянуть UsrResponsibleServiceManager из ServicePact (через ESQ)
var responsibleSmId = ... ; // SELECT UsrResponsibleServiceManager FROM ServicePact WHERE Id = servicePactId
if (responsibleSmId == Guid.Empty) {
    throw new Exception("Невозможно списать время: у договора не назначен ответственный сервис-менеджер. Обратитесь к руководителю сервиса.");
}
```

Канонически (подтверждено NotebookLM): обычный `throw new Exception(...)` в `OnSaving` показывается пользователю **модальным окном с кнопкой «ОК»**. Клиент перехватывает `success: false` от сервера и выводит сообщение через `showInformationDialog`.

**Шаг 2 — расчёт `UsrIsFormallyWorkTime`** (для подсветки):

- Если `UsrOverTime == true`:
  - Определить календарь для договора (каскад из `TermCalculatorITILService.GetCalendarId()`):
    1. `ServiceInServicePact.CalendarId` — если есть.
    2. `ServicePact.CalendarId` — если есть.
    3. `base.GetCalendarId()` (системный календарь).
  - `var actions = new TermCalculatorActions(UserConnection, calendarId);`
  - `bool isWorkTime = actions.IsTimeInWorkingInterval(UsrWorkDate);`
  - `UsrIsFormallyWorkTime = isWorkTime` — `true` означает «переработка проставлена, но время формально рабочее → подозрительно, подсветить руководителю».
- Если `UsrOverTime == false` → `UsrIsFormallyWorkTime = false`.

**API подтверждён 2026-04-25** проверкой пакета `SLM/Schemas/TermCalculatorActions/TermCalculatorActions.cs` (строки 109–125, метод `IsTimeInWorkingInterval`). Класс публичный (`public class TermCalculatorActions`), namespace `BPMSoft.Configuration`. Реализация: проверяет `DayInCalendar.DayType_IsWeekend` + наличие `WorkingTimeInterval`, чьи `From <= time <= To`. **`ICalendarRepository`/`IsWorkTime` в платформе 1.9 НЕТ** — это API для другой версии BPMSoft/Creatio.

**Namespace:** `BPMSoft.Configuration.Calendars`
**Dependency injection:** `ClassFactory.Get<ICalendarRepository>(new ConstructorArgument("userConnection", UserConnection))`

**Требует перезапуск Kestrel** после публикации (это EventListener).

### 3.5. Цикл согласования — DCM + Визирование

**Принципиальное решение (Q4):** DCM-кейс запускается **только для записей с переработкой** (`UsrOverTime == true`). Записи без переработки — стадии нет, виз нет, идут в выгрузку напрямую как «готовые к выгрузке».

**Канал создания (popup vs ручная деталь) на запуск DCM не влияет** — единственный критерий запуска кейса — значение `UsrOverTime` на момент создания записи.

**Каноническая настройка условия запуска (подтверждено NotebookLM):**

1. В мастере раздела `UsrLaborRecords` → «Настройка вида» → «Настроить кейсы раздела»:
   - «По какой колонке строятся стадии кейса?» → `UsrLaborRecordsStage` (новая lookup-колонка состояний).
   - «По какой колонке настраивать условие запуска кейса?» → `UsrOverTime`.
2. В дизайнере кейса `UsrLaborRecordsCase`:
   - Условие запуска кейса: `UsrOverTime = true`.
3. **BPMN-обёртка для запуска кейса не требуется** — встроенный механизм мастера разделов делает это сам.
4. Документация рекомендует отключить любые BPMN-процессы перевода по стадиям для разделов с DCM.

#### Стадии DCM (для записей с `UsrOverTime == true`)

```
┌──────────────┐    ┌──────────────────────┐    ┌─────────────────────┐
│  Черновик    │───▶│  На верификации      │───▶│  На утверждении     │
│ (создание)   │    │  (виза СМ договора)  │    │  (виза руководителя)│
└──────────────┘    └──────────────────────┘    └─────────────────────┘
                            │                           │
                            │ виза отрицательная        │ виза отрицательная
                            ▼                           ▼
                    ┌──────────────────────┐    ┌─────────────────────┐
                    │  На доработке        │    │  Отклонено          │
                    │                      │    │  (конечная стадия)  │
                    └──────────────────────┘    └─────────────────────┘
                            │
                            │ инженер правит + «Отправить на согласование заново»
                            ▼
              [Новая виза СМ — цикл с Этапа 1, Q3]

На утверждении → виза положительная ───▶ Утверждено (доступно для выгрузки)
```

**Записи без переработки (`UsrOverTime == false`):**

```
[Создание] ──▶ [Готова к выгрузке]   (DCM не запускается, стадии нет, виз нет)
```

**Элементы DCM:**

- Стадии: `Черновик`, `На верификации`, `На утверждении`, `На доработке`, `Утверждено`, `Отклонено`. **Стадии «Завершено» нет** — для записей без переработки кейс просто не запускается.
- В стадии `На верификации` — элемент **«Визирование»**:
  - Раздел визирования: `UsrLaborRecords`
  - Идентификатор записи: `[#Основная запись.Id#]`
  - Кому отправить: `[#Основная запись.UsrServicePact.UsrResponsibleServiceManager#]`
  - Цель: «Верификация переработки по обращению №[#Основная запись.UsrCase.Number#]»
  - Можно делегировать: да
  - При результате «Положительная» → `На утверждении`
  - При результате «Отрицательная» → `На доработке`
- В стадии `На утверждении` — элемент **«Визирование»**:
  - Кому отправить: роль `CTI.Руководители согласования трудозатрат` (точное имя — на этапе реализации).
  - При «Положительная» → `Утверждено`
  - При «Отрицательная» → `На доработке` → возврат на Этап 1 (новая виза СМ, Q3).

**Визы:** объект `UsrLaborRecordsVisa` создаётся автоматически при включении галки «Доступно визирование в разделе» (необратимая операция, R5).

**Утверждение виз — штатными средствами (Q6, без кастома):** подтверждено NotebookLM — кнопки «Утвердить» / «Отклонить» / «Сменить визирующего» доступны из коробки в трёх местах:
1. Группа «Визирование» в Центре уведомлений (колокольчик).
2. Панель действий («Следующие шаги») страницы записи.
3. Вкладка «Визы» страницы записи.

**Права:** элемент «Изменить права доступа» в DCM — на каждой стадии даём права на чтение/утверждение соответствующей роли; при переходе на следующую стадию — восстанавливаем базовые права. Детали — на этапе реализации.

### 3.6. Раздел «Управление трудозатратами»

**Новый раздел BPMSoft** (мастер разделов → создать новый), объект — `UsrLaborRecords`.

**Реестр** — колонки (по ТЗ 3.4.3, адаптированы):
| Колонка | Источник |
|---|---|
| Дата работ | `UsrWorkDate` |
| Дата создания | `CreatedOn` |
| Сотрудник | `CreatedBy` |
| Обращение | `UsrCase.Number` |
| Договор | `UsrServicePact.Name` |
| Номер Лида | `UsrLeadNumber` |
| Длительность | виртуальная `UsrTotalMinutes` (или `Hours / Minutes` отдельно) |
| Переработка | `UsrOverTime` |
| **Стадия** | стадия DCM |
| Сервис-менеджер | `UsrServicePact.UsrResponsibleServiceManager` |
| Разрешены сверх. | `UsrServicePact.UsrOvertimeAllowed` |
| Комментарий | `UsrWorkComments` |
| **Формально рабочее время** | `UsrIsFormallyWorkTime` — с **подсветкой жёлтым** (правило цветового выделения) |

**Фильтры** (фиксированные):

- Период (`UsrWorkDate`).
- Сотрудник (`CreatedBy`).
- Обращение.
- Договор.
- **Тип записи:** «Все» / «Только переработки» / «Только без переработки» (Q4 — нужен для отделения записей с DCM от выгружаемых напрямую).
- Стадия (`UsrLaborRecordsStage`) — **отображается только в режимах «Все» и «Только переработки»** (для записей без переработки стадии нет).
- «Требуют моего внимания» — комплексный фильтр: `Stage IN (На верификации, На утверждении) AND UsrServicePact.UsrResponsibleServiceManager = currentContactId` (для сервис-менеджера), `Stage = На утверждении` (для руководителя).

**Действия раздела (MVP):** только стандартные.

- **«Экспорт в Excel»** — стандартное действие BPMSoft.

**Утверждение/отклонение в MVP — по одной записи штатными средствами** (Q6, подтверждено NotebookLM, без кастомного кода):

1. Группа «Визирование» в Центре уведомлений (колокольчик) — действия Утвердить/Отклонить/Сменить визирующего.
2. Панель действий («Следующие шаги») страницы записи — те же кнопки.
3. Вкладка «Визы» страницы записи — детали и действия.

**Массовое утверждение/отклонение — в v2** (см. раздел 10). Реализация: кастомное действие раздела через `getSectionActions` + `BatchQuery` на `UsrLaborRecordsVisa`. Из MVP убрано, чтобы снять риск R4 (массовое обновление `Status` визы напрямую может не триггерить DCM-переходы).

**Права на раздел:**
- Инженер — чтение своих записей (через объектные права).
- Сервис-менеджер — чтение записей по своим договорам (`UsrServicePact.UsrResponsibleServiceManager = currentContact`).
- Руководитель сервис-менеджеров — чтение всех записей.
- Руководитель (утверждающий) — чтение всех записей + право утверждать.
- Аналитик — чтение всех записей + экспорт.
- Администратор — полный доступ.

**Роли:**

- `CTI.Инженеры` (орг. роль) — уже есть, не трогаем.
- `CTI.Ответственные сервис-менеджеры` (функциональная) — NEW.
- `CTI.Руководители сервис-менеджеров` (функциональная) — NEW.
- `CTI.Руководители согласования трудозатрат` (функциональная) — NEW.
- `CTI.Аналитики трудозатрат` (функциональная) — **NEW, отдельная роль** (Q2). Права: чтение всех `UsrLaborRecords` и `UsrLaborRecordsVisa`, экспорт в Excel. **Не имеет** права утверждать визы (отдельный игрок процесса — контроль и выгрузка).

## 7. Тестовый сценарий (end-to-end) для приёмки

1. **Инженер — переработка в выходной.** На CasePage отправляет ответ клиенту в субботу в 11:00.
   - Появляется popup списания с датой сегодняшней.
   - Инженер ставит 1 час, признак «Переработка», комментарий.
   - Подтверждает → сообщение опубликовано, запись `UsrLaborRecords` создана, `UsrSourceMessage` заполнен.
   - DCM запускается (`UsrOverTime = true`), переводит в стадию «На верификации», создана виза для ответственного СМ.

2. **Сервис-менеджер — утверждение.** Получает уведомление в колокольчике.
   - Открывает центр уведомлений → группа «Визирование» → «Утвердить» на записи.
   - DCM переводит в «На утверждении», создана виза для руководителя.

3. **Руководитель — утверждение по одной (MVP).** В разделе «Управление трудозатратами»:
   - Фильтр «Требуют моего внимания» → видит запись.
   - Открывает страницу записи → нажимает «Утвердить» на панели действий (или то же действие в центре уведомлений / на вкладке «Визы»).
   - Инженер получает email «…согласована».
   - DCM → «Утверждено».
   - *v2-приложение:* выбирает несколько записей → действие «Утвердить выбранные» (массовое — НЕ в MVP).

4. **Аналитик** в разделе применяет фильтр «Период: прошлая неделя, тип записи = только переработки, стадия = Утверждено» → Экспорт в Excel → проверяет колонку длительности в минутах.

5. **Инженер — обычное сообщение без переработки (Q5 уточнение).** Пишет внутренний комментарий на CasePage в среду 14:00.
   - Появляется popup (popup обязателен на любое ручное сообщение инженера).
   - Инженер ставит 30 минут, **галку «Переработка» НЕ ставит**, комментарий.
   - Подтверждает → сообщение опубликовано, запись создана, `UsrOverTime = false`.
   - **DCM не запускается**, стадии нет, виз нет — запись готова к выгрузке.

6. **Инженер — ручное добавление через деталь (запасной канал).** Списание задним числом без сообщения в ленту:
   - Открывает CasePage → деталь «Трудозатраты» → «Добавить».
   - Заполняет: дата = вчера 18:00, длительность 2 часа, признак «Переработка» доступен и отмечен.
   - Сохраняет → запись создана, `UsrSourceMessage` пустой, DCM запускается (галка «Переработка» стоит).

7. **Негативный сценарий — переработка в формально рабочее время.** Инженер ставит переработку в понедельник 14:00.
   - `OnSaving` ставит `UsrIsFormallyWorkTime = true`.
   - В реестре руководителя запись подсвечена жёлтым.
   - Руководитель отклоняет с комментарием «Не учитывается как сверхурочная работа».
   - Инженер получает email с причиной.

8. **Возврат на доработку (Q3).** Руководитель отклоняет визу с пометкой «вернуть на доработку».
   - Инженер правит запись → нажимает «Отправить на согласование заново» → стадия «На верификации» → **новая виза СМ** (полный цикл с Этапа 1).

9. **Защита от договоров без СМ (Q1).** Инженер пытается списать время на обращение, у договора которого `UsrResponsibleServiceManager` пустой.
   - В popup-lookup «Обращение» соответствующие обращения не отображаются (фильтр) — основной путь блокируется на UI.
   - Если запись всё-таки создаётся (например, через деталь или импорт) → `OnSaving` бросает `Exception` → пользователь видит модальное окно: «Невозможно списать время: у договора не назначен ответственный сервис-менеджер. Обратитесь к руководителю сервиса.»

10. **Попытка списать 4 дня назад.** UI отклоняет, системная настройка `UsrLaborRecordsMaxBackdateDays = 3` (календарных дня, Q10).

## 10. Что не входит в MVP (для явности)

**Отложено в v2 (с зафиксированным решением):**

- **Массовое утверждение/отклонение виз** (Q6) — кастомное действие раздела `getSectionActions` + `BatchQuery` на `UsrLaborRecordsVisa`. В MVP — утверждение по одной штатными средствами.
- **Удаление поля `UsrOverTimeApproval`** (Q7) — после аудита использования в отчётах/интеграциях.

**Отложено в волну 2.5 / 3 нотификаций:**

- **Контроль списания при ответе через внешнюю почту** (Q5.5) — детектор «входящее письмо клиента → исходящее инженера не через UI» + задача инженеру на списание + механизм эскалации. Прорабатывается отдельно с заказчиком.

**Не входит в проект labor-records вовсе:**

- Кастомная форма выгрузки (используем стандартный экспорт раздела).
- Дашборд/аналитика по трудозатратам.
- Интеграция с внешними финансовыми системами через API.
- Автосогласование по правилам.
- Миграция записей из EV-Time (отдельной задачей, вне проекта).
- Мобильное приложение (используется веб-интерфейс BPMSoft).
