## 0. Резюме

- Основной подход выбран: **каноническое визирование BPMSoft + дизайнер кейсов (DCM)** для цикла согласования переработок. Решение снимает ~40% кастомной разработки (UI кнопок, центр уведомлений, права, делегирование, история согласований), даёт пользователям знакомый платформенный UX и упрощает поддержку.
- Ключевой клиентский компонент — **всплывающая карточка списания при отправке сообщения в ленту обращения** — реализуется через замещение `SocialMessagePublisherPage` и `ModalBox.show()` + отдельный модуль карточки. Паттерн подтверждён документацией.
- **Определение переработки** основано на платформенном API `ICalendarRepository` + `ICalendar.IsWorkTime(DateTime)` (namespace `BPMSoft.Configuration.Calendars`). Используется тот же каскад календарей, что и SLA: `ServiceInServicePact → ServicePact → ServiceItem → BaseCalendar`.
- **Массовое утверждение/отклонение виз** — не поддерживается платформой из коробки. Реализуется кастомным действием раздела «Управление трудозатратами» через `getSectionActions` + `BatchQuery` на объекте виз.
- В ТЗ v1.5 обнаружено **одно серьёзное внутреннее расхождение** (раздел 9 «Схема процесса» рассинхронизирован с разделом 3–5) и **несколько уточняющих вопросов**, которые нужно закрыть с заказчиком до начала разработки (см. раздел 8).

### 3.2. Конвейер «списание при отправке сообщения в ленту»

```
┌──────────────────────────────────────────────────────────────┐
│  CasePage → SocialMessagePublisherPage (замещение в CTI)     │
│                                                              │
│  onPublishButtonClick (переопределён):                       │
│    ├─ если this.get("LaborRecordsConfirmed") === true:       │
│    │     сброс флага → callParent() → сохраняет сообщение    │
│    │                                                         │
│    └─ иначе:                                                 │
│          sandbox.loadModule("UsrLaborRecordsModalPage") в    │
│          ModalBox.show({width: 560, height: 360}) —          │
│          с префиллом: Case, Date=now, Duration, OverTime     │
│                                                              │
│  После подтверждения в модалке:                              │
│    ├─ BatchQuery:                                            │
│    │   1) InsertQuery("UsrLaborRecords") — создаёт запись    │
│    │      + UsrSourceMessage = <будет заполнен на 2-м шаге>  │
│    ├─ this.set("LaborRecordsConfirmed", true)                │
│    ├─ this.set("PendingLaborRecordId", <id>)                 │
│    └─ this.onPublishButtonClick() — повторный вызов          │
│          → теперь уходит в callParent() и публикует сообщение│
│                                                              │
│  После успешной публикации (onPublishSuccess):               │
│    UpdateQuery("UsrLaborRecords") — проставить UsrSourceMessage│
│    по PendingLaborRecordId (ссылка на созданный Activity)    │
└──────────────────────────────────────────────────────────────┘
```

**Схема модалки (`UsrLaborRecordsModalPage`):**
- Модуль AMD, зависимости: `["ModalBox", "ConfigurationConstants", "css!UsrLaborRecordsModalPageCss"]`.
- Поля: Дата работ (DateTime), Часы, Минуты, Переработка (чекбокс), Комментарий (textarea).
- Кнопки: «Подтвердить», «Отмена».
- Валидация на стороне модалки:
  - Дата в диапазоне `[now - N дней; now]` (N читается из SysSettings).
  - Длительность ≥ 1 минута.
- При нажатии «Подтвердить» — возвращает данные в `CasePage` через `sandbox.publish(...)`.

**Важные нюансы:**
- Модалка **не** пишет в БД сама — запись создаёт CasePage (чтобы был единый транзакционный контекст).
- При отмене модалки сообщение тоже **не публикуется** (прерываем `onPublishButtonClick`).
- `UsrSourceMessage` заполняется **после** публикации сообщения, потому что ID `Activity` известен только после сохранения. Это двухшаговая операция, но атомарность не критична (записи связаны через `UsrCase`, максимум — потеряем детализацию «к какому именно сообщению относится списание»).

### 3.4. Определение «формально рабочего времени» (для подсветки)

**EventListener `UsrLaborRecordsEventListener`** на объекте `UsrLaborRecords`:
- Событие: `OnSaving`
- Логика:
  - Если `UsrOverTime == true`:
    - Определить календарь для договора:
      1. `UsrServicePact.UsrResponsibleServiceManager.Calendar` — если есть (у Contact есть Calendar? — проверить)
      2. `UsrServicePact.Calendar`
      3. `SysSettings["UsrLaborRecordsCalendarId"]` (по умолчанию `BaseCalendar`)
    - `ICalendarRepository.GetById(calendarId).IsWorkTime(UsrWorkDate)` → bool
    - `UsrIsFormallyWorkTime = IsWorkTime(workDate)` (true означает «переработка проставлена, но время формально рабочее — подозрительно»)
  - Если `UsrOverTime == false` → `UsrIsFormallyWorkTime = false`

**Namespace:** `BPMSoft.Configuration.Calendars`
**Dependency injection:** `ClassFactory.Get<ICalendarRepository>(new ConstructorArgument("userConnection", UserConnection))`

**Требует перезапуск Kestrel** после публикации (это EventListener).

**Элементы DCM:**

- Переход `Черновик → На верификации`: условие `UsrOverTime == true` (автоматически при создании записи с переработкой).
- Переход `Черновик → Завершено`: условие `UsrOverTime == false`.
- В стадии `На верификации` — элемент **«Визирование»**:
  - Раздел визирования: `UsrLaborRecords`
  - Идентификатор записи: `[#Основная запись.Id#]`
  - Кому отправить: `[#Основная запись.UsrServicePact.UsrResponsibleServiceManager#]`
  - Цель: «Верификация переработки по обращению №[#Основная запись.UsrCase.Number#]»
  - Можно делегировать: да
  - При результате «Положительная» → `На утверждении`
  - При результате «Отрицательная» → `На доработке`
- В стадии `На утверждении` — элемент **«Визирование»**:
  - Кому отправить: роль «Руководитель сервиса» (согласовать точное имя с заказчиком)

| Сотрудник | `CreatedBy` |
| Обращение | `UsrCase.Number` |
| Договор | `UsrServicePact.Name` |
| Номер Лида | `UsrLeadNumber` |
| Длительность | виртуальная `UsrTotalMinutes` (или `Hours / Minutes` отдельно) |
| Переработка | `UsrOverTime` |
| **Стадия** | стадия DCM |
| Сервис-менеджер | `UsrServicePact.UsrResponsibleServiceManager` |
| Разрешены сверх. | `UsrServicePact.UsrOvertimeAllowed` |
| Комментарий | `UsrWorkComments` |
| **Формально рабочее время** | `UsrIsFormallyWorkTime` — с **подсветкой жёлтым** (правило цветового выделения) |

**Фильтры** (фиксированные):
- Период (`UsrWorkDate`)
- Сотрудник (`CreatedBy`)
- Обращение
- Договор
- Только переработки (`UsrOverTime = true`)
- Стадия (`UsrLaborRecordsStage`)
- «Требуют моего внимания» — комплексный фильтр: `Stage in (На верификации | На утверждении) AND UsrServicePact.UsrResponsibleServiceManager = currentContactId` (для сервис-менеджера), `Stage = На утверждении` (для руководителя)

**Действия раздела** (`getSectionActions`):
- «Утвердить выбранные» — кастомная кнопка, `BatchQuery` над объектом `UsrLaborRecordsVisa`:
  - Фильтр: `UsrLaborRecords IN selectedRows AND Status = Ожидает AND Visor = currentContactId`
  - Установить `Status = Положительная`, `SolutionDate = now`, `Comment = ''` (если разрешено без комментария)
  - После — перечитать реестр; DCM сам обработает переход стадии
- «Отклонить выбранные» — открыть модалку ввода причины (одна для всех), затем аналогичный `BatchQuery` с `Status = Отрицательная`
- «Экспорт в Excel» — стандартное действие BPMSoft

**Права на раздел:**
- Инженер — чтение своих записей (через объектные права).
- Сервис-менеджер — чтение записей по своим договорам (`UsrServicePact.UsrResponsibleServiceManager = currentContact`).
- Руководитель сервис-менеджеров — чтение всех записей.
- Руководитель (утверждающий) — чтение всех записей + право утверждать.
- Аналитик — чтение всех записей + экспорт.
- Администратор — полный доступ.

**Роли:**
- `CTI.Инженеры` (орг. роль) — уже есть, не трогаем.
- `CTI.Ответственные сервис-менеджеры` (функциональная) — NEW.
- `CTI.Руководители сервис-менеджеров` (функциональная) — NEW.

## 7. Тестовый сценарий (end-to-end) для приёмки

1. **Инженер** на CasePage отправляет ответ клиенту в субботу в 11:00.
   - Появляется popup списания с датой сегодняшней.
   - Инженер ставит 1 час, признак «Переработка», комментарий.
   - Подтверждает → сообщение опубликовано, запись `UsrLaborRecords` создана, `UsrSourceMessage` заполнен.
   - DCM переводит запись в стадию «На верификации», создана виза для ответственного СМ.
2. **Сервис-менеджер** получает уведомление в колокольчике.
   - Открывает центр уведомлений → «Утвердить» на записи.
   - DCM переводит в «На утверждении», создана виза для руководителя.
3. **Руководитель** в разделе «Управление трудозатратами»:
   - Фильтр «Требуют моего внимания» → видит запись.
   - Выбирает несколько записей → «Утвердить выбранные».
   - Инженер получает email «…согласована».
   - DCM → «Утверждено».
4. **Аналитик** в разделе применяет фильтр «Период: прошлая неделя, стадия = Утверждено» → Экспорт в Excel → проверяет колонку длительности в минутах.
5. **Инженер** вручную через деталь на CasePage создаёт запись без переработки (ручное добавление):
   - Поле «Переработка» скрыто.
   - DCM сразу переводит в «Завершено» (без согласования).
6. **Негативный сценарий:** инженер ставит переработку в понедельник 14:00 (формально рабочее).
   - `OnSaving` ставит `UsrIsFormallyWorkTime = true`.
   - В реестре руководителя запись подсвечена жёлтым.
   - Руководитель отклоняет с комментарием «Не учитывается как сверхурочная работа».
   - Инженер получает email с причиной.
7. **Возврат на доработку:** руководитель нажимает «Вернуть на доработку» (отрицательная виза + настройка в DCM).
   - Инженер правит запись → стадия снова «На верификации» → новая виза СМ.
8. **Попытка списать 3 дня + 1 минуту назад:** UI отклоняет, системная настройка = 3 дня.
