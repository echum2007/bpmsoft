# ТЗ: Подсистема регистрации трудозатрат — v1

**Дата:** 2026-04-07  
**Статус:** Черновик, на обсуждении  
**Пакет:** CTI

---

## 1. Назначение

Подсистема регистрации трудозатрат предназначена для фиксации времени, затраченного сотрудниками сервисного департамента на решение обращений и другие задачи. Данные трудозатрат хранятся в BPMSoft и выгружаются во внешние финансовые системы предприятия.

Подсистема заменяет внешнюю систему EV-Time — все функции учёта трудозатрат переносятся в BPMSoft.

---

## 2. Текущее состояние

Реализовано (на продуктиве):

| Компонент | Описание |
|-----------|----------|
| Объект `UsrLaborRecords` | Записи трудозатрат с полями: Обращение, Договор, Номер Лида, Часы, Минуты, Переработка, Согласование переработки, Комментарий |
| Деталь на CasePage | Вкладка «Списание трудозатрат» в обращении |
| Деталь на ServicePactPage | История трудозатрат в сервисном договоре |
| Бизнес-правило | Автозаполнение Договора и Номера Лида из обращения |
| Поле `UsrLeadNumber` в ServicePact | Номер Лида в сервисном договоре |

## 3. Доработки

### 3.1. Расширение модели данных `UsrLaborRecords`

#### 3.1.1. Дата списания

- **Поле:** `UsrWorkDate` (тип: Date)
- **Назначение:** Дата фактического выполнения работ. Может отличаться от даты создания записи (`CreatedOn`).
- **Валидация:** Дата списания не может быть раньше даты создания записи. Списание задним числом запрещено.
- **Значение по умолчанию:** Текущая дата.

#### 3.1.2. Тип работ

- **Поле:** `UsrWorkType` (тип: Lookup)
- **Справочник:** `UsrWorkType` (новый)
- **Значения:** «Плановые работы», «Аварийные работы»
- **Назначение:** Характер выполненных работ. В рамках одного обращения могут быть как плановые, так и аварийные работы — тип определяется на уровне каждой записи трудозатрат.
- **Обязательность:** Да.

#### 3.1.3. Автоопределение переработки

- **Поле:** `UsrOverTime` (существует, тип: Boolean)
- **Изменение поведения:** Система автоматически устанавливает признак переработки при создании/сохранении записи, если дата списания (`UsrWorkDate`) выпадает на нерабочее время по рабочему календарю системы BPMSoft.
- **Ручное изменение:** Инженер не управляет этим полем вручную — значение определяется системой.

#### 3.1.4. Комментарий при отклонении

- **Поле:** `UsrApprovalComment` (тип: Text)
- **Назначение:** Комментарий согласующего при отклонении переработки.
- **Заполняется:** Руководителем/администратором при отклонении.

#### 3.1.5. Статус согласования

- **Поле:** `UsrApprovalStatus` (тип: Lookup). Заменяет текущее булево поле `UsrOverTimeApproval`.
- **Справочник:** `UsrApprovalStatus` (новый)
- **Значения:** «Не требует согласования», «Ожидает согласования», «Согласовано», «Отклонено»
- **Логика:**
  - Если `UsrOverTime = false` → статус «Не требует согласования»
  - Если `UsrOverTime = true` → статус «Ожидает согласования» при создании
  - Руководитель меняет на «Согласовано» или «Отклонено»

### 3.2. Расширение модели данных `ServicePact`

#### 3.2.1. Разрешение сверхурочных

- **Поле:** `UsrOvertimeAllowed` (тип: Boolean)
- **Назначение:** Признак того, что по данному договору допускаются сверхурочные трудозатраты.
- **Значение по умолчанию:** Нет (false).
- **Использование:** Информационное поле для принятия решения при согласовании переработок. Не блокирует создание записей с признаком переработки.

---

### 3.3. Валидации при списании трудозатрат

#### 3.3.1. Ограничение часов и минут

- **Часы:** 0–23
- **Минуты:** 0–59
- **Реализация:** Валидация на клиенте (JavaScript) при сохранении записи. При вводе значения за пределами диапазона — не давать сохранить, показать сообщение об ошибке.

#### 3.3.2. Дата списания не раньше даты создания

- Валидация при сохранении: `UsrWorkDate >= дата(CreatedOn)`.
- Сообщение при нарушении: «Дата списания не может быть раньше даты создания записи».

#### 3.3.3. Обязательное списание при добавлении комментария

- **Контекст:** При добавлении сообщения в ленту обращения (ответ заказчику или внутреннее сообщение).
- **Поведение:** Система не позволяет сохранить комментарий к обращению, если по этому обращению не существует хотя бы одной записи трудозатрат, созданной текущим пользователем в текущий день.
- **Сообщение:** «Необходимо списать трудозатраты перед добавлением комментария».
- **Открытый вопрос:** Уточнить — проверяется наличие списания за сегодня? Или привязка к конкретному комментарию (одно списание = один комментарий)?

### 3.4. Рабочее место «Управление трудозатратами»

#### 3.4.1. Общее описание

Отдельный раздел BPMSoft (пункт в боковом меню). Предоставляет полный реестр всех записей трудозатрат с возможностью фильтрации, согласования переработок и выгрузки данных.

#### 3.4.2. Пользователи

- **Руководитель** — просмотр всех записей, согласование/отклонение переработок, выгрузка
- **Администратор** — полный доступ

Доступ ограничен ролью (создать организационную роль или функциональную роль `UsrLaborRecordsAdmin`).

Инженеры работают с трудозатратами только через деталь в обращении — доступа к разделу «Управление трудозатратами» у них нет.

#### 3.4.3. Реестр

Колонки реестра:

| Колонка | Источник |
|---------|----------|
| Дата списания | `UsrWorkDate` |
| Дата создания | `CreatedOn` |
| Сотрудник | `CreatedBy` |
| Обращение | `UsrCase` (номер) |
| Договор | `UsrServicePact` (название) |
| Номер Лида | `UsrLeadNumber` |
| Тип работ | `UsrWorkType` |
| Часы | `UsrWorkDurationHours` |
| Минуты | `UsrWorkDurationMinutes` |
| Переработка | `UsrOverTime` |
| Статус согласования | `UsrApprovalStatus` |
| Разрешены сверхурочные (в договоре) | `UsrCase.ServicePact.UsrOvertimeAllowed` |
| Комментарий | `UsrWorkComments` |
| Комментарий согласования | `UsrApprovalComment` |

#### 3.4.4. Фильтры

Гибкая фильтрация по всем параметрам, включая:
- Период (дата списания от — до)
- Сотрудник
- Договор / Номер Лида
- Обращение
- Тип работ (плановые / аварийные)
- Только переработки
- Статус согласования (ожидает / согласовано / отклонено)

Реализация: стандартные фильтры раздела BPMSoft + быстрые фильтры.

#### 3.4.5. Согласование переработок

**Действия в реестре:**

- Кнопка/меню с выбором действия: **«Согласовать»** / **«Отклонить»**
- При отклонении — диалог ввода комментария (обязательное поле)
- Массовое согласование: выбрать несколько записей → применить действие ко всем

**Уведомление инженера:**

- При согласовании — email инженеру: «Ваша переработка по обращению [номер] от [дата] согласована»
- При отклонении — email инженеру: «Ваша переработка по обращению [номер] от [дата] отклонена. Причина: [комментарий]»

**Права доступа:**

- Согласование доступно только пользователям с ролью `UsrLaborRecordsAdmin` (или аналогичной)
- Инженеры не могут изменять статус согласования и комментарий согласования

### 3.5. Выгрузка данных

#### 3.5.1. Механизм

Использовать встроенную функциональность экспорта реестра BPMSoft (Excel). Данные выгружаются с учётом текущих фильтров.

#### 3.5.2. Диалог выгрузки

Дополнительный диалог выбора параметров выгрузки:
- Период (от — до)
- Другие фильтры (по необходимости)

**Открытый вопрос:** Достаточно ли стандартного экспорта реестра с предварительной настройкой фильтров? Или нужен кастомный диалог выгрузки с собственной формой?

## 5. Границы версии v1

**Входит:**
- Расширение модели данных (новые поля и справочники)
- Валидации при списании (часы/минуты, дата, обязательность)
- Автоопределение переработки по календарю
- Рабочее место с реестром, фильтрами, согласованием
- Email-уведомления о результате согласования
- Выгрузка через стандартный экспорт реестра

**Не входит (возможно в будущих версиях):**
- Процесс визирования с несколькими участниками
- Автоматическое согласование по условиям
- Интеграция с внешними финансовыми системами (API выгрузки)
- Дашборд/аналитика по трудозатратам
