### 🟡 Дополнительная находка 4.3 — Что если UsrOverTime изменили после создания?

**Проблема:** если запись создана с `UsrOverTime = false` (DCM не запустился), а потом инженер открывает её и ставит галку — DCM запустится? Условие запуска проверяется при создании или при каждом изменении?

**Документация BPMSoft (DCM):** условие запуска проверяется при сохранении записи. Если до изменения UsrOverTime=false и запись была вне кейса, после изменения на true — DCM может запустить кейс. Но если кейс уже запускался однажды и завершился — повторного запуска не будет.

**Сценарий риска:** инженер по ошибке списал без переработки → понял ошибку → правит запись с `UsrOverTime=true`. Запустится ли DCM?

**Решение:**
- **A.** Принять, что записи нельзя править после создания (бизнес-правило: запись в стадии "Готова к выгрузке" read-only). Просто, но негибко.
- **B.** Разрешить правку, проверить поведение DCM в этом сценарии при тестировании T-18.

**Правка:** добавить в T-18 (E2E) сценарий 11:
> 11. Изменение UsrOverTime после создания. Создать запись с `UsrOverTime=false` → проверить, что DCM не запустился. Открыть запись → поставить `UsrOverTime=true` → сохранить. Проверить, запустился ли DCM. По результату — либо принять поведение, либо добавить бизнес-правило.

- **Вариант C (рекомендуется):** проверить наличие синхронного механизма в платформе. Если есть — использовать.
- **Вариант B:** в `onPublishButtonClick` сначала ждать инициализацию роли (флаг `IsCurrentUserEngineerLoaded`), при необходимости — отложить открытие popup на ~100мс.

### 🟡 Дополнительная находка 4.8 — Серверная защита по дате задним числом

**Проблема:** в T-07 EventListener не проверяет ограничение по дате (Q10). Аналогично Находке 1.2 (длительность). Если запись создана в обход popup и BR (через SQL или импорт) с датой `Now - 30 дней` — сервер пропустит.

**Правка:** в T-07 добавить блок проверки даты:
```csharp
var workDate = entity.GetTypedColumnValue<DateTime>("UsrWorkDate");
var maxBackdateDays = (int)Core.Configuration.SysSettings.GetValue(uc, "UsrLaborRecordsMaxBackdateDays", 3);
var minDate = DateTime.Now.AddDays(-maxBackdateDays);
if (workDate < minDate) {
    throw new Exception($"Дата работ не может быть раньше чем {maxBackdateDays} дней назад.");
}
if (workDate > DateTime.Now) {
    throw new Exception("Дата работ не может быть в будущем.");
}
```

Расширяет Находку 1.2 — серверная валидация всех ограничений.
