## 1. Основные инструменты no-code

**Мастер разделов** — центральный инструмент для создания и настройки разделов без кода. Под капотом он генерирует JavaScript-схемы (клиентские модели) и конфигурационные объекты.

**Как открыть:**
- Новый раздел: **Дизайнер системы** → **«Мастер разделов»**
- Существующий раздел: открыть раздел → **«Настройка вида»** → **«Открыть мастер раздела»**

**Требование:** системная операция «Доступ к разделу «Конфигурация»

## 2. Создание и редактирование разделов

### 2.1 Создание нового раздела

При создании раздела доступны два пути:

#### Путь 1: Новый объект (рекомендуется для нового функционала)

| Поле | Описание | Пример |
|------|---------|--------|
| **Заголовок** | Название раздела в боковой панели | «Обращения» |
| **Код (на английском)** | Уникальный идентификатор объекта `[Префикс][ИмяОбъекта]`. **После первого сохранения не меняется!** | `CTICaseV2` |
| **Иконка** | PNG/SVG, прозрачный фон, 38×38 px | иконка обращения |
| **Рабочее место** | Раздел навигации, где появится раздел | «Управление сервисом» |
| **Индексировать для поиска** | Включить в полнотекстовый поиск | ✓ для важных справочников |
| **Доступно визирование** | Вкл./откл. функциональность согласования | ✓ для документов |

#### Путь 2: Существующий объект (для обёртки над справочником)

Используется для добавления удобного интерфейса к уже существующей таблице. Если объект базовый (системный) — теги и группы будут недоступны.

### 2.2 Что создаётся автоматически при создании раздела

| Объект | Назначение | Вид |
|--------|-----------|-----|
| `<Code>` | Основная таблица раздела | Справочник |
| `<Code>File` | Вложения (файлы и ссылки) | Справочник |
| `<Code>Folder` | Иерархия групп (папок) | Справочник |
| `<Code>InFolder` | Связи записей с группами (многие ко многим) | М:М связь |
| `<Code>Tag` | Справочник доступных тегов | Справочник |
| `<Code>InTag` | Теги конкретных записей (М:М) | М:М связь |
| `<Code>RequestsVisa` | Записи о визах (согласованиях) | Справочник |

**Все объекты видны в:** Дизайнер системы → Управление конфигурацией

**Важно:** После создания раздела нужно **перелогиниться** для появления раздела в навигации (кэш пересчитывается при входе).

## 3. Настройка страниц раздела

### 3.1 Структура страницы

```
Раздел
├── Страница 1 (например, основная карточка)
│   ├── Вкладка 1 (например, "Основные сведения")
│   │   ├── Блок (ProfileContainer или ColumnContainer)
│   │   │   ├── Поле 1
│   │   │   ├── Поле 2
│   │   │   └── ...
│   │   └── Блок 2
│   ├── Вкладка 2 (например, "История")
│   └── Вкладка 3 (например, "Связанные данные")
└── Страница 2 (например, альтернативная форма)
```

### 3.2 Вкладки

| Свойство | Описание |
|----------|---------|
| **Нельзя удалить все** | Минимум одна вкладка всегда остаётся |
| **Вкладка "Лента"** | При удалении восстанавливается только разработкой (через схему) |
| **Порядок** | Меняется перетаскиванием в мастере |
| **Видимость** | Можно управлять через бизнес-правила |

### 3.3 Блоки (контейнеры для полей)

#### ProfileContainer

- **Где используется:** основной блок в «шапке» страницы (левая часть обычно)
- **Ограничение:** НЕ поддерживает два поля в одну строку — всегда друг под другом
- **Решение:** если нужны два поля в строку → `ColumnContainer` на вкладке

#### ColumnContainer

- **Где используется:** многоколоночные блоки на вкладках
- **Преимущество:** поддерживает гибкую раскладку полей
- **Пример:** две колонки: "Сведения об исполнителе" и "Параметры SLA"

### 3.4 Типы полей

| Тип поля | Описание | Примеры |
|----------|---------|---------|
| **Текст** | Строка или многострочный текст | Название, Описание, Комментарий |
| **Целое число** | Целые числовые значения | Приоритет (1-5), Количество попыток |
| **Дробное число** | Числа с точкой | Рейтинг (4.5), Коэффициент |
| **Дата** | Выбор даты через календарь | Дата создания, Дата завершения |
| **Дата и время** | Дата + время | Время начала SLA, Время ответа |
| **Логическое** | Чекбокс (да/нет) | «Срочно?», «Закрыто?» |
| **Справочник** | Lookup — ссылка на запись другого объекта | Исполнитель (→ Contact), Услуга (→ Service) |
| **Изображение** | Загрузка одного изображения | Логотип, Аватар |
| **Файл** | Загрузка одного файла | Вложение |

**Примечание:** Для загрузки нескольких файлов используется встроенная деталь «Вложения» (`<Code>File`).

## 4. Бизнес-правила и их применение

### 4.1 Что такое бизнес-правило

**Бизнес-правило** — условная логика, управляющая поведением полей на странице в реальном времени, без написания кода.

**Где находится:** Мастер разделов → вкладка «Бизнес-логика» → «Бизнес-правила»

### 4.2 Структура бизнес-правила

```
ЕСЛИ условие выполнено
    ТО выполнить действие на элемент (поле, вкладка, блок)
```

**Условия:**
- Простое сравнение: `Поле = Значение`
- Логические операторы: AND, OR
- Операторы сравнения: =, <, >, <=, >=, <>
- Формулы: в условии можно использовать арифметические выражения

**Действия:**

| Действие | Описание | Пример |
|----------|---------|--------|
| **Показывать элемент** | Показать/скрыть поле, вкладку, блок | Если Type = "Инцидент" → показать вкладку "Анализ корневой причины" |
| **Делать поле редактируемым** | Включить/заблокировать редактирование | Если Status = "Закрыто" → заблокировать все поля |
| **Делать поле обязательным** | Сделать обязательным для заполнения | Если Priority = "Критический" → обязательно заполнить "Исполнитель" |
| **Добавлять фильтр значений** | Фильтровать список lookup-поля | Если выбран Контракт → фильтровать Услуги только по этому контракту |
| **Заполнять значение поля** | Автоматически подставить значение | Если Category = "Сеть" → StatusValue по умолчанию "Диагностика" |

### 4.3 Примеры бизнес-правил для Service Desk

#### Пример 1: Обязательный исполнитель при критическом приоритете

```
ЕСЛИ Priority = "Критический"
ТО Assignee (Исполнитель) → сделать обязательным
```

**Результат:** при попытке сохранить обращение с критическим приоритетом без исполнителя появится ошибка «Поле обязательно».

#### Пример 2: Видимость вкладки по типу обращения

```
ЕСЛИ Type = "Инцидент"
ТО вкладка "SLA" → показывать

ЕСЛИ Type = "Запрос" (Request)
ТО вкладка "SLA" → скрывать
```

#### Пример 3: Фильтрация услуг по контракту

```
ЕСЛИ ServiceContract заполнен
ТО Service (Услуга) → добавить фильтр значений по ServiceContract
```

**Результат:** в lookup-поле "Услуга" будут видны только услуги, входящие в выбранный контракт.

#### Пример 4: Автозаполнение статуса по категории

```
ЕСЛИ Category = "Сетевая проблема"
ТО Status → заполнить значение "Диагностика сети"
```

#### Пример 5: Блокировка полей в закрытом обращении

```
ЕСЛИ Status = "Закрыто" (Closed)
ТО [Description, Assignee, Priority, DueDate] → делать нередактируемыми
```

### 4.4 Формулы в бизнес-правилах

В условиях и значениях можно использовать формулы над полями. Это расширяет no-code логику.

**Примеры:**
```
// Условие: время до истечения SLA < 1 часа
SLADueTime - Now < 3600  (в секундах)

// Автозаполнение: указать дату завтра
DueDate = Today + 1

// Условие: обращение старше 7 дней и ещё открыто
Age > 7 AND Status != "Закрыто"
```

**Поддерживаемые функции:**
- Арифметика: `+`, `-`, `*`, `/`, `%`
- Сравнение: `=`, `<`, `>`, `<=`, `>=`, `<>`
- Логика: `AND`, `OR`, `NOT`
- Функции даты: `Now`, `Today`, дни от даты
- Функции строк: конкатенация (в некоторых случаях)

## 5. [НОВОЕ] Пропущенные компоненты: фильтры, сортировки, экспорты

### 5.1 Фильтры реестра

**Фильтр реестра** — инструмент для ограничения видимости записей в списке на основе условий (без кода).

**Где настраивается:** Мастер разделов → вкладка раздела → кнопка «Фильтры»

**Виды фильтров:**

| Вид | Описание | Пример |
|-----|---------|--------|
| **Быстрый фильтр** | Текстовое поле для быстрого поиска по полям | Поиск по названию обращения |
| **Расширенный фильтр** | Сохраняемые фильтры по условиям | «Мои открытые обращения», «Просроченные SLA» |
| **Фильтр по датам** | Специальный фильтр для дат | Обращения за последний месяц |
| **Фильтр по людям** | Lookup-фильтр | Обращения, назначенные на меня |

**Примеры для Service Desk:**

```
Фильтр 1: "Мои открытые обращения"
  Assignee = CurrentUser AND Status != "Закрыто"

Фильтр 2: "Критические задержки"
  Priority = "Критический" AND SLADueTime < Now

Фильтр 3: "Ожидание внешнего подрядчика"
  Status = "На рассмотрении" AND ExternalContractor is not empty
```

### 5.2 Сортировки

**Сортировка реестра** — упорядочивание записей по одному или нескольким полям.

**Где настраивается:** в самом реестре (через кнопку «Сортировка») или через мастер разделов → «Дополнительные параметры»

**Стандартные сортировки для обращений:**

```
Сортировка 1 (по умолчанию):
  1. Priority (убывание — критические сверху)
  2. SLADueTime (возрастание — спешные первыми)
  3. CreatedOn (убывание — новые первыми)

Сортировка 2 (по датам):
  1. CreatedOn (убывание)
  2. Status (по значению)
```

**Примечание:** пользователь может менять сортировку в реестре в любой момент, система запомнит его выбор.

### 5.3 Экспорты (выгрузка данных)

**Экспорт** — выгрузка данных реестра в файл (Excel, CSV, PDF и т.д.).

**Где находится:** реестр → кнопка «Экспорт» (в панели инструментов)

**Доступные форматы:**
- **Excel** (.xlsx) — с форматированием, графиками
- **CSV** — простой текстовый формат
- **PDF** — отчёт в форме таблицы

**Что экспортируется:**
- Видимые колонки реестра (пользователь может убрать/добавить колонки)
- Строки текущего фильтра (если фильтр активен, экспортируются только отфильтрованные записи)

**Например, Service Desk:**
```
Экспорт "Обращения за месяц"
  Видимые колонки: № обращения, Название, Исполнитель, Статус, Приоритет, SLA, Создано
  Текущий фильтр: CreatedOn >= 1.03.2026 AND CreatedOn <= 31.03.2026
  Формат: Excel
  Результат: файл Report_March2026.xlsx с 250 строками
```

### 5.4 Правила цветового выделения

**Правило выделения** — условное форматирование строк реестра на основе значений полей.

**Где настраивается:** Мастер разделов → вкладка «Бизнес-логика» → «Правила цветового выделения»

**Структура:**
```
ЕСЛИ условие
ТО применить стиль (цвет фона, текста, иконку)
```

**Примеры для Service Desk:**

```
Правило 1: "Критические обращения"
  ЕСЛИ Priority = "Критический"
  ТО красный фон (#FF6B6B)

Правило 2: "Новые обращения"
  ЕСЛИ Status = "Новое"
  ТО жёлтый фон (#FFD93D)

Правило 3: "Просроченные SLA"
  ЕСЛИ SLADueTime < Now AND Status != "Закрыто"
  ТО красная иконка ⚠️ + жёлтый фон

Правило 4: "Мои обращения"
  ЕСЛИ Assignee = CurrentUser
  ТО синий текст
```

**⚠️ Ограничение:** не поддерживаются условия с типом **Дата/Дата и время** напрямую. Для этого нужно использовать формулы.

## 6. [НОВОЕ] Детали и их настройка

### 6.1 Что такое деталь

**Деталь** — встроенный реестр связанных записей на странице основного объекта. Это один из мощнейших инструментов no-code кастомизации.

**Отличие от отдельного раздела:**
- Деталь видна в контексте родительской записи
- Данные деталь привязаны к записи (одна запись → много связанных)
- Редактирование может быть как в отдельной форме, так и прямо в таблице

### 6.2 Типы деталей

| Тип | Описание | Когда использовать |
|-----|---------|-------------------|
| **Обычная деталь** | Реестр с кнопками "Создать", "Удалить", "Редактировать" | Для большинства 1:М связей |
| **Редактируемый реестр** | Строки редактируются прямо в таблице (inline editing) | Для быстрого ввода табличных данных |
| **Деталь с детализацией** | Открытие записи детали в модальном окне | Для сложных форм с множеством полей |

### 6.3 Создание детали (no-code путь)

**Через мастер разделов:**

1. Открыть раздел в мастере
2. На странице → кнопка **«+Добавить деталь»**
3. Выбрать:
   - Объект детали (какая таблица будет источником данных)
   - Связь (какое поле в таблице детали указывает на родительскую запись)
4. Выбрать колонки для отображения
5. Настроить видимость, сортировку, фильтры
6. Сохранить

### 6.4 Примеры деталей для Service Desk

#### Деталь 1: История обращения (Timeline)

```
Раздел: Обращения
Деталь: Активности (Activities)
  Связь: Case → CaseActivity.Case
  Видимые колонки: Тип, Содержание, Создано, Автор
  Сортировка: По дате создания (новые сверху)
  Фильтр: Показывать только активности последних 30 дней
  Редактируемость: Нет (только просмотр)
```

#### Деталь 2: Файлы и вложения

```
Раздел: Обращения
Деталь: Вложения (встроенная, создаётся автоматически)
  Связь: Case → CaseFile
  Видимые колонки: Имя файла, Размер, Дата загрузки, Кто загрузил
  Редактируемость: Да (можно добавить/удалить файлы)
```

#### Деталь 3: Услуги в контракте

```
Раздел: Сервисные контракты
Деталь: Услуги контракта (редактируемый реестр)
  Связь: ServiceContract → ServiceInServiceContract
  Видимые колонки: Услуга, Уровень обслуживания, Часы работы, Цена
  Редактируемость: Да (можно менять данные прямо в таблице)
```

### 6.5 Настройка видимости и поведения детали

**Бизнес-правила могут управлять деталями:**

```
ЕСЛИ Status = "Закрыто"
ТО деталь "Активности" → скрывать

ЕСЛИ Type = "Инцидент"
ТО деталь "Связанные запросы" → показывать

ЕСЛИ Priority = "Низкий"
ТО деталь "История изменений" → делать нередактируемой
```

## 7. [НОВОЕ] Service Desk в контексте no-code

### 7.1 Структура раздела "Обращения" (Case)

Service Desk в BPMSoft построен на объекте `Case`. Типичная конфигурация no-code кастомизации:

#### Главная страница обращения

**Блок ProfileContainer (шапка):**
- Номер обращения (число, автоинкремент)
- Название (текст)
- Описание (многострочный текст)
- Контакт/Заявитель (lookup → Contact)
- Статус (lookup → CaseStatus)
- Приоритет (lookup → Priority)

**Вкладка "Исполнение":**
- Исполнитель (lookup → Employee)
- Дата/время ответа (SLA-aware)
- Дата/время завершения (SLA-aware)
- Комментарий исполнителя (многострочный)

**Вкладка "Сервис":**
- Сервисный контракт (lookup → ServiceContract)
- Услуга (lookup → Service, с фильтром по контракту)
- Категория (lookup → CaseCategory)
- Подкатегория (lookup, фильтруется по категории)

**Вкладка "История":**
- Деталь: Активности (timeline)
- Деталь: Вложения (файлы)
- Деталь: История изменений

**Вкладка "SLA":**
- SLA период (дата/время, когда заканчивается SLA)
- Статус SLA (выполнено/нарушено)
- Оставшееся время (формула на основе SLADueTime)
- График обслуживания (lookup → ServiceSchedule)

### 7.2 Бизнес-правила Service Desk (примеры реальной конфигурации)

#### Правило 1: Обязательный контракт и услуга

```
ЕСЛИ Status != "Закрыто" (обращение открыто)
ТО ServiceContract → обязательное поле
ТО Service → обязательное поле
```

#### Правило 2: Фильтрация услуг и категорий

```
ЕСЛИ ServiceContract выбран
ТО Service → фильтровать только услуги этого контракта
ТО CaseCategory → фильтровать только категории контракта
```

#### Правило 3: Видимость вкладок по типу

```

```

```

#### Правило 5: Блокировка полей в закрытом обращении

```

ТО [Assignee, Priority, Description] → нередактируемые
ТО деталь "Активности" → только просмотр

```

#### Правило 6: Автозаполнение дефолтных значений

```

ТО Status → заполнить "Новое"
ТО Priority → заполнить "Средний"

## 8. [НОВОЕ] Под капотом: от no-code к коду

### 8.1 Как работает мастер разделов

Когда вы создаёте раздел через мастер, система:

1. **Создаёт объекты** в конфигурации (Case, CaseFile, CaseFolder и т.д.)
2. **Генерирует клиентские схемы** (JavaScript) → хранятся в пакете
3. **Генерирует C# сервис** (если нужны веб-сервисы)
4. **Записывает метаданные** в системные таблицы

### 8.2 Связь бизнес-правил с JavaScript кодом

#### Что происходит в no-code:

```
Мастер разделов → вкладка "Бизнес-правила"
┌────────────────────────────────┐
│ ЕСЛИ Priority = "Критический"  │
│ ТО Assignee → обязательное     │
└────────────────────────────────┘
```

#### Что генерируется в JavaScript (блок 7–8):

```javascript
// В клиентской схеме CasePage.js
define("CasePage", [], function() {
    return {
        entitySchemaName: "Case",
        
        // Раздел "Бизнес-правила" компилируется в rules
        rules: [
            {
                ruleType: "RequiredField",  // действие: обязательное
                triggeredByColumnChange: ["Priority"],  // триггер
                businessRuleConfigs: [
                    {
                        "comparisonType": 4,  // = равно
                        "leftExpression": {
                            "columnPath": "Priority",
                            "expressionType": 1
                        },
                        "rightExpression": {
                            "value": "<GUID Priority 'Критический'>",
                            "expressionType": 0
                        }
                    }
                ],
                resultParameters: [
                    {
                        "parameterName": "Assignee",  // поле
                        "dataValueType": 10  // GUID (справочник)
                    }
                ]
            }
        ],
        
        // Далее methods, attributes, diff...
    };
});
```

**Вывод:** no-code мастер генерирует `rules` массив в клиентской схеме!

### 8.5 Пример: полный путь от no-code к коду

**Сценарий:** в мастере создан раздел "Обращения" с правилом "Если приоритет критический → исполнитель обязателен"

**no-code (мастер разделов):**
```
✓ Создать объект Case
✓ Добавить поле Priority (справочник)
✓ Добавить поле Assignee (справочник Contact)
✓ Бизнес-правило: Priority = Критический → Assignee обязательное
✓ Сохранить → система генерирует код
```

**code (что создалось в пакете CTI):**

```javascript
// Файл: CTI/Schemas/CasePage/CasePage.js
define("CasePage", [], function() {
    return {
        entitySchemaName: "Case",
        
        // Атрибуты (модель данных)
        attributes: {
            "Priority": {
                dataValueType: 10,  // GUID
                isLookup: true,
                lookupListConfig: { listSchemaName: "Priority" }
            },
            "Assignee": {
                dataValueType: 10,  // GUID
                isLookup: true,
                isRequired: false,  // по умолчанию НЕ обязательное
                lookupListConfig: { listSchemaName: "Contact" }
            }
        },
        
        // Визуальные изменения (diff)
        diff: [
            {
                operation: "insert",
                name: "Priority",
                values: { labelText: "Приоритет", ... }
            },
            {
                operation: "insert",
                name: "Assignee",
                values: { labelText: "Исполнитель", ... }
            }
        ],
        
        // Бизнес-правила
        rules: [
            {
                ruleType: "RequiredField",
                triggeredByColumnChange: ["Priority"],
                businessRuleConfigs: [
                    {
                        comparisonType: 4,  // =
                        leftExpression: {
                            columnPath: "Priority",
                            expressionType: 1  // column
                        },
                        rightExpression: {
                            value: "<GUID для 'Критический'>",
                            expressionType: 0  // constant
                        }
                    }
                ],
                resultParameters: [
                    { parameterName: "Assignee" }
                ]
            }
        ],
        
        // Методы (обработчики событий)
        methods: {
            onPriorityChanged: function() {
                // обработка изменения приоритета
            }
        }
    };
});
```
