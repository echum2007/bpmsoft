# Блок 1. Начало работы и первое приложение

**Цель:** настроить среду разработки и пройти полный цикл создания приложения — от пакета до переноса на другую среду.

**Статус:** ✅ завершён

---

## 1. Контекст задачи

Документация ведёт через создание приложения для интернет-магазина. Задача: автоматизировать наполнение «Предложения дня» товарами по критериям. По ходу решения проходим весь цикл — от настройки среды до переноса.

Пример универсален — он учит создавать разделы, детали, процессы и сервисы. Конкретная предметная область (продукты/предложения) не принципиальна: те же приёмы используются при кастомизации обращений, сервисных договоров и любых других сущностей.

---

## 2. Предварительные настройки

Перед началом любой разработки нужно настроить три системных параметра. Путь: Дизайнер системы → Системные настройки.

### 2.1. Издатель (Maintainer)

Имя автора всех изменений. По умолчанию `Customer`, рекомендуется сменить на название компании. Это значение записывается в `descriptor.json` каждого пакета.

### 2.2. Префикс названия объекта (SchemaNamePrefix)

Короткий префикс, который автоматически добавляется к именам всех создаваемых объектов. По умолчанию `Usr`. Если сменить на `Bpm`, то при создании раздела «Продукты» объект назовётся `BpmProduct`.

Зачем: при командной разработке сразу видно, кто создал объект, и нет конфликтов имён.

### 2.3. Текущий пакет

Пакет, в который система будет автоматически сохранять все результаты кастомизации через мастер разделов, мастер деталей и т.д. Задаётся после создания своего пакета.

---

## 3. Создание пользовательского пакета

Всё, что ты разрабатываешь, живёт в пакете. Пакет `Custom` использовать **не рекомендуется**.

**Путь:** Дизайнер системы → Управление конфигурацией → кнопка «+» в строке «Все пакеты».

**При создании указываешь:**
- Название (например, `BpmPackage`)
- Описание (необязательно)
- Версию (по умолчанию `1.0.0`)
- Чекбокс «Компилировать в отдельную сборку» — для обычной разработки оставляешь выключенным

**Добавление зависимостей** — после создания нажимаешь «Создать и добавить зависимости» → «Добавить» → выбираешь пакет, от которого наследуешься. Выбор зависит от сборки:
- `ProductCore` — для примера из документации
- `Constructor` — для сборки «Конструктор»
- `SalesEnterprise_Marketing_ServiceEnt` — для коммерческих сборок

Зависимость = наследование: твой пакет «видит» всё, что есть в тех пакетах, от которых он зависит.

## 4. Этап 1 — Создание разделов

**Раздел** — полноценная «страница» приложения со списком записей (реестром) и карточкой редактирования. Аналог таблицы в БД + интерфейс для работы с ней.

### Создание через Мастер разделов

Путь: Дизайнер системы → Настройка системы → Мастер разделов.

Для каждого раздела заполняешь:
- **Заголовок** — отображаемое имя (например, «Продукты»)
- **Код (на английском)** — системное имя (например, `BpmProduct`)
- **Рабочее место** — группа в боковом меню (например, «Приложения»)

### Настройка страницы

Мастер показывает визуальный редактор: перетаскиваешь колонки нужного типа (текст, число, справочник, дата и т.д.) в свободные области страницы.

### Что создаёт мастер за кулисами

- **Объект** (схема данных) — таблица в БД со всеми колонками
- **Клиентский модуль** (модель представления) — JavaScript-файл с поведением страницы
- **Запись в конфигурации** — регистрация раздела в системе

В разделе «Конфигурация» можно увидеть сгенерированный JavaScript, например:

```javascript
define("BpmProduct1Page", [], function() {
    return {
        entitySchemaName: "BpmProduct",
        // ... diff, methods, attributes
    };
});
```

Это AMD-модуль — подробно разберём в Блоке 6.

## 5. Этап 2 — Создание детали

**Деталь** — компонент интерфейса, который показывает связанные записи другого объекта на странице текущей записи. Например, на странице обращения можно видеть его активности, комментарии, файлы.

### Связь «один-ко-многим»

Деталь работает по принципу: «покажи все записи объекта X, у которых колонка Y равна Id текущей записи». Это классическая связь «один-ко-многим» в базе данных.

### Три способа создания деталей

| Способ | Сложность | Когда использовать |
|---|---|---|
| Мастер разделов | Без кода | Простые детали при настройке страницы |
| Мастер деталей | Без кода | Тонкая настройка, регистрация по существующему объекту |
| IDE системы | Код (JS + SQL) | Полный контроль — объект + схема представления + SQL-регистрация |

### Структура детали (три схемы)

Для каждой детали нужны:
1. **Схема объекта детали** — определяет структуру данных и связь с объектом раздела
2. **Схема модели представления детали** — определяет интерфейс (реестр, поля, кнопки)
3. **Схема страницы записи детали** — определяет карточку редактирования записи

### Типы деталей

- Деталь с **редактируемым реестром** — редактирование прямо в списке
- Деталь со **страницей добавления** — для добавления открывается отдельная карточка
- Деталь с **выбором из справочника** — выбор записей из другого раздела
- Деталь с **полями** — ввод данных в разные поля
- Деталь типа **«Файлы и ссылки»** — прикрепление файлов и ссылок

## 6. Этап 3 — Добавление данных

Для работы приложения нужны начальные данные — справочники, системные настройки и т.д.

**Справочник** — объект, записи которого используются как варианты выбора в справочных полях.

### Привязка данных к пакету

Данные хранятся в базе. При переносе пакета на другую среду переносится только структура (объекты, схемы), но **не записи**. Чтобы данные тоже переносились, их нужно **явно привязать к пакету**.

**Путь:** Конфигурация → выбрать пакет → «Добавить» → «Данные» → указать объект → выбрать записи.

После привязки записи будут включены в экспорт пакета.

## 7. Этап 4 — Валидация данных

Первое знакомство с бизнес-процессами. Простой процесс, который запускается при сохранении записи и проверяет условие (например, корректность заполнения поля «Код продукта»).

На этом этапе важно увидеть принцип: бизнес-процесс можно «привязать» к событию в системе (создание записи, изменение поля и т.д.).

Подробно BPMN и все элементы процессов — в **Блоке 15**.

## 8. Этап 5 — Автозаполнение «Предложения дня»

Более сложный бизнес-процесс. Задача: по нажатию кнопки автоматически наполнить деталь продуктами, подходящими по категории.

### Алгоритм процесса

1. Прочитать количество продуктов с нужной категорией
2. Если продуктов нет — завершить
3. Иначе — создать записи в детали

### Используемые элементы бизнес-процесса

| Элемент | Назначение |
|---|---|
| **Читать данные** | Выборка записей из БД |
| **Исключающее «ИЛИ»** (шлюз) | Ветвление по условию |
| **Добавить данные** | Создание записей |
| **Формула** | Вычисления |
| **Задание-сценарий** | Вставка кода C# в процесс |

### Первое прикосновение к JavaScript

На этом этапе в клиентский модуль страницы добавляется кнопка для запуска процесса. Появляется код вида:

```javascript
define("BpmDealsOfDayPage", ["ProcessModuleUtilities"],
    function(ProcessModuleUtilities) {
    return {
        entitySchemaName: "BpmDealsOfDay",

        messages: {
            "CompleteDeal": {
                "mode": BPMSoft.MessageMode.BROADCAST,
                "direction": BPMSoft.MessageDirectionType.SUBSCRIBE
            }
        },

        methods: {
            // ... бизнес-логика: запуск процесса, обработка результата
        },

        diff: [
            // ... описание элементов интерфейса (кнопки, поля)
        ]
    };
});
```

Ключевые понятия, которые здесь видны:
- `define(...)` — AMD-модуль (Блок 6)
- `messages` — механизм обмена сообщениями между модулями (Блок 10)
- `methods` — методы страницы (Блок 7)
- `diff` — описание изменений интерфейса (Блок 7)

Подробно все эти конструкции — в **Блоках 5–8**.

## 9. Этап 6 — Создание веб-сервиса

Первое знакомство с серверным C#-кодом. Задача: создать сервис, возвращающий сумму товаров из предложения дня.

### Что такое веб-сервис в BPMSoft

C#-класс, который наследуется от `BaseService` и помечается атрибутами. Технология основана на WCF.

**Путь создания:** Конфигурация → выбрать пакет → «Добавить» → «Исходный код» → написать C#-класс → сохранить → опубликовать.

```csharp
    using BPMSoft.Core;

    public class BpmDayOfferAmountService : BaseService
    {
        [OperationContract]
        [WebInvoke(Method = "POST", ...)]
        public decimal GetOfferAmount(string offerName)
        {
            // Логика: запрос к БД через ESQ, подсчёт суммы
        }
    }
}
```

Ключевые понятия, которые здесь видны:
- `BaseService` — базовый класс сервисов BPMSoft
- `[ServiceContract]`, `[OperationContract]` — атрибуты WCF
- `EntitySchemaQuery` — ORM-запрос к данным (Блок 12)
- `UserConnection` — контекст текущего пользователя

Подробно C# и серверная разработка — в **Блоках 11–13**.

## 10. Перенос приложения

Финальный этап — перенос на другую среду (dev → test → prod).

### Способ

1. **Экспорт:** Конфигурация → меню пакета → «Экспорт» → скачивается .zip
2. **Импорт:** на целевой среде → Дизайнер системы → «Установка и удаление приложений» → «Добавить приложение» → «Установить из файла»
3. После импорта — выйти из системы и выполнить повторный вход

### Почему важна привязка данных

Без привязки данных (Этап 3) при переносе перенесётся только структура (объекты, схемы, процессы), но записи справочников останутся на исходном стенде. На новой среде приложение будет «пустым».

Подробно управление поставками (рабочие среды, git, пакеты-сборки) — в **Блоке 19**.

## Итоги Блока 1

**Что нужно запомнить:**

1. **Перед началом разработки** — настрой Издатель, Префикс, Текущий пакет.

2. **Всегда работай в своём пакете** — не в `Custom`, не в системных.

3. **Раздел** = таблица в БД + интерфейс. Создаётся через Мастер разделов.

4. **Деталь** = связанные записи на странице. Связь через справочную колонку (один-ко-многим).

5. **Привязка данных к пакету** — обязательна для переноса справочников и начальных данных.

6. **Бизнес-процесс** — визуальная схема автоматизации. Может запускаться по событию или по кнопке.

7. **Веб-сервис** — C#-класс, наследник `BaseService`. Точка входа для внешних и внутренних вызовов.

8. **Перенос** — экспорт пакета в .zip → импорт на целевой среде.

9. **Мастер генерирует код** — за каждым визуальным действием стоят JavaScript-модули, объекты БД и конфигурационные записи. В следующих блоках мы научимся читать и писать этот код.

## Применение к Service Desk

В сборке «Управление сервисом» те же самые приёмы используются для:
- Создания и кастомизации разделов обращений (Cases)
- Настройки деталей на странице обращения (активности, комментарии, файлы)
- Бизнес-процессов обработки обращений (маршрутизация, уведомления, эскалация)
- Веб-сервисов для интеграции с внешними системами (Telegram, почта)
- Расчёта SLA — пользовательских стратегий сроков реакции и разрешения
