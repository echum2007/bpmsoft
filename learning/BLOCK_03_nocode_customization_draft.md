# Блок 3. Кастомизация No-code

**Цель:** уметь настраивать разделы, страницы, бизнес-правила и детали без написания кода.

**Статус:** 🔄 черновик — дополняется после практики

---

## 1. Мастер разделов — главный инструмент no-code настройки

**Мастер разделов** — визуальный редактор для создания и изменения разделов, страниц, полей и бизнес-логики без кода. Под капотом он генерирует клиентские схемы (замещающие модели представления) — то самое, что в Блоках 7–8 пишется вручную.

**Как открыть:**
- Новый раздел: Дизайнер системы → «Мастер разделов»
- Существующий раздел: открыть раздел → кнопка «Настройка вида» → «Открыть мастер раздела»

Требуется системная операция «Доступ к разделу «Конфигурация»».

---

## 2. Создание нового раздела

При создании раздела мастер предлагает два пути.

**Путь 1 — новый объект** (создаётся вместе с разделом)

| Поле | Описание |
|------|---------|
| Заголовок | Название раздела в боковой панели |
| Код | `[Префикс][ИмяОбъекта]` — латиница, без пробелов. Например: `BpmPromoAction`. После первого сохранения не меняется! |
| Иконка | PNG/SVG, прозрачный фон, до 38×38 px |
| Рабочее место | Куда добавить раздел в навигации |
| Индексировать для поиска | Включить в полнотекстовый поиск |
| Доступно визирование | Включить функциональность согласования записей |

**Путь 2 — существующий объект** (раздел поверх уже существующей таблицы)

Используется, например, чтобы добавить удобный интерфейс для существующего справочника. Если объект базовый (системный) — теги и группы будут недоступны.

### Что создаётся автоматически при создании раздела

| Объект | Назначение |
|--------|-----------|
| `<Code>` | Основная таблица раздела |
| `<Code>File` | Вложения (файлы и ссылки) |
| `<Code>Folder` | Иерархия групп |
| `<Code>InFolder` | Связи записей с группами |
| `<Code>Tag` | Теги раздела |
| `<Code>InTag` | Теги конкретных записей |
| `<Code>RequestsVisa` | Визы (согласования) |

Все объекты видны в «Управление конфигурацией». После создания — повторный вход в систему для отображения раздела в навигации.

## 3. Настройка страниц раздела

Страница — форма для просмотра/редактирования одной записи. Один раздел может иметь несколько страниц (например, разные для разных типов обращений).

### Структура страницы

Страница → Вкладки → Блоки → Поля.

**Вкладки:**
- Нельзя удалить все вкладки — одна всегда остаётся
- Вкладка «Лента» при удалении восстанавливается только через разработку
- Порядок меняется перетаскиванием

**Блоки:**
- `ProfileContainer` — блок в «шапке» страницы (левая часть). Ограничение: два поля в одной строке не поддерживаются
- `ColumnContainer` — многоколоночный блок на вкладках

### Типы полей

| Тип | Описание |
|-----|---------|
| Текст | Строка или многострочный текст |
| Целое / дробное число | Числовые значения |
| Дата / Дата и время | Выбор через календарь |
| Логическое | Чекбокс |
| Справочник | Lookup — ссылка на запись другого объекта |
| Изображение | Загрузка изображения |
| Файл | Загрузка файла |

---

## 4. Мини-карточки

**Мини-карточка** — всплывающее окно с краткой информацией о записи при наведении на lookup-поле.

Настраиваются в мастере раздела. Для каждого режима (просмотр / редактирование / создание) — свой набор полей. Доступны только после создания страницы раздела.

---

## 5. Аналитика на странице раздела

В реестре раздела можно включить панель аналитики — дашборд с виджетами, привязанный к разделу. Добавляется через мастер раздела → вкладка «Аналитика».

Для Service Desk: в разделе «Обращения» — аналитика по открытым обращениям, SLA, исполнителям прямо в реестре.

## 6. Бизнес-правила

Механизм управления поведением полей на странице в зависимости от условий — без кода.

**Путь:** мастер раздела → вкладка «Бизнес-логика» → «Бизнес-правила»

### Доступные действия

| Действие | Описание |
|----------|---------|
| Показывать элемент | Показать/скрыть поле или вкладку |
| Делать поле редактируемым | Включить/заблокировать редактирование |
| Делать поле обязательным | Сделать обязательным для заполнения |
| Добавлять фильтр значений | Фильтровать список lookup-поля по условию |
| Заполнять значение поля | Автоматически подставить значение |

### Примеры для Service Desk

```
УСЛОВИЕ: «Приоритет» = «Критический»
ДЕЙСТВИЕ: поле «Ответственный» → сделать обязательным

УСЛОВИЕ: «Тип» = «Инцидент»
ДЕЙСТВИЕ: вкладка «SLA» → показывать

УСЛОВИЕ: «Сервисный контракт» заполнен
ДЕЙСТВИЕ: поле «Категория» → фильтровать значения по контракту
```

### Формулы в бизнес-правилах

В условиях и значениях можно использовать арифметические выражения и функции над полями — это расширяет возможности no-code логики.

---

## 7. Правила ввода

Механизм фильтрации допустимых значений в lookup-полях на основе значений других полей на странице.

Пример: поле «Услуга» показывает только услуги, входящие в выбранный «Сервисный контракт».

---

## 8. Детали (no-code)

**Деталь** — встроенный реестр связанных записей на странице. Добавляется через мастер раздела → страница → кнопка добавления детали.

**Редактируемый реестр** — разновидность детали, где строки редактируются прямо в таблице без открытия отдельной страницы.

---

## 9. Angular-модули на странице

Преднастроенные Angular-компоненты, добавляемые через мастер без кода:

| Модуль | Назначение |
|--------|-----------|
| Слайдер | Числовое поле в виде ползунка |
| Ссылка | Поле-гиперссылка |
| Кнопка | Кнопка с действием (вызов процесса) |
| Множественный выбор | Несколько значений из справочника |
| Текст | Форматированный текстовый блок |
| Изображение | Блок с изображением |
| Матрица | Табличный редактор значений |

Кастомные Angular-компоненты — в Блоке 10.

---

## 10. Продвинутый дизайнер интерфейсов

Отдельный инструмент для создания страниц на базе Angular (в отличие от классического мастера на ExtJS).

**Путь:** Дизайнер системы → «Продвинутый дизайнер интерфейсов»

| Классический мастер | Продвинутый дизайнер |
|--------------------|---------------------|
| Основан на ExtJS | Основан на Angular |
| Страницы и разделы ExtJS | Angular-страницы раздела |
| Ограниченный набор компонентов | Расширенная библиотека |
| Нет поддержки кастомных Angular-компонентов | Полная поддержка Angular |

Продвинутый дизайнер — направление развития UI в BPMSoft. Глубокое изучение — в Блоке 10.

### 🔬 Лаб. 3.1 — Создание нового раздела через мастер

**Сценарий:** нужен справочный раздел для ведения категорий оборудования, к которому относится обращение.

**Задание:**
1. Открыть Мастер разделов, создать новый раздел на основе существующего объекта — любого подходящего справочника из системы
2. Задать название, код с префиксом организации, иконку
3. Добавить страницу с полями: «Название», «Описание», «Ответственный»
4. Добавить раздел в рабочее место «Конструктор»
5. Сохранить, перелогиниться, убедиться что раздел появился в навигации

**Проверить:** сколько объектов создалось в «Управление конфигурацией» с кодом нового раздела?

**Под капотом:** мастер сгенерировал замещающую модель представления раздела — JavaScript-схему в пакете. В Блоке 7 будем писать такие схемы вручную.

---

### 🔬 Лаб. 3.2 — Бизнес-правила на странице обращения

**Сценарий:** при критическом приоритете обращения исполнитель должен быть обязательным.

**Задание:**
1. Открыть мастер раздела «Обращения»
2. Создать бизнес-правило: если «Приоритет» = «Критический» → поле «Ответственный» / «Исполнитель» → обязательное
3. Сохранить, проверить: создать тестовое обращение с критическим приоритетом, попробовать сохранить без исполнителя

**Дополнительно:** добавить правило автозаполнения или фильтрации lookup-поля по условию.

**Под капотом:** бизнес-правила компилируются в секцию `rules` клиентской схемы. В Блоке 7 разберём как они выглядят в коде.

---

### 🔬 Лаб. 3.3 — Деталь на странице обращения

**Сценарий:** добавить деталь на страницу обращения — реестр связанных записей.

**Задание:**
1. Через мастер раздела «Обращения» добавить деталь на отдельную вкладку
2. Выбрать объект с реальной связью с обращением (например, «Активности», «Файлы», или другой доступный)
3. Настроить отображаемые колонки
4. Проверить результат на реальном обращении

**Под капотом:** деталь генерирует отдельную схему `BaseDetailV2` — разберём в Блоке 8.

---

### 🔬 Лаб. 3.4 — Правила цветового выделения

**Сценарий:** операторам нужно видеть критические обращения в реестре сразу.

**Задание:**
1. Создать правило: обращения с приоритетом «Критический» → красный фон в реестре раздела
2. Создать второе правило: обращения со статусом «Новое» → жёлтый фон (другой приоритет)
3. Назначить на роль, проверить в реестре

**Помнить:** условия с типом «Дата/Дата и время» не поддерживаются в правилах выделения.

## Итоги блока

1. **Мастер разделов** генерирует клиентские схемы — понимание no-code инструментов даёт понимание того, что именно делает код в Блоках 7–8.

2. **Код раздела не меняется** после первого сохранения — задавать осмысленно с префиксом с самого начала.

3. При создании раздела автоматически создаётся **7 связанных объектов**.

4. **Бизнес-правила** — самый мощный no-code инструмент: видимость, обязательность, фильтрация, автозаполнение по условиям.

5. `ProfileContainer` не поддерживает два поля в одну строку — известное ограничение.

6. **Продвинутый дизайнер** на Angular — направление развития UI, классический мастер на ExtJS — текущая основа.
