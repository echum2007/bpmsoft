## Кейс 1: Зависание окна «Конфигурация» после публикации C#-схемы

**Симптом:** После публикации схемы «Исходный код» окно конфигурации зависает, не реагирует.

**Решение:** Перезагрузить страницу конфигурации (F5). Публикация при этом проходит успешно — зависание только на стороне UI.

**Урок:** Это известное поведение BPMSoft 1.9. Зависание UI не означает ошибку компиляции. Проверять результат публикации после перезагрузки страницы.

## Кейс 2: Бизнес-процесс не запускается после развёртывания (ложная тревога)

**Дата:** 2026-04-11

**Симптом:** Процесс `RunSendEmailToCaseGroupV2` («Отправка email сообщения группе о назначении обращения») перестал появляться в журнале процессов после развёртывания пакета CTI (CC-уведомления + индикатор обслуживания).

**Ход диагностики:**
1. Проверили структуру процесса в PKG — два StartSignal с фильтрами
2. Проверили метаданные замещающего объекта Case — нашли флаг D33=true (незнакомый)
3. Проверили активность процесса — активен ✅
4. Перекомпилировали всё + перезапустили Kestrel
5. Создали тестовое обращение — не запустился
6. **Проверили условия фильтра** — оказалось, тестовое обращение не соответствовало фильтру

**Корневая причина:** Не баг, а несоответствие тестовых данных фильтру процесса. Процесс запускается **только** при:
- `Owner` (Ответственный) = NULL
- `Group` (Группа) IS NOT NULL

Тестовое обращение имело заполненного ответственного → фильтр не сработал.

**Решение:** Создать обращение без ответственного, но с группой → процесс запустился.

**Извлечённые уроки:**

1. **Всегда первым делом проверять условия фильтра стартового сигнала** — прежде чем подозревать поломку
2. **Структура стартовых сигналов процесса** (из metadata.json):
   - `DZ5 = 1` → событие INSERT (создание записи)
   - `DZ5 = 2` → событие UPDATE (изменение записи)
   - `DZ12` → массив UId отслеживаемых колонок (для UPDATE)
   - `DZ13` → JSON с фильтром (условия срабатывания)
   - `FC2` → EntitySchemaUId объекта, к которому привязан сигнал
3. **Замещающие объекты (ExtendParent=true) НЕ ломают стартовые сигналы процессов** — сигналы привязаны к базовому UId объекта, замещение его не меняет
4. **Флаг D33 в метаданных замещения** — не влияет на процессные сигналы (природа флага до конца не выяснена)
5. **«Изменить процесс» в контекстном меню объекта** — относится к DCM (Dynamic Case Management), а не к BPMN-сигналам
6. **Системные процессы нельзя пересохранить напрямую** — только «Сохранить новую версию» (создаёт замещение в вашем пакете)
7. **Расположение метаданных процессов:** `src/PKG_BPMSoft_Full_House_1.9.0.14114/CaseService/Schemas/RunSendEmailToCaseGroupV2/metadata.json`

## Общие правила диагностики BPMSoft

### Процесс не запускается — чек-лист
1. ✅ Проверить **условия фильтра** стартового сигнала (DZ13 в metadata.json)
2. ✅ Проверить **активность** процесса (галка «Активная версия»)
3. ✅ Проверить **журнал процессов** — может процесс запускается, но падает с ошибкой
4. ✅ Попробовать «Компилировать всё» + перезапуск Kestrel
5. ✅ Проверить, работают ли **другие** процессы по этому же объекту (локализация: объект vs процесс)
6. ⚠️ Крайняя мера: «Сохранить новую версию» (создаёт замещение процесса в CTI)

### После публикации схемы — проблемы
1. UI завис → F5, проверить результат
2. Ошибка компиляции → читать текст ошибки, часто подсказывает missing using или опечатку
3. EventListener не срабатывает → перезапуск Kestrel обязателен
4. Клиентский модуль не обновился → Ctrl+Shift+R (очистка кэша браузера)
