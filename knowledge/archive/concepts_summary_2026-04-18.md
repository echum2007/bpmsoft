# Технический справочник разработчика BPMSoft

Консолидированные стандарты разработки, архитектурные ограничения и решения по нюансам платформы BPMSoft, выявленные в ходе реализации Волны 2 (система уведомлений) и оптимизации пакета CTI.

---

## 1. Методология разработки и DevOps-протоколы

### 1.1. Управление пакетами и Git-стратегия

Синхронизация файловой разработки и базы данных требует строгого соблюдения дисциплины версионности.

- **Ветвление:**
  - `master`: Зеркало промышленной среды. Прямые коммиты запрещены.
  - `dev`: Ветка для интеграции и тестирования перед деплоем.

- **Синхронизация:** При подключении репозитория к среде разработки рабочая папка пакета CTI заменяется на клонированный репозиторий.

- **Критический запрет (Upgrade Risk):** Категорически запрещено использовать кнопку **«Сохранить новую версию»** для системных BPMN-процессов.
  - *Последствие:* Создается замещение процесса в пакете CTI, которое «замораживает» логику. При обновлении платформы (kernel upgrade) новые исправления и функционал вендора не будут применены к этому процессу, что создает технический долг.

### 1.2. Перенос изменений (Deploy Guide) и «Призрачные схемы»

Миграция пакета между средами (Dev → Prod) выполняется через экспорт/импорт .zip архивов.

1. **Бэкап:** Экспорт текущего прода. Формат имени: `CTI_YYYY-MM-DD_HH.MM.SS_before-feature.zip`.

2. **Проблема «Призрачных схем» (Ghost Schemas):** Объекты, удаленные в БД, могут сохраняться в старых архивных копиях репозитория.
   - *Стандарт:* Перед деплоем всегда выполнять свежий экспорт с прода для очистки локального архива. Путь `src/CTI/CTI/` должен обновляться после каждой успешной миграции.

3. **Git-тегирование:** Установка тегов на master (точки отката) обязательна перед импортом.

4. **Импорт:** Установка через UI BPMSoft.

5. **Restart:** см. раздел 1.3.

### 1.3. Требования к перезапуску Kestrel

Перезапуск веб-сервера Kestrel необходим для переинициализации серверных ресурсов.

| Тип изменений | Перезапуск | Примечание архитектору |
| --- | --- | --- |
| **Схемы «Исходный код» (C#)** | **Да** | Компиляция сборок и загрузка в домен приложения. |
| **EntityEventListener** | **Да** | Регистрация слушателей событий БД происходит строго при старте. |
| **Макросы (IMacrosInvokable)** | **Да** | Новые провайдеры регистрируются через рефлексию при инициализации. |
| **Feature Toggles** | **Нет*** | Переключение флага применяется на лету, но если флаг активирует новый C#-код, перезапуск обязателен. |
| **JS/CSS/AMD модули** | **Нет** | Загружаются браузером как статические ресурсы. |
| **BPMN (без скриптов)** | **Нет** | Интерпретируются движком динамически. |

## 2. Архитектура модели данных и логика Case

### 2.1. Предикаты и статусы переоткрытия

Логика смены статуса на **«Получен ответ»** (UId `f063ebbe-fdc6-4982-8431-d8cfa52fedcf`, ранее «Переоткрыто») зашита в C#-предикат `ReopeningCondition`.

- **Логика:** `!IsReopenStatus && !IsFinalStatus && (IsResolved || IsPaused)`

- **Распределение флагов:** В системе из 22 статусов переход в «Получен ответ» возможен только для 16.
  - `IsPaused = true`: 14 статусов (например, «Ожидает ответа»).
  - `IsResolved = true`: 2 статуса («Решено», «Работы выполнены»).
  - `IsFinal = true`: 2 статуса («Закрыто», «Отменено») — переоткрытие заблокировано.

- **Нюанс:** Целевой статус «Получен ответ» сам не имеет установленных флагов (все `false`), что предотвращает циклическую обработку.

### 2.2. SLA vs Категория: Фундаментальный принцип

**SLA определяет строгость контроля, а не категория.**

- Категория (`CaseCategory`: Инцидент/Запрос) влияет на шаблоны и маршрутизацию, но не на расчеты сроков.

- *Архитектурный стандарт:* Не используйте категорию как фильтр для уведомлений о просрочке. Ошибочная классификация при регистрации («Запрос» вместо «Инцидента») создаст слепую зону. Все эскалации привязываются строго к расчетным показателям SLA.

### 2.3. Нюансы сущности Activity (Email)

- **ServiceProcessed (Boolean):** Выявлена дивергенция. Путь C# устанавливает `true`, старые BPMN — `false`.
  - *Решение:* Во всей новой логике принудительно устанавливать `true` для паритета с промышленным C#-путем.

- **CopyRecepient (CC):** Стандарт разделителя — `"; "` (точка с запятой и пробел).

- **Нормализация CC:** При программном заполнении из входящих писем необходимо удалять токены `< >` и проверять наличие `@`, иначе JS-валидатор на фронтенде заблокирует сохранение.

## 3. Система уведомлений (Волна 2)

### 3.1. Инициатор vs Пользователь услуги

Платформа разделяет получателей через настройки `DefaultNotifyContactInCase` и `DefaultNotifyInitiatorInCase`.

- Для каждого правила `CaseNotificationRule` требуется создание отдельных шаблонов для обеих ролей, даже если они часто совпадают.

### 3.2. UsrEmployeeNotificationManager: Ловушка видимости

Переход к единому менеджеру уведомлений на C# (вместо разрозненных BPMN) несет скрытый риск.

- **Visibility Trap:** Системный процесс `UsrProcess_0c71a12CTI5` содержит хардкод CC на роль **«1-я линия»** (UId `e142ad2e`). При деактивации этого процесса в пользу нового менеджера, группа «1-я линия» перестанет получать уведомления.

- **Стандарт правила (UsrEmployeeNotificationRule):** Должен включать поле `UsrRecipientType` со значениями: *Owner, Group, Manager, Watchers (Наблюдатели)*.

### 3.3. Шаблоны и Invokable-макросы

Для инъекции текста последнего письма клиента в уведомление инженеру используется `IMacrosInvokable`.

- **Регистрация макроса:** Требует SQL-скрипта с указанием `ParentId`.
- **Вызов:** `#@Invoke.UsrLatestCustomerEmailGenerator#`

## 4. Frontend-разработка и UI-нюансы

### 4.1. AMD и загрузка ресурсов

Использование префикса `css!` в массиве зависимостей `define` обязательно.

- **Диагностика:** Если классы в DOM присутствуют, но стили не применяются, проверьте консоль DevTools. Состояние **"CSS rules found: "** для целевого файла — прямой признак отсутствия префикса `css!`.

### 4.2. Кастомизация ProfileContainer

`ProfileContainer` на странице обращения использует специфический `ViewGenerator`.

- **Ограничение:** Не поддерживает `itemType: 6` (LABEL).
- **Решение:** Использовать `Container` (`itemType: 7`) и манипулировать `innerHTML` в методе `render()` модуля страницы.

### 4.3. Ловушки Binding (visible)

- **Строковые атрибуты:** Привязка `visible: { bindTo: "UsrStatusName" }` не сработает как boolean. Если в строке "Active", это не значит `true` для парсера привязок.

- **Вычисляемые методы:** Могут не пересчитываться, если зависимости не объявлены в модели явно. Безопаснее использовать вспомогательный boolean-атрибут.

## 5. Инструментарий отладки и диагностики

### 5.1. Browser Console (ExtJS)

Стандартные `Ext.ComponentQuery` в BPMSoft часто ограничены. Используйте менеджер:

- `Ext.Object.getKeys(Ext.ComponentManager.all.map)` — получение всех ID компонентов.
- Сигнатура проблемы: если компонент есть в ComponentManager, но `document.getElementById` возвращает `null`, значит компонент создан, но не отрисован (ошибка `render` или `visible: false`).

### 5.2. Диагностика BPMN-сигналов (Step-by-Step)

Если процесс не запускается на входящее событие:

- **Step 0:** Проверить Feature Toggle `RunReopenCaseAndNotifyAssigneeClass`. Если он равен 1, платформа вызывает C#-код, полностью игнорируя BPMN-замещение `UsrSendNotificationToCaseOwnerCustom1`.

- **Step 1:** Проверить фильтр `StartSignal`. Типовая ошибка: фильтр `Owner IS NULL` блокирует запуск, если ответственный был назначен в момент создания записи.

### 5.3. База данных (PostgreSQL)

- **Функциональные роли:** Всегда используйте представление **VwSysFunctionalRole**. Прямой запрос к таблице `SysFunctionalRole` в PostgreSQL вернет неполные данные из-за специфики архитектуры представлений платформы.

- **Feature Toggles:** Запрос состояния:

  ```sql
  SELECT * FROM AdminUnitFeatureState 
  JOIN Feature ON Feature.Id = FeatureId 
  WHERE Feature.Code = 'CodeName'
  ```

  (Таблицы `FeatureState` не существует).

## 6. Технические нюансы (Gotchas) — Сводная таблица

| Проблема | Причина | Решение |
| --- | --- | --- |
| **Исчезновение "1-й линии"** | Отключение старых BPMN при миграции на C#-менеджер. | Явное добавление роли в правила `UsrEmployeeNotificationRule`. |
| **Дублирование CC** | Входящие письма в формате Name + email. | C# нормализация: удаление `< >`, Split по пробелу, `string.Join` через `"; "`. |
| **Пустое письмо инженеру** | Шаблон 18834f34 не содержит макрос контента. | Регистрация макроса `UsrLatestCustomerEmailGenerator` с `ParentId: 16339f82`. |
| **BPMN не срабатывает** | Активен Feature Toggle на C# логику ядра. | Проверка `RunReopenCaseAndNotifyAssigneeClass` перед изменением процессов. |
| **Стили не применяются** | Отсутствие префикса `css!` в AMD. | Проверка через DevTools: `CSS rules found:`. |
| **Запрет обновления ядра** | Использование «Сохранить новую версию». | Запрет замещения системных процессов в пакете CTI. |
