## Чек-лист диагностики: процесс не запускается

1. ✅ **Step 0:** Проверить Feature Toggle `RunReopenCaseAndNotifyAssigneeClass` — если `=1`, платформа вызывает C#-код, полностью игнорируя BPMN-замещение `UsrSendNotificationToCaseOwnerCustom1`
2. ✅ Проверить **условия фильтра** стартового сигнала (`DZ13` в `metadata.json`)
3. ✅ Проверить **активность** процесса (галка «Активная версия»)
4. ✅ Проверить **журнал процессов** — может процесс запускается, но падает с ошибкой
5. ✅ Попробовать «Компилировать всё» + перезапуск Kestrel
6. ✅ Проверить, работают ли **другие** процессы по этому же объекту (локализация: объект vs процесс)
7. ⚠️ Крайняя мера: «Сохранить новую версию» (создаёт замещение в CTI)

**Правило:** Сначала всегда проверять фильтры и условия — не подозревать поломку.

## Чек-лист после публикации схемы

| Проблема | Решение |
|---|---|
| UI завис после публикации C# | F5, проверить результат — зависание нормально |
| Ошибка компиляции | Читать текст ошибки (missing using, опечатка) |
| EventListener не срабатывает | **Перезапуск Kestrel обязателен** |
| Клиентский модуль не обновился | Ctrl+Shift+R (очистка кэша браузера) |

## Кейс 1: Зависание окна «Конфигурация» после публикации C#

**Симптом:** После публикации схемы «Исходный код» окно конфигурации зависает.

**Решение:** F5. Публикация при этом проходит успешно — зависание только на стороне UI.

**Урок:** Известное поведение BPMSoft 1.9. Зависание UI ≠ ошибка компиляции.

## Кейс 2: Процесс не запускается после деплоя (ложная тревога)

**Дата:** 2026-04-11

**Симптом:** `RunSendEmailToCaseGroupV2` не появлялся в журнале процессов после деплоя CTI.

**Корневая причина:** Тестовое обращение не соответствовало фильтру процесса. Процесс запускается только при:
- `Owner` (Ответственный) = NULL
- `Group` (Группа) IS NOT NULL

**Решение:** Создать обращение без ответственного, но с группой → процесс запустился.

**Выводы:**
- Замещения ExtendParent не ломают стартовые сигналы
- Флаг `D33` в метаданных замещения — не влияет на процессные сигналы
- «Изменить процесс» в контекстном меню → это DCM, не BPMN
- Системные процессы нельзя пересохранить напрямую → «Сохранить новую версию»

## Призрачные схемы (Ghost Schemas)

Объекты, удалённые в БД, могут сохраняться в старых архивных копиях репозитория.

**Стандарт:** перед деплоем всегда выполнять свежий экспорт с прода — путь `src/CTI_<дата>/CTI/CTI` должен обновляться после каждой успешной миграции. Не деплоить из старого архива.

## Отладка ExtJS — поиск незарендеренных компонентов

Стандартный `Ext.ComponentQuery.query()` в BPMSoft ненадёжен. Использовать менеджер:

```javascript
// Получить все зарегистрированные ID компонентов
Ext.Object.getKeys(Ext.ComponentManager.all.map)

// Если компонент есть в ComponentManager, но document.getElementById возвращает null —
// компонент создан, но не отрисован (ошибка render или visible: false)
```

## Типичные ошибки в C#-коде

| Ошибка | Причина | Решение |
|---|---|---|
| `GetColumnValue<T>` не компилируется | Нет `using BPMSoft.Common` | Добавить using |
| EventListener не срабатывает | Не перезапущен Kestrel | Перезапуск Kestrel |
| CC не попадает в письмо | Activity уже создана до изменения | Проверить порядок вызовов |
| Процесс падает без ошибки в UI | Смотреть журнал процессов | Дизайнер системы → Журнал процессов |

## Кейс 3: CS0246 при публикации BPMN-процесса со ScriptTask

**Дата:** 2026-04-21

**Симптом:** `CS0246: The type or namespace name 'UsrMyClass' could not be found` при публикации BPMN-процесса. Класс существует, опубликован, в редакторе схемы ошибок нет.

**Корневая причина:** В новом BPMN-процессе пустое пространство имён (вкладка **Методы → Usings**). Компилятор не знает где искать класс `UsrMyClass`.

**Решение:** Свойства процесса → вкладка **«Методы»** → **«Добавить пространство имён +»** → ввести `BPMSoft.Configuration` → сохранить → опубликовать.

**Урок:** При создании любого BPMN-процесса с ScriptTask, использующего пользовательские C# классы — сразу добавлять `BPMSoft.Configuration` в Usings. По умолчанию список пустой.

## Кейс 4: Ошибка 42809 при публикации объекта с FK на VwSysRole

**Дата:** 2026-04-21

**Симптом:** При публикации объекта с Lookup-колонкой на `VwSysRole` — ошибка PostgreSQL `42809: referenced relation "VwSysRole" is not a table`.

**Корневая причина:** `VwSysRole` — это PostgreSQL VIEW, а не таблица. PostgreSQL не допускает внешние ключи на представления.

**Решение:** Использовать объект `SysAdminUnit` вместо `VwSysRole`. `SysAdminUnit` — реальная таблица, содержит роли, группы и пользователей. `VwSysRole` — лишь вьюха поверх неё с фильтром по типу.

## Кейс 5: Справочник EmailTemplateMacros не найден в UI

**Дата:** 2026-04-21

**Симптом:** Нужно зарегистрировать invokable-макрос. В разделе «Справочники» → поиск по «макрос» — ничего не находит.

**Корневая причина:** Таблица `EmailTemplateMacros` не зарегистрирована как справочник в UI BPMSoft.

**Решение:** Только через прямой SQL INSERT:
```sql
INSERT INTO "EmailTemplateMacros" ("Name", "ParentId", "ColumnPath")
VALUES (
    'UsrLatestCustomerEmailGenerator',
    (SELECT "Id" FROM "EmailTemplateMacros" WHERE "Name" = '@Invoke' LIMIT 1),
    'BPMSoft.Configuration.UsrLatestCustomerEmailGenerator'
);
```
После INSERT — привязать запись к пакету CTI через элемент «Данные» (тип установки: Установка).

**Урок:** Для переноса между средами — обязательно привязывать через «Данные» пакета, иначе INSERT нужно повторять на каждой среде вручную.
