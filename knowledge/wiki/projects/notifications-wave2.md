## Задача 2.3 — Текст ответа клиента в уведомлении инженера

**Детали:** `projects/notifications-wave2/TASK_2_3_DESIGN.md`  
**План внедрения:** `projects/notifications-wave2/TASK_2_3_DEPLOY.md`  
**Статус:** ✅ Внедрено на тестовой среде 2026-04-21

### Суть

Когда клиент отвечает на обращение (email) → инженер получает уведомление. Сейчас уведомление не содержит текст ответа клиента. Нужно включить текст письма в уведомление + убрать зависимость от смены статуса.

### Что было сделано (2026-04-21)

**Новые компоненты в пакете CTI:**

| Компонент | Тип | Назначение |
|---|---|---|
| `UsrLatestCustomerEmailGenerator` | C# Исходный код | IMacrosInvokable — возвращает Body входящего письма по ActivityId |
| `UsrSendEmailToCaseOwnerOnReply` | C# Исходный код | Читает правила из UsrEmployeeNotificationRule, отправляет email через EmailWithMacrosManager |
| `UsrEmployeeNotificationEventType` | Объект (BaseLookup) | Справочник типов событий: CustomerReply, OwnerAssigned, GroupAssigned, StatusChanged |
| `UsrEmployeeNotificationRecipientType` | Объект (BaseLookup) | Справочник типов получателей: Owner, Group, Role |
| `UsrEmployeeNotificationRule` | Объект (BaseEntity) | Правила уведомлений: событие + статус + категория → шаблон + получатель |
| `UsrSendEmailToCaseOwnerOnReplyProcess` | BPMN | StartSignal на Activity INSERT (входящий email) → ScriptTask |
| `Уведомление инженеру — ответ клиента` | EmailTemplate | Клон шаблона 18834f34 с макросом `#@Invoke.UsrLatestCustomerEmailGenerator#` |

**Что деактивировано:**
- `UsrProcess_0c71a12CTI5` — старый процесс (только при смене статуса, хардкод CC)

**Данные в UsrEmployeeNotificationRule:**
- Правило 1: EventType=CustomerReply, RecipientType=Owner → шаблон «Уведомление инженеру — ответ клиента»
- Правило 2: EventType=CustomerReply, RecipientType=Role, Role=«1-я линия поддержки» → тот же шаблон

### Ключевые находки при внедрении (2026-04-21)

**1. Префикс Usr обязателен для всех колонок объекта**
Платформа запрещает создавать колонки без префикса `Usr`. Все колонки `UsrEmployeeNotificationRule` должны начинаться с `Usr`: `UsrIsActive`, `UsrEventType`, `UsrCaseStatus` и т.д.

**2. SysRole — это View, FK невозможен**
`VwSysRole` — PostgreSQL VIEW, не таблица. Попытка создать FK на неё даёт ошибку `42809: referenced relation is not a table`. Для колонки «Роль» нужно использовать объект `SysAdminUnit` (реальная таблица).

**3. EmailTemplateMacros нет в UI**
Справочник `EmailTemplateMacros` («Макросы шаблонов сообщений») недоступен через раздел «Справочники». Регистрация invokable-макроса — только через SQL INSERT + привязка через элемент «Данные» пакета CTI.

**4. Кнопка «Пользовательский макрос» не показывает invokable-макросы**
В дизайнере шаблонов кнопка показывает только поля объекта. Invokable-макрос вставляется только вручную через «Редактировать исходный HTML»: `#@Invoke.UsrLatestCustomerEmailGenerator#`

**5. Пространство имён в BPMN-процессе обязательно**
При создании BPMN-процесса с ScriptTask, использующим пользовательский C# класс, нужно явно добавить `BPMSoft.Configuration` на вкладке **Методы → Usings**. Без этого публикация падает с `CS0246`. По умолчанию пространство имён пустое.

**6. Фильтр StartSignal: «Тип сообщения», не «Направление»**
В фильтрах StartSignal для Activity нет поля «Направление». Входящее письмо фильтруется по полю **«Тип сообщения» = «Входящее»**.

**7. Макрос [#Subject#] для темы обращения**
В шаблонах CTI встречается ошибочный макрос `[#Owner.Salutation#]` вместо темы обращения. Правильный макрос: `[#Subject#]` — подтверждено в существующих системных шаблонах.

## Ловушка видимости при переходе на новый менеджер

> ⚠️ **Visibility Trap:** `UsrProcess_0c71a12CTI5` содержит хардкод CC на роль «1-я линия» (UId `e142ad2e`). При деактивации этого процесса группа «1-я линия» перестанет получать уведомления если не добавить её явно в `UsrEmployeeNotificationRule`. **Решение:** Правило 2 с RecipientType=Role заменяет хардкод — добавить ДО деактивации старого процесса.
