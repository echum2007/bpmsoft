## Data-Driven диагностика: сначала запрос, потом гипотеза

**Правило:** при любой проблеме — сначала запросить реальное состояние системы, потом строить гипотезы.

Типичная ловушка: тратить 2+ часа на "деплой сломал процесс", "Kestrel кэширует", "баг в коде" — когда реальная причина обнаруживается за 5 минут первым же SELECT-ом.

**Пример из практики:** процесс `RunSendEmailToCaseGroupV2` не срабатывал при тестировании. Гипотезы: сломан деплой, кэш, код. Реальная причина: тестовый кейс был создан с Owner, а фильтр StartSignal требует `Owner IS NULL AND Group IS NOT NULL`. Тест не соответствовал условию — процесс работал корректно.

### Диагностический порядок

1. **Запросить данные** — `VwSysProcess`, `MetaData`, `FeatureState`, реальные записи
2. **Проверить условия** — фильтры StartSignal, feature toggles, статусы
3. **Проверить конфигурацию** — наличие и активность процессов
4. **Только после этого** — проверять инфраструктуру (Kestrel, деплой)

```sql
-- Проверить активные процессы и их фильтры
SELECT "Name", "Enabled", "MetaData" FROM "VwSysProcess"
WHERE "Name" LIKE '%Notify%' OR "Name" LIKE '%Send%'
ORDER BY "Name";
```

### Полный экспорт таблицы вместо точечных запросов

При маппинге неизвестной архитектуры (например, карты уведомлений) — экспортировать **всю** таблицу, а не делать точечные запросы по ожидаемым именам. Процессы с нечитаемыми именами (`UsrProcess_0c71a12CTI5`) не найти через `WHERE name LIKE '%notification%'`.

## Read-only протокол доступа к dev-серверу

**Правило по умолчанию:** только чтение (SELECT). `INSERT`, `UPDATE`, `DELETE`, `DDL`, рестарт сервисов — только с явного разрешения Евгения.

| Операция | Разрешена по умолчанию |
|---|---|
| SELECT | ✅ Да |
| Чтение файлов, логов | ✅ Да |
| INSERT / UPDATE / DELETE | ❌ Требует разрешения |
| DDL (CREATE / DROP / ALTER) | ❌ Требует разрешения |
| `sudo systemctl restart bpmsoft` | ❌ Требует разрешения |

## Иерархия инструментов: правильный инструмент прежде удобного

Перед поиском информации — выбирать лучший инструмент, а не привычный:

1. **NotebookLM блокноты** — 5–7× эффективнее по токенам, AI-синтез с цитатами
2. **edu.bpmsoft.ru** — официальная актуальная база знаний BPMSoft
3. **Локальный PDF** — если нет сети
4. **Локальная wiki** — заметки проекта, не авторитетный источник платформы
5. **Гипотеза из головы** — последний резерв, верифицировать через 1–3

> ⚠️ Правило: всегда верифицировать конкретные API-имена, классы и методы из NotebookLM по реальному коду — блокноты могут смешивать BPMSoft 1.9 и Creatio.

## Проверка документации перед принятием нового ПО

Перед деплоем новой версии инфраструктурного ПО:

1. Документация покрывает **именно ту версию**, которую деплоим?
2. Задокументирован management/admin API (создание объектов, управление)?
3. Changelog не содержит "removed" или "changed API" без документированной замены?
4. Если нет — считать версию **не готовой к использованию**.

**Урок Stalwart v0.16:** REST Management API тихо удалён, заменён на JMAP без документации. Веб-панель сломана из-за бага PKCE. SMTP/IMAP работают — но управлять сервером невозможно. Потрачено несколько часов. Проверка документации до деплоя заняла бы 5 минут.
